# 多图上传与素材融合改造开发文档

更新时间：2026-06-13

## 1. 目标

在不破坏现有 V1/V2 生成链路的前提下，把当前“单图上传 + 单组素材用途”的体验升级为“多图上传 + 多素材意图融合”。用户可以上传多张图片，系统识别每张图的关键特征，并按用户文案指定抽取主体、Logo、背景、版式、色彩、文案内容等元素参与生图。

本轮先完成多图功能上线前必须具备的底座：

1. 上传大小限制。
2. 单次生成绑定图片数量限制。
3. 上传素材归属到 Veyra 账号。
4. 素材读取、完成、生成引用时做账号隔离。
5. 输出完整开发方案和配置清单。

## 2. 当前代码现状

V1 位于 `src_skeleton/app`，V2 位于 `custom_media_agent_2_0/app`。两者共享浏览器外壳和视觉风格，但后端产品逻辑保持隔离。

V1 已有高级素材模型：

1. `Asset`、`AssetIntent`、`CreateAssetUploadRequest`。
2. `asset_intents` 支持多个素材意图。
3. `reference_image_paths(asset_plan, max_images=5)` 支持向 provider 传多张参考图。
4. 当前前端仍只取 `files[0]`，生成提交也偏单图体验。

V2 已有更完整的多素材后端雏形：

1. `CreateCreativeRunRequest.assets` 支持多条 `CreativeRunAssetInput`。
2. `asset_binding.py` 能按多素材生成 binding plan、provider input plan。
3. 已遵守“选定模板控制画面框架，上传素材填充框架”的 V2 Template Lock 原则。
4. 当前前端仍只取 `files[0]`，多图体验尚未开放。

## 3. 已完成的底座改造

新增配置：

1. V1 `MAX_ASSET_UPLOAD_BYTES`，默认 `12582912`。
2. V1 `MAX_ASSET_UPLOAD_COUNT`，默认 `6`。
3. V2 `V2_MAX_UPLOADED_ASSET_BYTES`，默认 `12582912`。
4. V2 `V2_MAX_UPLOADED_ASSET_COUNT`，默认 `6`。

新增安全与账号边界：

1. V1 `Asset.veyra_user_id`，上传时记录当前 Veyra 用户。
2. V2 `UploadedAsset.veyra_user_id`，上传时记录当前 Veyra 用户。
3. V1/V2 上传声明大小超限时拒绝。
4. V1/V2 上传实际 base64 内容超限时拒绝。
5. V1/V2 素材读取、上传内容、完成分析均校验账号可见性。
6. V1 生成引用素材时，在 Veyra 开启后校验素材归属；未开启 Veyra 时继续交给原素材规划逻辑，避免误伤本地老流程。
7. V2 生成引用其他账号素材时，在 asset context 阶段视为不可用，由现有 `asset_missing` 失败机制阻止纯文本退化生成。

## 4. 多图上传正式实现方案

### 4.0 OpenAI 官方多图输入约束

本项目多图功能按 OpenAI 官方图像接口能力设计，不把多图误当作纯文本 `images.generate`。

官方约束与本项目落点：

1. 多图参考/融合应优先走支持输入图的图像编辑链路，例如 Images API 的 image edit 或 Responses API 的 image generation tool 输入图能力。
2. OpenAI 图像编辑接口支持同时传入多张输入图，官方上限高于本项目当前配置；本项目为适配 3M VPS 带宽，默认限制为 6 张、单张 12MB。
3. Provider prompt 必须用自然顺序描述输入图，例如“Image 1 是产品主体，Image 2 是背景氛围，Image 3 是 Logo”，不得暴露内部 `asset_id`、`case_id`、`source_url`。
4. 如果使用 mask，局部编辑的 mask 只应绑定首张被编辑底图；多图素材融合不默认启用 mask。
5. 多图会提高 image-token/输入图片额度消耗与延迟，前端必须提示数量，后端必须继续做硬限制。
6. 不能假设所有 provider 都支持相同的多图上限。V1/V2 OpenAI 通道可按本项目配置上限传图，Gemini 暂不可用时保持禁用，Mock 通道只记录引用数量。
7. `input_fidelity` 等模型相关参数必须做 provider 能力保护；不为不支持的模型盲目传参。

当前代码中，V1 OpenAI provider 已在有参考图时使用 `client.images.edit(image=[...])`；V2 OpenAI provider 已在 `input_images` 存在时使用 `client.images.edit(image=[...])`。本轮改造重点是让桌面端和 H5 真正传入多张素材，并把 provider 参考图上限从硬编码 5 改为配置化上限。

### 4.1 前端交互

桌面和 H5 都应从单文件控件升级为多文件队列：

1. `input[type=file]` 增加 `multiple`。
2. 上传后显示素材列表，每张图独立卡片展示缩略图、文件名、状态、识别摘要。
3. 每张图可设置：
   - 素材用途：主体、风格、Logo、人脸、背景、构图、色彩、反向参考。
   - 约束强度：必须保留、强参考、弱参考。
   - 备注：用户说明“取这张图里的瓶子”“用这张的背景”“只提取文案”等。
4. 提供全局快捷用途，但最终 payload 必须落到每张图的独立意图。
5. 超过 `MAX_ASSET_UPLOAD_COUNT` / `V2_MAX_UPLOADED_ASSET_COUNT` 时前端直接提示，不发起上传。
6. 单张超过大小上限时前端提示，后端继续作为最终防线。

### 4.2 V1 改造

V1 保持高级素材体系，不引入 V2 模块。

1. `state.assetIds` 从单图使用变为素材队列，并新增 `state.assets` 保存文件名、状态、预览 URL、视觉摘要。
2. `handleAsset` 改为 `handleAssets(files)`，逐张调用 `/v1/assets/upload-url`、`/content`、`/complete`。
3. `asset_intents` 按每张图生成，不再只用 `state.assetIds[0]`。当前第一阶段采用“全局用途作用于所有素材”的简化策略，后续可扩展为每张图独立 role/strength/notes。
4. `build_advanced_asset_plan` 继续作为素材规划核心。
5. `reference_image_paths` 的 `max_images` 改为读取 `settings.max_asset_upload_count` 或 provider 能力上限。
6. 历史记录展示每张素材的 role、notes、vision profile。

### 4.3 V2 改造

V2 坚持 Strict Isolation，不调用 V1 接口、不读取 V1 存储、不导入 V1 模块。

1. `v2State.uploadedAssets` 已是数组，前端上传逻辑改为追加队列而不是覆盖数组。
2. `v2AssetPayload()` 使用每张素材自己的 roles、strength、notes；当前第一阶段先把全局用途同步到所有上传素材，保留资产对象字段用于下一阶段逐图编辑。
3. `build_asset_context()` 继续生成多素材 `asset_binding_plan`。
4. 必须保留 hard identity assets 为 V2-native `input_images`，不能把主体/Logo/人脸/必需背景降级成文本。
5. 如果用户选择模板，模板仍是最高优先级 frame；上传素材只能填可替换 slot。
6. 如果用户未选模板，Claude Code 可结合多张上传图与检索案例决定视觉框架。
7. 最终提示词不得泄露 `asset_id`、`case_id`、`provider_id`、`source_url` 等内部标识。

### 4.4 Prompt 与素材意图

多图上传不是简单“把多张图传给模型”，必须先形成结构化素材意图：

1. 从用户文案、每张图备注、识别摘要推导 `fusion_mode`。
2. 区分 Logo 是产品表面贴标、场景物体上的标识，还是画布角标。
3. 区分“抽取内容”与“沿用构图”。
4. 选定模板时，上传构图参考不能覆盖模板结构，除非用户明确解锁模板。
5. 未选模板时，上传构图/版式可以成为 frame primary。

## 5. 配置清单

本地 `.env` / VPS 环境变量建议：

```env
MAX_ASSET_UPLOAD_BYTES=12582912
MAX_ASSET_UPLOAD_COUNT=6
V2_MAX_UPLOADED_ASSET_BYTES=12582912
V2_MAX_UPLOADED_ASSET_COUNT=6
```

如 VPS 带宽较低，建议先保持 6 张以内；如果后续接 CDN 或对象存储，再评估提高上限。

## 7. 开发章节

第一章：底座安全与配置。已完成。

第二章：V2 多图上传 UI。优先做，因为 V2 后端多素材语义更完整。

第三章：V2 多素材意图编辑与 payload。每张图独立 role、strength、notes。

第四章：V2 多图端到端测试。覆盖模板锁定、未选模板、Logo/主体/背景组合。

第五章：V1 多图上传 UI。复用同一视觉风格，但只接 V1 自己的 `/v1/assets/*`。

第六章：V1 高级素材多图端到端测试。

第七章：文档、配置、部署说明同步。

本轮落地边界：

1. 完成桌面端/H5 的多文件选择、逐张上传、队列展示、清空。
2. 完成 V1/V2 多图 payload 与后端现有素材链路闭环。
3. 完成 provider 参考图上限配置化。
4. 完成单测与浏览器 smoke 验证。
5. 暂不做“每张图单独弹窗编辑角色/强度/备注”的复杂 UI；先让多图可用、链路正确、安全边界清晰。

## 8. 风险与注意事项

1. 多图会增加上传体积，VPS 3M 带宽下必须限制数量和单图大小。
2. Provider 对 input images 的数量、尺寸、类型支持可能不同，应做能力裁剪。
3. 不允许 V2 回退到 V1 `legacy_image_bridge` 实现多图。
4. 不允许因 Claude 超时直接绕过 Claude 做确定性生成；应沿用 checkpoint/压缩继续策略。
5. 上传素材属于用户隐私资产，不能跨账号读取或复用。
6. 前端可以提前拦截，但后端必须是最终安全边界。
