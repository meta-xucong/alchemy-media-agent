# 26. V1 素材高级版测试验收与风控

## 1. 目标

本文定义 V1 素材高级版的测试矩阵、验收标准、失败处理和风控要求。高级版会把用户素材更强地纳入生图链路，因此测试重点不只是“能生成”，还包括用途是否生效、授权是否校验、历史是否可追溯、provider 不支持时是否清晰失败。

## 2. 测试原则

1. 基础版回归优先。任何高级版改动都不能影响基础版。
2. 高级版先测结构化计划，再测真实 provider 效果。
3. 上传图片不能只变成提示词摘要；支持参考图的 provider 必须真实收到图片输入。
4. provider 能力必须可模拟，不能把所有测试绑定到真实付费 API。
5. 风控类测试必须覆盖拒绝路径，不能只覆盖成功路径。
6. 历史记录和可复制提示词属于核心功能，必须纳入验收。
7. 视觉理解和视觉复检可以分阶段上线，但 schema、状态和失败降级必须先可观测。

## 3. 测试分层

| 层级 | 范围 | 是否调用真实模型 | 目标 |
|---|---|---|---|
| Unit | schema、校验、AssetPlan、PromptPlan | 否 | 规则稳定 |
| Contract | provider adapter 能力声明和请求结构 | 否或 mock | 接口不漂移 |
| Integration | 上传、分析、创建 job、历史记录 | 默认 mock | 后端链路完整 |
| Live Smoke | OpenAI/Gemini 少量真实调用 | 是 | 验证 provider 可用 |
| E2E | 前端基础参数、素材用途、生成、历史弹窗 | 可 mock 或 live | 用户路径可用 |
| Regression | 基础版旧路径 | 否或 mock | 防止破坏 1.0 |

## 4. 基础版回归用例

| ID | 用例 | 期望 |
|---|---|---|
| V1B-001 | 不传 `asset_mode`，只传 prompt | 按基础版生成 |
| V1B-002 | `asset_mode=basic` 且传 `asset_ids` | 沿用当前逻辑 |
| V1B-003 | 基础版历史记录 | 不显示高级素材计划 |
| V1B-004 | 基础版 provider 切换 GPT Image/Gemini | requested/actual provider 正确 |
| V1B-005 | 基础版失败重试 | 错误信息保持当前口径 |

验收要求：高级版上线前，基础版自动化测试必须全部通过。

## 5. 高级版结构校验用例

| ID | 用例 | 输入 | 期望 |
|---|---|---|---|
| V1A-001 | 高级版无素材用途 | `asset_mode=advanced`，无 `asset_intents` | 返回 `asset_role_required` |
| V1A-002 | 混用基础和高级 | 同时传 `asset_ids` 和 `asset_intents` | 返回 `asset_mode_conflict` |
| V1A-003 | 素材未 ready | 使用 `scanning` 素材 | 返回 `asset_not_ready` |
| V1A-004 | 未知 role | `role=foo` | 返回 validation error |
| V1A-005 | priority 越界 | `priority=999` | 返回 validation error |
| V1A-006 | strength 越界 | `strength=2` | 返回 validation error |
| V1A-007 | local_edit 无 mask | `role=local_edit` 无 `mask_id` | 返回 `mask_required` |
| V1A-008 | 上传非图片素材 | `mime_type=application/pdf` | 返回 `unsupported_asset_type` 或空 `upload_url` |
| V1A-009 | 视觉画像失败 | analyzer 超时或失败 | job 可降级，但历史记录 `vision_profile_status=failed` |

## 6. 高级版用途用例

### 6.1 风格参考

| ID | 用例 | 期望 |
|---|---|---|
| V1A-STYLE-001 | 上传摄影风格图，选择风格参考 | `PromptPlan.style_block` 包含色调、光线、构图 |
| V1A-STYLE-002 | provider 支持参考图 | `AssetPlan.provider_input_mode=reference_image` |
| V1A-STYLE-003 | provider 支持参考图并真实调用 | trace 记录 `reference_image_count>0` 或等效 file id |
| V1A-STYLE-004 | provider 不支持参考图 | 弱参考可退化为视觉画像提示，并写 warning |

验收重点：风格影响最终提示词，但不承诺复刻素材内容。

### 6.2 主体参考

| ID | 用例 | 期望 |
|---|---|---|
| V1A-SUBJ-001 | 上传商品图，选择主体参考 | `PromptPlan.subject_block` 要求保持主体关键特征 |
| V1A-SUBJ-002 | provider 不支持 image reference | 返回 `provider_capability_mismatch` 或降级提示 |
| V1A-SUBJ-003 | 多主体素材超过上限 | 返回数量限制错误 |

验收重点：不能假装所有模型都能主体保真。能力不支持时要清楚提示。

### 6.3 Logo/标识

| ID | 用例 | 期望 |
|---|---|---|
| V1A-LOGO-001 | 上传透明 Logo，选择 Logo 模式 | 要求 `logo_or_trademark_allowed=true` |
| V1A-LOGO-002 | 未确认 Logo 权利 | 返回 `asset_consent_required` |
| V1A-LOGO-003 | 第一阶段未启用后处理 | `PromptPlan.layout_block` 要求预留区域 |
| V1A-LOGO-004 | 启用后处理 | 输出图片中叠加原始 Logo，记录 `postprocess_steps` |

验收重点：精确 Logo 不依赖模型绘制，后处理记录可追溯。

### 6.4 人物脸/身份

| ID | 用例 | 期望 |
|---|---|---|
| V1A-FACE-001 | 选择人物脸用途但未授权 | 返回 `asset_consent_required` |
| V1A-FACE-002 | 已授权但 provider 不支持身份参考 | 返回 `provider_capability_mismatch` |
| V1A-FACE-003 | 已授权且 provider 支持 | 生成任务保存授权快照 |

验收重点：身份参考是高风险能力，必须显式授权和审计。

### 6.5 局部修改

| ID | 用例 | 期望 |
|---|---|---|
| V1A-EDIT-001 | 无 mask | 返回 `mask_required` |
| V1A-EDIT-002 | 有 mask 且 provider 支持 | 调用 image edit/mask edit |
| V1A-EDIT-003 | provider 不支持 | 返回 `provider_capability_mismatch` |

## 7. 历史记录验收

每条高级版输出必须包含：

- `asset_mode=advanced`。
- 原始提示词。
- 最终提示词。
- 素材用途列表。
- `asset_plan` 摘要。
- `prompt_plan` 摘要。
- requested provider/model。
- actual provider/model。
- fallback 标记。
- postprocess 记录。
- warnings。

前端历史弹窗要求：

- 原始提示词和最终提示词分开显示。
- 长文本使用滚动框。
- 用户可以复制完整提示词。
- 高级素材用途用中文标签展示。

## 8. Provider Contract Tests

每个图片 provider adapter 必须通过统一 contract：

| 测试 | 要求 |
|---|---|
| `test_provider_capabilities_declared` | 返回能力列表 |
| `test_text_to_image_request_shape` | 纯文本请求结构正确 |
| `test_reference_image_request_shape` | 参考图请求结构正确或明确不支持 |
| `test_reference_image_file_is_sent` | gpt-image-2/OpenAI 路线使用图片输入，而不是纯文本 prompt |
| `test_edit_request_shape` | 改图请求结构正确或明确不支持 |
| `test_vision_profile_schema` | 上传图生成的 `AssetVisionProfile` 字段完整 |
| `test_visual_review_schema` | 出图复检结果字段完整 |
| `test_provider_fallback_metadata` | fallback 时记录 requested/actual |
| `test_rate_limit_classification` | 429 不误报 502 |

## 9. Live Smoke 策略

真实调用成本较高，默认不进普通 CI。建议用环境变量控制：

```env
RUN_LIVE_IMAGE_TESTS=false
LIVE_IMAGE_PROVIDER=openai_image
LIVE_IMAGE_MODEL=gpt-image-2
LIVE_IMAGE_TEST_MAX_COUNT=2
```

Live Smoke 最小用例：

1. 纯文本基础版生成 1 张。
2. 高级版风格参考生成 1 张。
3. 高级版风格参考必须在 provider raw summary 或 trace 中证明走了 reference/edit 图片输入路径。
4. 如果启用 Logo 后处理，使用 mock 底图测试叠加，不消耗模型额度。

如果实际 provider 是 gpt-image-2 或兼容别名，验收重点不是“prompt 里有没有描述素材”，而是：

- 请求是否包含上传图片文件或 provider file id。
- provider 操作是否为 `images.edit`、`image_reference` 或等效参考图能力。
- 输出历史是否记录 `reference_image_count`、`api_operation`、`requested_model`、`actual_model`。

## 10. 额度和限流

高级版可能上传多张参考图，容易触发图片输入额度限制。必须实现：

- 单任务参考图数量上限。
- 单用户并发上限。
- 单 provider 429 冷却。
- 同组织额度打满时不盲目换同组织 key。
- 429 返回前端明确提示，不记成网关故障。
- 等待模式下允许后台继续，不因前端轮询时间短误报失败。

建议错误码：

| code | 说明 |
|---|---|
| `provider_rate_limited` | provider 限流 |
| `provider_quota_exceeded` | 额度耗尽 |
| `provider_cooldown_active` | provider 冷却中 |

## 11. 上传安全

上传阶段必须检查：

- 文件大小。
- MIME 类型。
- 扩展名与真实内容一致。
- 图片尺寸上限。
- 动图帧数上限。
- 病毒或恶意载荷扫描。
- EXIF 隐私信息处理。
- 私有对象存储，不能长期暴露公开 URL。

建议限制：

```env
ASSET_UPLOAD_MAX_MB=25
ASSET_IMAGE_MAX_PIXELS=25000000
ASSET_ALLOWED_MIME=image/png,image/jpeg,image/webp
ASSET_SIGNED_URL_TTL_SECONDS=900
```

当前 V1 高级版只允许图片素材。PDF、DOC、PPT、CSV 等文件必须在上传入口拒绝，避免进入 reference image 调用链路。

## 12. 内容与权利风控

| 风险 | 触发条件 | 处理 |
|---|---|---|
| 肖像权 | `portrait_identity` | 必须授权，记录审计 |
| 商标/Logo | `logo_overlay` | 必须确认权利 |
| 儿童或敏感人群 | 人物素材检测到风险 | 提示并限制用途 |
| 版权风格复刻 | 用户要求复刻在世艺术家或品牌风格 | 改写为泛化风格 |
| 违法内容 | prompt 或素材触发安全策略 | 拒绝 |
| 隐私泄露 | 素材含证件、手机号、地址 | 脱敏或拒绝 |

注意：provider 安全拒绝不能通过切换 provider 自动绕过。可以提示用户修改请求。

## 13. 失败信息规范

失败信息要面向用户，不暴露内部栈：

```json
{
  "code": "provider_capability_mismatch",
  "message": "当前生图模型不支持用素材做主体参考，请切换支持参考图的模型，或改用基础版。",
  "details": {
    "provider": "mock_image",
    "missing_capability": "image_reference"
  }
}
```

日志中保留 debug 信息：

- request id。
- job id。
- provider。
- model。
- asset ids。
- failure stage。
- provider raw error 摘要。

## 15. 上线闸门

上线前必须满足：

1. 基础版回归通过。
2. 高级版 schema 单测通过。
3. 高级版 mock integration 通过。
4. 至少一个真实 provider 的 live smoke 通过。
5. 支持参考图的 provider 能证明真实传入图片输入。
6. `AssetVisionProfile` 生成、失败和降级状态可追踪。
7. 视觉复检结果可保存和展示。
8. 未授权人脸和 Logo 请求能正确拒绝。
9. 历史记录能复制原始提示词和最终提示词。
10. 限流、provider 不支持、素材未 ready 都有友好提示。
11. 默认配置下高级版可关闭，便于灰度。

## 16. 手工验收脚本

第一轮手工验收建议：

1. 打开生图 V1.0 基础版。
2. 不上传素材生成 1 张，确认旧逻辑正常。
3. 上传一张风格图，选择高级版，选择“风格参考”，确认素材出现视觉画像摘要。
4. 生成 1 张，确认历史或 trace 记录了 reference image 调用形态。
5. 上传一张 Logo，选择高级版，选择“Logo/标识”，确认后处理步骤记录。
6. 切换 Gemini/GPT Image，确认历史里模型显示正确。
7. 打开历史大图，确认原始提示词、最终提示词、视觉复检结果都可查看或复制。
