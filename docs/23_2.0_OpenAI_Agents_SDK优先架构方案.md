# 23. 2.0 OpenAI Agents SDK 优先架构方案

## 1. 定位

本方案取代 `22_2.0_Agent化案例库与中枢调度方案.md` 作为 2.0 主方案。

新版原则：

1. 最大化使用 OpenAI Agents SDK 的原生能力。
2. 原 Veyra Agent 框架只保留最高价值的思想，不保留其自建 executor 形状。
3. 中枢 agent、资源 provider、模型 provider、工具 provider 仍保留，但尽量进入 Agents SDK 的 agent graph、tools、handoffs、sessions、guardrails、tracing 体系。
4. 后端服务仍负责持久化、队列、定时同步、provider 调用、风控硬门禁和对象存储。

一句话：

```text
OpenAI Agents SDK 是 2.0 的 agent runtime。
后端 domain services 是可信执行层。
资源、模型、外部 agent 都通过 SDK tools 或 SDK agent graph 暴露给中枢。
```

## 2. 为什么这样更好

OpenAI Agents SDK 的官方定位很贴合本项目：`Agent` 是带 instructions、tools、handoffs、guardrails、structured outputs 的 LLM 单元，`Runner` 负责 turns、tools、guardrails、handoffs、sessions。它还原生支持：

1. `agents as tools`：中枢 agent 调用专业 agent，同时保留对用户会话的控制。
2. `handoffs`：当某个专业 agent 需要接管对话时，可以正式交接。
3. `function tools`：后端服务能力以工具形式暴露。
4. `sessions`：多轮对话和炼金历史由 SDK session 管理。
5. `guardrails`：输入、输出、tool 调用都可以做安全和契约检查。
6. `tracing`：LLM generation、tool calls、handoffs、guardrails、custom events 都能进入 trace。

因此新版不再把“外部 agent registry + process executor + 自定义 stage runner”作为基础框架。那些东西只保留为思想：

1. 能力声明。
2. 超时和失败降级。
3. 结构化 JSON 契约。
4. provenance 和审计。
5. 版权门禁不可绕过。

## 4. 核心设计原则

### 4.1 SDK 优先

凡是属于 agent runtime 的能力，优先使用 OpenAI Agents SDK：

1. agent 定义。
2. manager pattern。
3. handoff。
4. agents as tools。
5. structured output。
6. dynamic instructions。
7. context。
8. sessions。
9. guardrails。
10. tracing。
11. streamed runs。

不再先设计一套自定义 stage executor，然后把 SDK 塞进去。

### 4.2 中枢保持控制权

默认使用 manager pattern：

```text
CreativeManagerAgent 调用专业 agent as tools
```

这样中枢保留对用户会话、案例召回、prompt 合成、生图和复检的整体控制。

只有当某个专业 agent 应该接管用户对话时，才用 handoff。例如：

1. 用户在模板库中深度挑模板，handoff 到 `TemplateGuideAgent`。
2. 用户针对某张图连续修改，handoff 到 `RevisionAgent`。
3. 用户从图片任务切换到视频任务，handoff 到 `VideoPlannerAgent`。

### 4.3 后端服务是可信执行层

Agent 不直接拿 API key、不直接访问数据库、不直接调用供应商。

Agent 只能调用受控 tools：

```text
agent -> SDK tool -> backend domain service -> provider adapter -> vendor API
```

### 4.4 定时同步不放进 agent 主循环

GitHub 案例库和未来资源库的定时同步由后台 service/job 执行：

```text
Scheduler -> Resource Sync Service -> Provider Adapter -> Parser -> Indexer -> Publish
```

中枢 agent 可以调用 `request_resource_sync()` 发起“检查更新”或“手动同步”，但不负责长时间跑同步任务。

### 4.5 硬门禁不可被 agent 推翻

版权、商用、肖像、隐私、内容安全属于 deterministic gate。

Agent 可以：

1. 提醒风险。
2. 给出替代 prompt。
3. 请求用户确认授权。
4. 降级使用风格信号。

Agent 不可以：

1. 把 `finalUseAllowed=false` 的素材放入最终结果。
2. 绕过商用禁止。
3. 直接复制高风险品牌或名人 prompt。

## 5. Agent Graph

### 5.1 CreativeManagerAgent

定位：最高运行时中枢。

职责：

1. 理解用户输入。
2. 决定使用模板模式、智能增强模式、批量模式、修订模式或视频预留模式。
3. 调用案例检索、素材分析、prompt 合成、provider 选择、生图、评价和反馈工具。
4. 控制多 agent 协作，不轻易交出用户会话。
5. 输出用户可读响应和机器可读 task result。

建议配置：

```python
CreativeManagerAgent = Agent(
    name="Creative Manager",
    instructions=dynamic_manager_instructions,
    tools=[
        CaseStrategistAgent.as_tool(...),
        PromptComposerAgent.as_tool(...),
        VisualCriticAgent.as_tool(...),
        search_prompt_cases,
        get_prompt_case,
        request_resource_sync,
        analyze_assets,
        run_safety_check,
        select_image_provider,
        create_image_job,
        store_feedback,
    ],
    handoffs=[
        TemplateGuideAgent,
        RevisionAgent,
        VideoPlannerAgent,
    ],
    output_type=CreativeRunResult,
)
```

### 5.2 CaseStrategistAgent

类型：agent as tool。

职责：

1. 根据用户需求制定案例召回策略。
2. 决定类目、用途、风格、风险过滤、多样性比例。
3. 输出 `CaseRetrievalPlan`。

不直接查库，查库由 `search_prompt_cases()` 工具做。

### 5.3 PromptComposerAgent

类型：agent as tool。

职责：

1. 将用户需求、素材摘要、模板变量、案例风格信号合成为 `ImagePromptPlan`。
2. 生成最终 prompt。
3. 保留 prompt provenance。
4. 避免 raw prompt 简单拼接。

输出：

```json
{
  "promptPlan": {},
  "finalPrompt": "",
  "negativeConstraints": [],
  "usedCaseSignals": [],
  "riskNotes": []
}
```

### 5.4 VisualCriticAgent

类型：agent as tool。

职责：

1. 评价生成结果是否符合 prompt plan。
2. 检查审美、构图、文字、主体完整、品牌一致性。
3. 输出 `CriticReport` 和 `suggestedRevisionPatch`。

它可以调用 `score_generation_output()` 获取确定性评分，也可以补充审美判断。

### 5.5 RevisionAgent

类型：handoff 或 agent as tool。

使用规则：

1. 单次自动修订时，作为 tool。
2. 用户进入某张图的多轮炼金时，使用 handoff。

输出：

```json
{
  "preserve": [],
  "change": [],
  "remove": [],
  "editMode": "image_edit|regenerate",
  "promptPatch": ""
}
```

### 5.6 TemplateGuideAgent

类型：handoff。

职责：

1. 帮用户在模板库中挑方向。
2. 解释模板适合用途。
3. 收集模板变量。
4. 返回 `TemplateCustomizationPlan`。

### 5.7 VideoPlannerAgent

类型：handoff。

职责：

1. 识别视频需求。
2. 输出视频预留任务结构。
3. 如果当前视频 provider 未启用，则给出实验性或排队状态。

## 6. Tools 设计

### 6.1 Resource tools

```python
request_resource_sync(provider_id: str, mode: str) -> SyncRequestResult
get_resource_provider_status(provider_id: str) -> ProviderStatus
```

说明：

1. 只发起或查询同步，不在 agent run 中执行长任务。
2. 返回 sync job id、当前版本、上次发布时间。

### 6.2 Case tools

```python
search_prompt_cases(query: CaseSearchQuery) -> CaseSearchResult
get_prompt_case(case_id: str) -> PromptCaseDetail
list_prompt_templates(filters: TemplateFilters) -> TemplateList
get_prompt_template(template_id: str) -> PromptTemplateDetail
```

检索由后端 Case Intelligence Service 执行，支持：

1. keyword。
2. embedding。
3. category。
4. use case。
5. style tags。
6. risk filters。
7. diversity rerank。

### 6.3 Asset tools

```python
analyze_assets(asset_ids: list[str]) -> MaterialBrief
get_asset_brief(asset_id: str) -> AssetBrief
```

大文件不进入 agent prompt，只传摘要、缩略图引用、必要 metadata。

### 6.4 Safety tools

```python
run_safety_check(payload: SafetyCheckPayload) -> SafetyDecision
run_license_gate(payload: LicenseGatePayload) -> LicenseDecision
```

这些 tool 需要 tool guardrails：

1. 输入必须包含 tenant、project、trace。
2. 输出必须包含 allow/reject/quarantine。
3. `reject` 或 `quarantine` 不能被 manager 覆盖。

### 6.5 Generation tools

```python
select_image_provider(request: ProviderSelectionRequest) -> ProviderDecision
create_image_job(request: ImageJobRequest) -> ImageJobResult
get_generation_job(job_id: str) -> GenerationJobStatus
score_generation_output(output_id: str, prompt_plan: dict) -> ScoreReport
```

图片生成可以使用：

1. OpenAI Agents SDK hosted image generation tool，用于简单 OpenAI-only 路径。
2. 后端 `create_image_job()`，用于多 provider、异步队列、成本、对象存储和统一审计。

为了保持多 provider 架构，生产主路径建议使用后端 generation tool。

## 7. Runtime Flow

### 7.1 模板模式

```text
User selects template
  -> CreativeManagerAgent
  -> get_prompt_template()
  -> TemplateGuideAgent handoff if needed
  -> PromptComposerAgent.as_tool()
  -> run_safety_check()
  -> select_image_provider()
  -> create_image_job()
  -> VisualCriticAgent.as_tool()
  -> final response
```

### 7.2 智能增强模式

```text
User simple prompt
  -> CreativeManagerAgent
  -> CaseStrategistAgent.as_tool()
  -> search_prompt_cases()
  -> get_prompt_case() for top cases
  -> PromptComposerAgent.as_tool()
  -> run_safety_check()
  -> create_image_job()
  -> VisualCriticAgent.as_tool()
  -> optional RevisionAgent.as_tool()
  -> final response
```

### 7.3 炼金修改模式

```text
User selects output + feedback
  -> CreativeManagerAgent
  -> handoff to RevisionAgent if multi-turn
  -> get original prompt plan and output metadata
  -> run_safety_check()
  -> create image edit job
  -> VisualCriticAgent
  -> version tree update
```

### 7.4 批量生成模式

```text
User batch request
  -> CreativeManagerAgent
  -> BatchPlannerAgent.as_tool()
  -> PromptComposerAgent for each variant or matrix
  -> safety batch precheck
  -> create batch image jobs
  -> async progress via backend events
```

## 8. Sessions 和 Memory

优先使用 Agents SDK session 管理用户对话上下文。

推荐：

1. 本地开发：`SQLiteSession`。
2. 生产：`SQLAlchemySession` 或 `RedisSession`。
3. 长对话：启用 compaction session 或自定义摘要策略。

业务数据库仍保存：

1. session。
2. messages。
3. generation_jobs。
4. generation_outputs。
5. prompt plans。
6. case provenance。
7. user feedback。

SDK session 负责 agent 运行记忆，业务 DB 负责产品级事实记录。两者不要混为一体。

## 9. Guardrails

### 9.1 Input guardrails

用于：

1. 判断是否涉违法、色情、侵权、隐私。
2. 判断是否缺少授权声明。
3. 判断是否需要澄清。

### 9.2 Output guardrails

用于：

1. 确认最终输出符合 `CreativeRunResult` schema。
2. 确认 prompt plan 没有危险品牌/IP/名人复刻。
3. 确认用户响应不泄露内部 prompt、密钥、provider 配置。

### 9.3 Tool guardrails

最重要。

用于：

1. 防止 `create_image_job()` 在 safety 未通过时执行。
2. 防止 agent 请求未授权 provider。
3. 防止 `request_resource_sync()` 被滥用。
4. 防止读取其他 tenant 的 asset/case/job。

## 10. Tracing 和 Observability

Agents SDK tracing 作为 agent 运行主 trace。

需要额外接入：

1. OpenAI trace dashboard。
2. 自定义 trace processor，写入内部 `agent_traces` 或 OpenTelemetry。
3. 后端 tool span，记录 service latency、provider latency、cost、retry。
4. Resource sync trace，独立于 agent trace。

每次用户任务必须能串起：

```text
session_id
trace_id
agent_run_id
tool_call_id
generation_job_id
provider_request_id
output_id
case_ids_used
```

## 11. Resource Provider 重新定位

ResourceProvider 不再是 agent，也不是 stage。

它是后端资源供应接口：

```python
class ResourceProvider(Protocol):
    provider_id: str
    async def check_updates() -> ProviderChangeSet: ...
    async def fetch_changes(change_set) -> RawResourceBundle: ...
    async def parse(bundle) -> list[PromptCase]: ...
```

中枢通过 tools 使用它的结果：

1. `search_prompt_cases()`。
2. `get_prompt_template()`。
3. `request_resource_sync()`。

第一 provider 仍是：

```text
github.evolink.awesome-gpt-image-2-prompts
```

读取 canonical 英文 `cases/*.md`，不要索引多语言副本。

## 12. Model Provider 重新定位

### 12.1 LLM Provider

Agents SDK 负责 agent LLM 调用。

建议：

1. CreativeManagerAgent 默认使用 OpenAI 强推理模型。
2. 专业 agent 可按成本选择更轻模型。
3. Kimi/Anthropic-compatible 可通过 SDK model provider 或后端 LLM tool 接入。
4. Claude Code/Kimi 不再是主 runtime executor，只作为可选高级策略 tool 或外部中枢模型路径。

### 12.2 Image Provider

图片 provider 保持后端 registry：

```text
ImageGenerationAgent or tool
  -> Generation Service
  -> ImageProviderRegistry
  -> OpenAI image / Gemini image / other
```

原因：

1. 多 provider 能力差异大。
2. 图片任务常异步。
3. 需要对象存储和成本记录。
4. 需要批量、重试、失败恢复。

### 12.3 Embedding Provider

embedding 不建议由 agent 自己决定。

Case Intelligence Service 负责：

1. prompt case embedding。
2. image caption embedding。
3. query embedding。
4. rerank。

Agent 只调用 `search_prompt_cases()`。

## 13. 原 Veyra 思路保留清单

只保留这些思想：

1. 中枢负责策略和仲裁，但不做所有执行。
2. 每个专业能力必须有明确职责。
3. 所有输入输出尽量结构化。
4. 外部 provider 必须有健康、超时、失败降级。
5. 版权/商用/安全 gate 不可被 agent 推翻。
6. 每次运行必须可追溯。
7. 不要求用户读中间报告才能继续主链路。

不保留这些实现形状：

1. 自建 `AgentRegistry` 作为主 agent 框架。
2. 自建 `AgentExecutor` 作为主运行循环。
3. 固定 stage runner。
4. process agent 作为默认专业 agent 形态。
5. PPT 式 `supervisorReport` 字段体系。

如果未来确实要接入外部 process/http/mcp agent，把它们包装成 Agents SDK function tool 或 Hosted MCP tool，而不是反向控制 SDK。

## 15. 实施路线

### Phase 0：SDK Runtime Skeleton

1. 引入 OpenAI Agents SDK。
2. 建 `CreativeManagerAgent`。
3. 建 `SQLAlchemySession` 或 `SQLiteSession`。
4. 建基础 tools：`run_safety_check`、`create_image_job` mock。
5. 打通 streaming 和 tracing。

验收：

1. 用户输入一句话，manager 能调用 tool，创建 mock image job。
2. trace 中能看到 agent run 和 tool call。

### Phase 1：案例库工具化

1. 建 Resource Sync Service。
2. 接入 EvoLinkAI GitHub provider。
3. 建 `prompt_cases`、`prompt_templates`、`case_embeddings`。
4. 暴露 `search_prompt_cases()` 和 `get_prompt_case()` tools。

验收：

1. manager 能根据用户需求检索案例。
2. 输出结果带 case provenance。

### Phase 2：专业 agents as tools

1. `CaseStrategistAgent.as_tool()`。
2. `PromptComposerAgent.as_tool()`。
3. `VisualCriticAgent.as_tool()`。
4. structured output 全部 Pydantic 化。

验收：

1. 智能增强模式可以自动检索案例并生成 prompt plan。
2. 不出现 raw prompt 拼接。

### Phase 3：Handoffs 和多轮炼金

1. `RevisionAgent` handoff。
2. `TemplateGuideAgent` handoff。
3. `VideoPlannerAgent` handoff。
4. session memory 接入版本树。

验收：

1. 用户围绕同一张图多轮修改，agent 能记住上下文。
2. 版本树和 prompt patch 正确落库。

### Phase 4：生产化 guardrails 和 observability

1. input/output/tool guardrails。
2. 自定义 trace processor。
3. provider health dashboard。
4. eval set 和 golden cases。

验收：

1. 高风险 prompt 被拦截。
2. 失败 provider 不影响主链路。
3. 每个输出能追溯 prompt、case、provider、成本、score。

## 16. 新版结论

新版架构应该叫：

```text
OpenAI Agents SDK-first Creative Case Intelligence Platform
```

它和上一版最大的区别：

1. 中枢是 SDK manager agent，而不是外部 process supervisor。
2. 专业 agents 优先用 `agents as tools`。
3. 对话接管才用 handoff。
4. 后端 provider 通过 function tools 暴露。
5. Resource sync 是后台服务，不进入 agent 主循环。
6. 原 Veyra 框架只保留原则，不保留主结构。

这样才能最大化发挥 OpenAI Agents SDK 的优势，同时保留你要的中枢、多 provider、多仓库、多工具、多生图 agent 的开放性。

## 17. 官方依据

本方案参考了 OpenAI Agents SDK 的以下官方能力：

1. Agent 是带 instructions、tools、handoffs、guardrails、structured outputs 的核心单元。
2. `Agent + Runner` 管理 turns、tools、guardrails、handoffs、sessions。
3. 多 agent 常见模式包括 manager as tools 和 handoffs。
4. tools 支持 function tools、hosted tools、agents as tools、MCP 等。
5. tracing 会记录 LLM generations、tool calls、handoffs、guardrails 和 custom events。
6. sessions 支持跨 runs 自动维护对话历史，并有 SQLite、Redis、SQLAlchemy 等实现。
