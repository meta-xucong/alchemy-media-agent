# 密钥与部署配置

这份文档的目标不是“能启动”，而是让一台新 VPS 按照仓库现状部署后，**第一次就能稳定生图**。

本文档基于已经在线验证过的部署形态整理，适合下面这类生产组合：

- 图片 provider：`openai_gpt_image`
- 图片模型：`gpt-image-2`
- OpenAI 兼容网关：`https://aiself.vip/v1`
- 提示词规划 LLM：`anthropic` 路由到 `https://aiself.vip`

如果你用的是官方 OpenAI，而不是兼容网关，可以把 `OPENAI_BASE_URL` 留空。

## 1. 先记住这几个关键坑

### 1.1 `OPENAI_API_KEY` 不等于官方 OpenAI

如果实际可用的是 OpenAI-compatible API，而不是 `https://api.openai.com/v1`，必须明确设置：

```env
OPENAI_BASE_URL=https://aiself.vip/v1
```

不配这项时，图片 provider 会默认直连官方 OpenAI，常见症状是：

- 前端提示 `生成失败：Connection error.`
- 后端 job 里出现 `APIConnectionError`

### 1.2 `gemini_image` 目前不是 live provider

当前仓库里：

- `openai_gpt_image`：可用于真实生图
- `gemini_image`：占位 provider，未接入 live SDK
- `seedance`：占位视频 provider，未接入 live 任务 API

所以生产默认必须保持：

```env
DEFAULT_IMAGE_PROVIDER=openai_gpt_image
OPENAI_IMAGE_MODEL=gpt-image-2
```

### 1.3 运行时改动要能持久化

`docker-compose.yml` 已包含：

```yaml
volumes:
  - ./src_skeleton/.env:/app/.env
  - ./src_skeleton/.media_storage:/app/.media_storage
```

这很重要，因为前端的 `/v1/runtime/provider-settings` 会把运行时改动写回 `/app/.env`。如果不挂这个文件：

- 当前进程里改动会生效
- 但重建容器后会丢

## 2. GitHub Secrets / Variables 一次配齐

### 2.1 Repository Secrets

这些是敏感项，放到 `Settings -> Secrets and variables -> Actions -> Secrets`：

- `OPENAI_API_KEY`
- `ANTHROPIC_AUTH_TOKEN`
- `VPS_HOST`
- `VPS_USER`
- `VPS_SSH_KEY`
- `VPS_PORT`，可选，默认 `22`

### 2.2 Repository Variables

这些是非敏感运行参数，放到 `Settings -> Secrets and variables -> Actions -> Variables`：

```text
OPENAI_BASE_URL=https://aiself.vip/v1
ANTHROPIC_BASE_URL=https://aiself.vip
DEFAULT_IMAGE_PROVIDER=openai_gpt_image
OPENAI_IMAGE_MODEL=gpt-image-2
DEFAULT_LLM_PROVIDER=anthropic
KIMI_LLM_MODEL=kimi-for-coding
IMAGE_WORK_INTENSITY=atelier
```

如果你的生产环境就是当前这套，直接照抄即可。

## 3. GitHub Actions 一步部署

推荐优先使用仓库内置工作流：

`Actions -> Deploy VPS -> Run workflow`

默认输入：

- `deploy_dir=/opt/alchemy-media-agent`
- `app_port=8017`

工作流会完成这些事：

1. 把仓库同步到 VPS
2. 生成 `src_skeleton/.env`
3. 启动 `docker compose`
4. 用 `http://127.0.0.1:${APP_PORT}/v1/providers` 做健康检查

### 3.1 当前工作流默认写入的关键参数

仓库脚本现在已经把下面这些默认值固化进部署逻辑：

```env
MEDIA_AGENT_MODE=live
MOCK_IMAGE_PROVIDER_ENABLED=false
ORCHESTRATION_MODE=runtime_first

DEFAULT_IMAGE_PROVIDER=openai_gpt_image
DEFAULT_IMAGE_MODEL=gpt-image-2
OPENAI_IMAGE_MODEL=gpt-image-2

DEFAULT_LLM_PROVIDER=anthropic
DEFAULT_LLM_MODEL=kimi-for-coding
BACKUP_LLM_PROVIDER=openai
BACKUP_LLM_MODEL=gpt-5.5
OPENAI_LLM_MODEL=gpt-5.5
KIMI_LLM_MODEL=kimi-for-coding
IMAGE_WORK_INTENSITY=atelier

OPENAI_BASE_URL=https://aiself.vip/v1
ANTHROPIC_BASE_URL=https://aiself.vip
```

这套值已经过线上生图验证。

## 4. SSH 到服务器手动部署

如果不是走 GitHub Actions，也可以直接在服务器上：

```bash
export GITHUB_TOKEN="YOUR_GITHUB_TOKEN_WITH_REPO_READ"
export OPENAI_API_KEY="..."
export ANTHROPIC_AUTH_TOKEN="..."
export OPENAI_BASE_URL="https://aiself.vip/v1"
export ANTHROPIC_BASE_URL="https://aiself.vip"
export DEPLOY_DIR="/opt/alchemy-media-agent"
export APP_PORT="8017"

git clone "https://x-access-token:${GITHUB_TOKEN}@github.com/meta-xucong/alchemy-media-agent.git" "$DEPLOY_DIR"
cd "$DEPLOY_DIR"
./scripts/deploy_vps.sh --local
```

如果目录已存在，也可以：

```bash
cd /opt/alchemy-media-agent
git pull --ff-only
APP_PORT=8017 ./scripts/deploy_vps.sh --local
```

## 5. 反向代理与 HTTPS

应用容器建议只监听本机回环端口，再由 Nginx 反代：

- 容器：`127.0.0.1:8017`
- Nginx：对外暴露 `80/443`

一个常用 Nginx 配置片段如下：

```nginx
server {
    listen 80;
    server_name alchemy.example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name alchemy.example.com;

    ssl_certificate     /etc/letsencrypt/live/alchemy.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/alchemy.example.com/privkey.pem;

    client_max_body_size 25m;

    location / {
        proxy_pass http://127.0.0.1:8017;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## 6. 部署后必须做的冒烟验证

### 6.1 基础健康检查

```bash
curl -fsS http://127.0.0.1:8017/healthz
curl -fsS http://127.0.0.1:8017/v1/providers
curl -fsS http://127.0.0.1:8017/v1/runtime/provider-settings
```

期望至少满足：

- `default_image_provider = openai_gpt_image`
- `openai_base_url = https://aiself.vip/v1`
- `openai_api_key_configured = true`
- `anthropic_base_url = https://aiself.vip`
- `anthropic_api_key_configured = true`
- `gemini_image.configured = false`

### 6.2 真生图回归测试

注意：`/v1/sessions` 现在要求 `project_id`，不要只传 `title`。

PowerShell 示例：

```powershell
$session = Invoke-RestMethod `
  -Uri https://your-host/v1/sessions `
  -Method Post `
  -SkipCertificateCheck `
  -ContentType 'application/json' `
  -Body '{"project_id":"smoke","title":"smoke"}'

$body = @{
  session_id = $session.id
  prompt = 'a simple red apple on white background'
  count = 1
  size = '1024x1024'
  quality = 'low'
  output_format = 'png'
} | ConvertTo-Json

Invoke-RestMethod `
  -Uri https://your-host/v1/image/jobs `
  -Method Post `
  -SkipCertificateCheck `
  -ContentType 'application/json' `
  -Body $body | ConvertTo-Json -Depth 20
```

期望结果：

- `status = ready`
- `provider = openai_gpt_image`
- `model = gpt-image-2`
- `outputs` 非空

如果你看到的是：

- `status = failed`
- `error.detail.error_type = APIConnectionError`
- `message = Connection error.`

优先检查的不是模型权限，而是：

1. `OPENAI_BASE_URL` 是否真的指向了你的兼容 API
2. 当前 token 是否就是这条兼容 API 的 token
3. `v1/runtime/provider-settings` 里是否还是空值

## 7. 运行时设置改了以后如何确认已经落盘

前端或 API 调用 `/v1/runtime/provider-settings` 更新后，确认两件事：

### 7.1 当前进程生效

```bash
curl -fsS http://127.0.0.1:8017/v1/runtime/provider-settings
```

### 7.2 宿主机文件也生效

```bash
grep -n 'OPENAI_BASE_URL\|ANTHROPIC_BASE_URL' /opt/alchemy-media-agent/src_skeleton/.env
```

如果 API 里是新值，但宿主机 `.env` 还是旧值，说明你的 compose 没把 `src_skeleton/.env` 挂到 `/app/.env`。

## 8. 故障排查顺序

### 症状：页面提示 `Connection error.`

按这个顺序查：

1. `GET /v1/runtime/provider-settings`
2. 确认 `openai_base_url`
3. `GET /v1/providers`
4. 确认当前 live 的只有 `openai_gpt_image`
5. 直接打一条最小 `POST /v1/image/jobs`
6. 看返回的 job：
   - `status`
   - `provider`
   - `error.code`
   - `error.detail.error_type`

### 症状：误走 Gemini

当前仓库已经做了两层保护：

1. 前端会在 Gemini 不可用时自动切回 GPT
2. 后端 registry 会在 Gemini 不可用时兜回 `openai_gpt_image`

如果页面还像在“走 Gemini”，优先做强刷：

```text
Ctrl + F5
```

## 9. 已验证的生产结论

这套仓库当前最稳的生产用法是：

- 图片：`openai_gpt_image`
- 图片模型：`gpt-image-2`
- OpenAI base URL：`https://aiself.vip/v1`
- 提示词规划：`anthropic` 指向 `https://aiself.vip`
- 视频：先不要当成 live 功能承诺

如果你照着本文档部署，新 VPS 应该可以做到：

1. 首次部署即可成功出图
2. 页面改过的 provider/base URL 能持久化
3. 重建容器后不会丢配置
