# 30. 生产登录门禁与来源回跳开发文档

本文是 `29_Veyra_Sub2API_Alchemy_联动发布方案.md` 的下一步优化文档，专门解决两个生产问题：

1. Alchemy 不能在未登录状态下进入工作台、读取历史、查看配置或管理员页面。
2. 从不同入口进入 sub2api 登录页后，登录成功必须回到用户原本点击的目标：聚合平台回 sub2api，Alchemy 回 Alchemy，首页登录回首页。

本文不改变 sub2api 原有账号、密码、邮箱验证码、2FA、充值、API 中转和网关能力。Alchemy 仍然只通过 Veyra 扩展消费 sub2api 的账号与余额。

## 当前判断

### 已具备的接点

本地代码中已经具备以下基础：

1. Veyra Portal 首页已有 `target` 意图区分：
   - `sub2api-console`
   - `alchemy`
   - `alchemy-mobile`
   - `home`
2. sub2api 登录页已有最小 Veyra 回跳钩子：登录成功后，如果 `redirect` 以 `/_veyra/return` 开头，应使用 `window.location.assign(...)` 重新请求 portal 回跳页。
3. Alchemy 已能消费 sub2api 签发的一次性 ticket，换成本地 Alchemy Veyra session。
4. V1 / V2 生成链路已能基于 Veyra session 获取 sub2api 账户并扣费。
5. 管理员计费设置 API 已要求管理员权限，普通用户不可修改扣费参数。

### 仍需补齐的缺口

1. Alchemy 页面壳和部分只读接口仍偏本地开发模式，未登录可看到工作台或历史数据。
2. 匿名读取历史、输出下载、provider/runtime 设置、orchestrator 状态等接口需要在生产模式收紧。
3. `localStorage` 中的 Alchemy token 只能保护 XHR/API 请求，不能让服务端在首个 `GET /` 时判断用户是否登录。
4. 如果要做到“没登录连页面壳也不能进入”，需要给 Alchemy 增加同域 Cookie 会话，供 V1/V2 服务端在页面请求阶段校验。
5. 登录后跳转 sub2api 的问题通常来自两类原因：
   - 点击 Alchemy 时没有把 `redirect=/_veyra/return?target=alchemy` 完整带到登录页。
   - 登录页成功后忽略了 `redirect`，按默认逻辑跳到了 `/dashboard`。

## 目标流程

所有来源都必须通过 `redirect=/_veyra/return?target=...` 传递意图，不靠 referrer、历史路径或前端猜测。

| 用户动作 | 未登录时进入 | 登录成功后 | 已登录时 |
| --- | --- | --- | --- |
| 首页点“登录” | `/login?redirect=/_veyra/return?target=home` | 回 Veyra 首页 | 保持首页已登录状态 |
| 首页点“聚合平台” | `/login?redirect=/_veyra/return?target=sub2api-console` | 进入 `/dashboard` | 直接进入 `/dashboard` |
| 首页点“Alchemy”桌面版 | `/login?redirect=/_veyra/return?target=alchemy` | 签发 ticket 后进入 `https://alchemy.aiself.vip/?ticket=...` | 直接签发 ticket 并进入 Alchemy |
| 首页点“Alchemy”手机版 | `/login?redirect=/_veyra/return?target=alchemy-mobile` | 签发 ticket 后进入 `https://alchemy.aiself.vip/h5?ticket=...` | 直接签发 ticket 并进入 H5 |
| 直接打开 Alchemy 桌面版 | `https://aiself.vip/_veyra/return?target=alchemy` | 进入 Alchemy | 签发 ticket 并进入 Alchemy |
| 直接打开 Alchemy H5 | `https://aiself.vip/_veyra/return?target=alchemy-mobile` | 进入 H5 | 签发 ticket 并进入 H5 |

关键规则：

1. sub2api 登录页只允许同源相对路径 redirect，禁止外部 URL，避免开放重定向。
2. Alchemy 跨域跳转不由 `redirect` 直接决定，而由 `/_veyra/return` 根据固定 `target` 和服务端配置的 `alchemy_base_url` 决定。
3. ticket 必须短期、一次性、只服务于 Alchemy 登录交换。
4. `target=alchemy-mobile` 和 `target=alchemy` 使用相同账号校验，只是最终路径不同。

## 门禁分层

### 第一层：前端入口门禁

Alchemy 桌面和 H5 初始化时执行：

1. 如果 URL 有 `ticket`，先调用 `/api/v2/veyra/login` 换取 Alchemy session。
2. 如果本地已有有效 Alchemy token，继续进入页面。
3. 如果没有 token，且生产开启强制登录，先跳转到 Veyra return router：
   - 桌面：`https://aiself.vip/_veyra/return?target=alchemy`
   - H5：`https://aiself.vip/_veyra/return?target=alchemy-mobile`
   - 已有 sub2api 登录态时由 return router 直接签发 Alchemy ticket；未登录时再由 return router 进入 `/login?redirect=/_veyra/return?...`。
4. 如果 API 返回 401，清理本地 Alchemy token 和 account 缓存，再回到上述登录入口。

这一层解决用户体验，但不能单独作为安全边界，因为服务端首屏请求无法读取 localStorage。

### 第二层：后端 API 门禁

生产开启 `VEYRA_AUTH_ENABLED=true` 后，以下接口必须要求有效 Alchemy Veyra session。session 可以来自 `Authorization: Bearer <token>`，也可以来自第三层 Cookie。

V1 必须收紧：

```text
/v1/sessions*
/v1/assets*
/v1/image/jobs*
/v1/image/history*
/v1/video/jobs*
/v1/providers
/v1/runtime/provider-settings
/v1/outputs/{output_id}/download
/v1/outputs/{output_id}/thumbnail
/admin/billing
```

V2 必须收紧：

```text
/api/v2/creative/runs*
/api/v2/uploads*
/api/v2/orchestrator/status
/api/v2/task-queue/status
/api/v2/review-agent/status
/api/v2/provider-capabilities
/api/v2/runtime/model-settings
/api/v2/prompt-cases*
/api/v2/case-profiles*
/api/v2/templates
/api/v2/resource-providers*
/api/v2/image/jobs*
/api/v2/image/history*
/api/v2/outputs*
```

允许公开的接口应保持很少：

```text
/healthz
/api/v2/health
/api/v2/veyra/login
/api/v2/veyra/billing/settings/public
/static/*
/mobile-static/*
/share/*  # 仅在确认分享页需要公开时保留，后续建议升级为签名分享链接
```

历史数据可见性：

1. 普通用户只看自己的 Veyra 生成记录。
2. 管理员看所有用户记录。
3. 接入 Veyra 前已有的历史图片不迁移、不改写原始文件；接口返回时标记为 `veyra_legacy_public=true` 和 `record_label=旧版生图记录`，对所有已登录用户可见。
4. 匿名用户在生产模式下不返回历史图片。

### 第三层：Cookie 会话门禁

如果产品要求“没登录不能打开 Alchemy 页面壳”，需要增加 Cookie 会话：

1. `/api/v2/veyra/login` 换票成功后，除返回 JSON token，还设置：

```http
Set-Cookie: alchemy_veyra_session=<signed-session>; Path=/; HttpOnly; Secure; SameSite=Lax
```

2. V1 和 V2 共享同一个 `VEYRA_SESSION_SECRET`，都能校验该 Cookie。
3. 页面请求 `GET /`、`GET /h5`、`GET /mobile` 如果没有有效 Cookie：
   - 桌面跳 `https://aiself.vip/_veyra/return?target=alchemy`
   - H5 跳 `https://aiself.vip/_veyra/return?target=alchemy-mobile`
4. `GET /admin/billing` 只返回外挂计费控制台页面壳，不展示 sub2api 后台；页面内读取和修改计费配置时调用 `/api/v2/veyra/billing/settings`，该 API 必须要求 sub2api 管理员身份。
5. API 校验优先级：
   - 有 `Authorization` 就用 header。
   - 没有 header 再读 Cookie。
   - 两者都没有则 401。
5. 增加 `/api/v2/veyra/logout` 清理 Cookie，并让前端同步清理 `localStorage.alchemy_veyra_access_token`。

Cookie 使用 host-only `alchemy.aiself.vip` 即可，不需要设置跨主域 `Domain=.aiself.vip`。sub2api 与 Alchemy 不共享 Cookie，跨系统登录仍通过一次性 ticket 完成。

## sub2api 改造点

只允许保留最小外挂接点：

1. `extensions/veyra/portal/**`
   - 首页、手机版首页、portal JS、样式。
   - `routeTargets` 必须包含 `sub2api-console`、`alchemy`、`alchemy-mobile`、`home`。
2. `backend/internal/veyra/**`
   - portal config。
   - login-ticket 创建和交换。
   - internal account / debit API。
3. `frontend/src/views/auth/LoginView.vue`
   - 只保留登录成功后的 redirect 适配。
   - 原邮箱、密码、验证码、OAuth、2FA、条款逻辑不改。
4. 路由和配置接点：
   - `backend/internal/server/router.go`
   - `backend/internal/server/http.go`
   - `backend/internal/config/config.go`
   - wire 注入文件。

登录页适配原则：

```text
redirect 不存在 -> /dashboard
redirect = /_veyra/return?... -> window.location.assign(redirect)
redirect = 其他安全相对路径 -> router.push(redirect)
redirect = 外部 URL 或 // 开头 -> /dashboard
```

`sub2api-adapted` 更新官方 upstream 后，必须能通过 overlay 重新覆盖这些文件，不能把 Veyra 改动散落到原版业务模块里。

### V2 backend


需要补齐：

4. 对输出下载、缩略图、反馈、修订按 owner/admin 校验。

### 前端状态

桌面和 H5 统一：

```text
ticket -> /api/v2/veyra/login -> token + cookie -> load account/history/usage
401 -> clear local token/account -> redirect to sub2api login by target
logout -> /api/v2/veyra/logout -> clear local token/account -> Veyra home
```

用户界面不显示扣费比例和系统参数；这些仍只在管理员设置页展示。

## 配置项

建议新增或固化：

```text
VEYRA_AUTH_ENABLED=true
VEYRA_REQUIRE_UI_AUTH=true
VEYRA_LOGIN_BASE_URL=https://aiself.vip
VEYRA_ALCHEMY_DESKTOP_TARGET=alchemy

VEYRA_SESSION_SECRET=<long-random-secret>
VEYRA_SESSION_TTL_SECONDS=2592000
```

其中：

1. `VEYRA_AUTH_ENABLED` 控制账号桥和 API 强认证。
2. `VEYRA_REQUIRE_UI_AUTH` 控制页面级跳转；本地开发可关闭。
3. `VEYRA_LOGIN_BASE_URL` 只允许配置为可信 sub2api/Veyra 主站。
4. `VEYRA_SESSION_SECRET` 必须和 provider key、sub2api JWT secret 分开。

### 第二章：Alchemy session Cookie

验收目标：

1. ticket 换 session 后响应包含 `Set-Cookie`。
2. 无 Authorization 但有 Cookie 时，V1/V2 API 可以识别用户。

### 第三章：Alchemy 后端 API 收紧

验收目标：

1. 匿名访问 V1/V2 历史返回 401。
2. 匿名下载输出返回 401 或 404，不泄漏私有记录。
3. 普通用户只能读自己的记录和标注为“旧版生图记录”的 legacy public。
4. 管理员能读全部账号记录和全部“旧版生图记录”。
5. 生成接口匿名调用仍为 401，余额不足仍 fail-closed。

### 第四章：Alchemy 前端强制登录

验收目标：

1. 直接打开 `https://alchemy.aiself.vip/` 未登录会去 sub2api 登录。
2. 直接打开 `https://alchemy.aiself.vip/h5` 未登录会去 sub2api 登录。
3. 登录成功后自动回到对应桌面/H5。
4. 已登录刷新页面不跳走。
5. token 过期会清理状态并重新登录。

### 第五章：生产回归

必须覆盖：

1. `https://aiself.vip/` 首页正常。
2. `https://aiself.vip/v1/models` 仍是 API 网关，不被首页覆盖。
3. sub2api 登录、邮箱验证码、2FA、充值不受影响。
4. Alchemy V1/V2 真生图正常。
5. V1/V2 扣费分别按管理员配置金额执行。
6. 余额不足不调用 image provider。
7. 管理员入口只对 sub2api 管理员显示。

## 发布与回滚

发布顺序：

1. sub2api 先发回跳修正。
2. Alchemy 先发 Cookie/session 和 API 门禁。
3. 最后打开 `VEYRA_REQUIRE_UI_AUTH=true`。

回滚顺序：

1. 先关闭 Alchemy `VEYRA_REQUIRE_UI_AUTH`，恢复页面可打开。
2. 如 API 门禁误伤，再关闭 Alchemy `VEYRA_AUTH_ENABLED` 或回滚 Alchemy 容器。
3. sub2api 不需要回滚账号系统；只回滚 Veyra portal/login hook 即可。

数据安全：

1. 不删除 sub2api 用户、余额、充值和 idempotency 数据。
2. 不删除 Alchemy `.media_storage`、V2 history、usage JSONL。
3. Cookie/session 只是登录态，不是账务数据。

## 结论

可以做到“从 sub2api 点击登录跳 sub2api，从 Alchemy 点击登录跳 Alchemy”。正确做法是把来源作为受控 `target` 写入 sub2api 登录页的 `redirect`，登录成功后回到 `/_veyra/return`，再由 Veyra Portal 根据 target 执行下一跳。

同时，Alchemy 的“未登录也能进入”要按安全层级处理：前端门禁解决体验，API 门禁保护数据和扣费，Cookie 门禁才是真正阻止页面壳被未登录用户打开的方案。生产推荐三层都做。
