# 22. 2.0 Agent化案例库与中枢调度方案

## 1. 总结

2.0 的核心不应只是“提示词模板库”，而应升级为：

```text
中枢大脑
  -> 管理资源 provider
  -> 定时同步外部案例库和工具能力
  -> 指挥专业 agents 结构化案例
  -> 根据用户需求检索案例和提炼风格
  -> 生成可执行 PromptPlan
  -> 调度图片 provider 生图
  -> 自动评价、修订、沉淀反馈
```

当前 GitHub 仓库 `EvoLinkAI/awesome-gpt-image-2-API-and-Prompts` 作为第一个 `ResourceProvider` 接入。未来可以继续接入更多 GitHub 仓库、本地授权案例库、私有模板库、搜索 API、设计工具、辅助生图 agent、评测 agent。

关键原则：

1. 不是训练模型，而是构建“结构化案例库 + 多模态检索 + prompt 组合 agent + 质量闭环”。
2. 中枢大脑拥有最高调度权，但所有动作必须通过契约化 JSON 输出，由系统执行和审计。
3. 中枢可以决定召回哪些案例、调用哪些 agent、是否重跑某阶段，但不能绕过版权、商用、隐私和安全门禁。
4. 资源 provider、模型 provider、agent provider 三类 provider 必须分开建模，避免后续扩展混乱。
5. 第一版要以可落地为先，先把案例解析、标签、检索、模板化、prompt 合成跑通，再逐步加入更强的视觉评价和自动学习。

1. `Asset`：用户上传或用于最终生成的图片/文件。
2. `PromptCase`：外部案例，主要用于学习和匹配，不一定能直接用于最终输出。
3. `StyleSignal`：从案例中抽取的风格、构图、光线、色彩、用途等机器可读信号。
4. `PromptTemplate`：可交给用户选择和填写变量的产品化模板。

## 4. 三类 Provider

### 4.1 ResourceProvider

负责提供案例、模板、参考图、外部资源。

示例：

1. GitHub prompt case repo。
2. 本地授权案例库。
3. 公司内部品牌模板库。
4. Openverse/Wikimedia/Unsplash/Pexels 等素材源。
5. 设计工具导出的模板包。
6. 其他生图社区的公开 prompt 数据。

统一能力：

```json
{
  "providerId": "github.evolink.gpt-image-2-prompts",
  "kind": "resource",
  "resourceTypes": ["prompt_case", "output_image", "case_metadata"],
  "syncModes": ["manual", "scheduled", "webhook"],
  "licensePolicy": "cc0-with-third-party-risk-notice",
  "supportsIncrementalSync": true,
  "defaultSchedule": "0 */6 * * *",
  "timeoutSeconds": 120,
  "enabled": true
}
```

### 4.2 ModelProvider

负责 LLM、图片、视频、embedding、moderation。

这部分沿用现有 `docs/09_模型供应商适配层.md`，但要补充：

1. `EmbeddingProvider` 必须进入 MVP，因为案例匹配依赖 embedding。
2. 图片 provider 和 LLM provider 的密钥只能由后端 tool 调用，不让 agent 直接持有。
3. Kimi 作为 LLM provider 已在 Veyra 中验证，可以复用配置方式。

### 4.3 AgentProvider

负责接入外部专业 agent。

示例：

1. Claude Code 中枢 agent。
2. Prompt curator agent。
3. Visual critic agent。
4. Safety reviewer agent。
5. 第三方生图优化 agent。
6. 图像审美评分 agent。

统一能力沿用 Veyra 的 `AgentManifest`：

```json
{
  "agentId": "claude-creative-orchestrator-process",
  "kind": "external",
  "adapter": "process",
  "capabilities": ["control.orchestration", "case.routing", "prompt.strategy"],
  "stages": ["orchestrator"],
  "inputSchema": "CreativeOrchestratorContext@2.0",
  "outputSchema": "CreativeOrchestratorDecision@2.0",
  "timeoutSeconds": 420,
  "costTier": "high"
}
```

## 5. 中枢大脑设计

### 5.1 定位

中枢大脑叫 `ChiefCreativeOrchestrator`。

它是最高控制面，负责：

1. 用户意图判断。
2. 资源 provider 管理与同步策略。
3. 案例召回策略。
4. 风格学习策略。
5. prompt 组装策略。
6. 模型 provider 选择原则。
7. 生成候选策略。
8. 自动评价与返工策略。
9. 最终结果解释和交付摘要。

### 5.2 运行模式

建议保留三种模式：

| 模式 | 说明 | 适用 |
| --- | --- | --- |
| `runtime_first` | 默认高频路径，由轻量 runtime agents 执行，中枢只在关键节点介入 | 普通生图 |
| `claude_supervised` | 中枢先出策略，runtime agents 执行 | 高价值商业图、复杂需求 |
| `claude_first` | 中枢全程强调度，多轮检索、候选、评价、返工 | 高端定制、批量 campaign |

不要让 Claude Code 处理每个高频请求的全部细节。它适合做高层判断、复杂策略、资源调度、候选仲裁，不适合无差别进入所有普通生图任务。

### 5.3 中枢输出契约

借鉴 Veyra 的 `orchestratorPlan/stageCommands/qualityGates`，但改成图片平台字段：

```json
{
  "orchestratorPlan": {
    "mode": "claude_supervised",
    "objective": "生成高端香水电商广告图",
    "autonomyLevel": "high",
    "stageCommands": []
  },
  "resourceSyncDirectives": [],
  "caseRetrievalDirectives": [],
  "styleLearningDirectives": [],
  "promptAssemblyDirectives": [],
  "generationStrategy": {},
  "candidateStrategy": {},
  "revisionStrategy": {},
  "qualityGates": {},
  "riskDirectives": {}
}
```

其中：

1. `resourceSyncDirectives`：是否需要先检查某 provider 更新。
2. `caseRetrievalDirectives`：召回哪些类目、风格、用途、排除哪些风险。
3. `styleLearningDirectives`：从哪些案例中提取色彩、构图、光线、镜头、文字风格。
4. `promptAssemblyDirectives`：如何组合用户需求和案例精华，哪些字段必须保持用户优先。
5. `generationStrategy`：生图 provider、数量、画幅、质量、参考图角色。
6. `candidateStrategy`：是否多候选、多风格、多 provider 对比。
7. `revisionStrategy`：评价不达标时如何修订。
8. `qualityGates`：文字准确、品牌一致、主体完整、合规、审美评分等。
9. `riskDirectives`：商标、名人、版权、肖像、低俗、敏感内容处理。

### 5.4 Claude Code + Kimi 接入方式

复用 Veyra 的做法：

1. 中枢作为 process agent 接入。
2. 主系统通过 stdin 传入 JSON request。
3. 中枢通过 stdout 返回 JSON response。
4. Kimi 以 Anthropic-compatible 协议喂给 Claude Code。
5. 设置从后端环境注入，不从前端传真实 key。

环境注入形状：

```json
{
  "SUPERVISOR_AGENT_ENGINE_CLASS": "external_agents.creative_orchestrator.claude_engine:ClaudeCreativeOrchestratorEngine",
  "CLAUDE_SUPERVISOR_CLI": "claude",
  "CLAUDE_SUPERVISOR_MODEL": "kimi-for-coding",
  "CLAUDE_SUPERVISOR_SETTINGS_JSON": {
    "env": {
      "ANTHROPIC_BASE_URL": "https://api.moonshot.ai",
      "ANTHROPIC_AUTH_TOKEN": "${KIMI_API_KEY}",
      "ANTHROPIC_API_KEY": "${KIMI_API_KEY}",
      "CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC": "1",
      "CLAUDE_CODE_ATTRIBUTION_HEADER": "0"
    },
    "model": "kimi-for-coding"
  }
}
```

实际实现时，字段名可以从 `SUPERVISOR_*` 调整为 `CREATIVE_ORCHESTRATOR_*`，避免 PPT 项目语义污染。

推荐新命名：

```text
CREATIVE_ORCHESTRATOR_ENGINE_CLASS
CREATIVE_ORCHESTRATOR_CLI
CREATIVE_ORCHESTRATOR_SETTINGS_JSON
CREATIVE_ORCHESTRATOR_MODEL
CREATIVE_ORCHESTRATOR_TIMEOUT_SECONDS
CREATIVE_ORCHESTRATOR_FILE_ORCHESTRATION
```

## 6. Resource Sync Layer

### 6.1 职责

资源同步层负责：

1. 定时检查 provider 是否更新。
2. 只拉取增量变化。
3. 对外部内容做版本化。
4. 解析、去重、归一化、风险标记。
5. 生成 embedding 和索引。
6. 将稳定版本发布给 runtime 查询。

### 6.2 同步流程

```text
Provider Watcher
  -> Change Detector
  -> Fetcher
  -> Parser
  -> Normalizer
  -> Enricher
  -> License & Risk Gate
  -> Embedding Indexer
  -> Template Productizer
  -> Publish Knowledge Index
```

### 6.3 同步触发

支持三种方式：

1. `manual`：后台按钮或 CLI 手动触发。
2. `scheduled`：APScheduler/Celery Beat/cron 定时触发。
3. `webhook`：GitHub webhook 或未来 provider 回调。

第一版建议：

1. 每 6 小时检查 GitHub 最新 commit SHA。
2. 如果 SHA 未变，只记录 heartbeat，不重新解析。
3. 如果 SHA 变化，拉取 `cases/*.md`、`README.md`、`data/ingested_tweets.json`、必要图片 metadata。
4. 按 case hash 做增量更新。

### 6.4 同步状态

```text
idle -> checking -> fetching -> parsing -> enriching -> indexing -> published
                         |          |           |          |
                         -> failed  -> failed   -> failed  -> failed
```

失败策略：

1. 同步失败不影响已发布索引。
2. 单个案例解析失败进入 `quarantine`，不阻断整个 provider。
3. provider 连续失败超过阈值标记 unhealthy，中枢可选择跳过或降级。
4. 每次同步必须保存 `syncTrace`，方便追查格式变更。

## 7. 第一个 Provider：EvoLinkAI GitHub 案例库

### 7.1 接入策略

Provider ID：

```text
github.evolink.awesome-gpt-image-2-prompts
```

源：

```text
https://github.com/EvoLinkAI/awesome-gpt-image-2-API-and-Prompts
```

优先读取：

1. `cases/ad-creative.md`
2. `cases/character.md`
3. `cases/comparison.md`
4. `cases/ecommerce.md`
5. `cases/portrait.md`
6. `cases/poster.md`
7. `cases/ui.md`
8. `data/ingested_tweets.json`

不要把多语言副本全部入库。只取 canonical 语言版本，建议先用英文 `cases/*.md`。中文 UI 可以在结构化后再翻译标题、用途和标签。

### 7.2 解析内容

从 Markdown 中解析：

1. case number。
2. title。
3. category。
4. source URL。
5. author。
6. output image URL。
7. raw prompt。
8. prompt arguments，例如 `{argument name="product name" default="Soda"}`。
9. case anchor。
10. image directory。

从 `ingested_tweets.json` 补充：

1. tweet URL。
2. author handle。
3. suggested title。
4. category。
5. image dir。
6. added_at。

### 7.3 风险策略

仓库 license 是 CC0，但仍需做第三方风险标记：

1. 原作者归属可能不完整。
2. prompt 中可能出现真实品牌、logo、名人、IP。
3. output image 可作为参考和缩略图展示，但商业使用需保留 provenance。
4. 直接复制 prompt 不如提取风格结构。

默认处理：

```json
{
  "license": "CC0-1.0-repository-declared",
  "thirdPartyRisk": "medium",
  "finalUseAllowed": false,
  "styleLearningAllowed": true,
  "templateReuseAllowed": true,
  "requiresAttributionDisplay": true
}
```

解释：

1. `styleLearningAllowed=true`：可以抽象风格、构图、语言结构。
2. `templateReuseAllowed=true`：可作为用户可选模板，但要去品牌化、变量化。
3. `finalUseAllowed=false`：原 output image 不直接作为最终商用素材。

```json
  "promptSlots": [],
  "styleSignals": {},
  "useCases": [],
  "risk": {},
  "license": {},
  "status": "published"

```

### 8.2 StyleSignal Schema

```json
{
  "visualStyle": ["commercial photography", "premium product ad"],
  "useCase": ["ecommerce_main_image", "social_ad"],
  "industry": ["beauty", "food", "fashion", "consumer_goods"],
  "composition": ["centered product", "left text right product", "9-panel storyboard"],
  "lighting": ["soft diffused studio", "warm cinematic side light"],
  "camera": ["macro close-up", "low angle", "top-down flat lay"],
  "palette": ["black gold", "cream beige", "high-saturation tropical"],
  "textStrategy": ["headline area", "badge copy", "feature list"],
  "materialKeywords": ["glass", "marble", "condensation", "metallic gold"],
  "qualityKeywords": ["photorealistic", "8K", "shallow depth of field"],
  "bestFor": ["luxury product ad", "xhs cover", "product storyboard"],
  "avoidFor": ["strict logo reproduction", "regulated medical ads"],
  "templateSlots": ["product_name", "headline_text", "brand_logo", "aspect_ratio"]
}
```

### 8.3 结构化策略

不要只靠 regex。推荐两级结构化：

1. 规则解析：稳定抽取标题、图片、prompt、argument slots。
2. LLM Enrichment：用 Kimi 或轻量 LLM 把 raw prompt 转成 `StyleSignal`。

LLM enrichment 必须有 schema fallback：

1. 如果 JSON 不合法，使用修复器。
2. 如果字段缺失，使用规则标签补齐。
3. 如果风险命中，打 `quarantine`，不进入前台模板。

### 8.4 Embedding 策略

每个案例至少建三类向量：

1. `semantic_embedding`：标题 + 用户用途 + styleSignals + raw prompt 摘要。
2. `prompt_embedding`：raw prompt 及结构化 prompt 精华。
3. `visual_embedding`：output image 或图片 caption 的 embedding。

如果暂时没有多模态 embedding provider，先用图片 caption 替代视觉向量。

### 8.5 模板产品化

模板不是 raw prompt。模板应由 `Template Productizer Agent` 生成：

```json
{
  "templateId": "tpl_ecommerce_tropical_soda_ad",
  "caseId": "github.evolink.ecommerce.115",
  "displayName": "热带饮料广告海报",
  "previewImage": "...",
  "slots": [
    {"key": "product_name", "label": "产品名", "required": true, "default": "Soda"},
    {"key": "headline_text", "label": "主标题", "required": true},
    {"key": "palette", "label": "主色调", "required": false}
  ],
  "guardrails": [
    "不要保留原案例品牌或 logo",
    "用户未提供授权时不要生成真实商标"
  ],
  "promptBlueprint": {}
}
```

## 9. Runtime Generation Layer

### 9.1 模板模式

用户选择案例模板时：

```text
Template Selected
  -> Slot Filling
  -> User Customization Merge
  -> Safety/Risk Check
  -> PromptPlan
  -> Image Provider
  -> Critic
  -> Revision
```

核心规则：

1. 模板提供风格结构，不提供不可控复制。
2. 用户填写变量优先级高于案例默认值。
3. 真实品牌、logo、名人、IP 需要授权或风控提示。
4. 模板 prompt 版本化，生成结果可追溯到模板版本。

### 9.2 智能增强模式

用户没有选模板时：

```text
User Prompt
  -> Intent Classifier
  -> Case Retrieval
  -> Style Signal Fusion
  -> Prompt Composer
  -> Safety/Risk Check
  -> Image Generation
  -> Visual Critic
```

检索不是只取最相似，而是建议取：

1. `primary_matches`：语义最相关。
2. `style_matches`：视觉风格最接近。
3. `use_case_matches`：用途相同，比如电商主图、海报、头像。
4. `diversity_matches`：提供差异化选择，避免审美趋同。

Prompt Composer 不能简单拼接 prompt。它必须输出：

```json
{
  "userIntent": {},
  "selectedCaseSignals": [],
  "promptPlan": {},
  "finalPrompt": "",
  "negativeConstraints": [],
  "whyThisStyle": "",
  "trace": {
    "caseIds": [],
    "styleSignalsUsed": []
  }
}
```

## 12. 查漏补缺和设计修正

### 12.1 不要把案例库当训练集

706 个案例更适合作为检索和模板库，不适合训练模型。正确做法是：

1. 结构化案例。
2. 抽取风格信号。
3. 检索相似案例。
4. 合成新 prompt。
5. 记录反馈持续优化排序。

### 12.2 不要把中枢放进所有高频路径

Claude Code + Kimi 可以做最高控制面，但高频请求需要成本和延迟控制。

建议：

1. 普通请求走 `runtime_first`。
2. 模板选择和简单增强由轻量 agents 完成。
3. 高价值、复杂、批量或多轮失败任务才升级到 `claude_supervised`。

### 12.3 不要混淆资源授权和风格学习

外部案例图可以用于预览和风格学习，不代表可以作为最终商业素材直接使用。

需要区分：

1. `styleLearningAllowed`
2. `templateReuseAllowed`
3. `finalUseAllowed`
4. `requiresAttributionDisplay`

### 12.4 不要让 raw prompt 直接面向客户

客户应该看到可理解的模板卡片和变量表单，不是复制一大段英文 prompt。

### 12.5 需要多样性机制

只按相似度召回会导致结果趋同。召回排序应混合：

1. 相关性。
2. 用途一致。
3. 风格匹配。
4. 审美质量。
5. 多样性。
6. 风险低。
7. 历史满意度。

### 12.6 风控必须在 prompt 合成前后各做一次

前置检查：用户需求和素材是否高风险。  
后置检查：最终 prompt 是否引入了原案例中的品牌、logo、名人、IP、水印、敏感词。

### 12.7 定时同步不能直接覆盖线上索引

新版本应先进入 staging index：

```text
current published index
new staging index
  -> validation
  -> smoke retrieval
  -> publish switch
```

这样 provider 格式变动不会把线上检索打坏。

## 13. MVP 实施顺序

### Phase 1：基础接入

1. 增加 `ResourceProvider` registry。
2. 实现 EvoLinkAI GitHub provider。
3. 解析 canonical `cases/*.md`。
4. 入库 `prompt_cases`。
5. 抽取 prompt slots。
6. 保存 output image metadata。
7. 建基础文本检索。

验收：

1. 能同步 7 个主 case 文件。
2. 能识别新增/更新 case。
3. 能展示模板图库。
4. 每个模板有来源、作者、风险、变量槽。

### Phase 2：案例智能化

1. `StyleSignalExtractorAgent`。
2. `TemplateProductizerAgent`。
3. embedding 索引。
4. 智能增强模式。
5. Prompt Composer 输出结构化 PromptPlan。

验收：

1. 用户输入简单需求，能召回相关案例。
2. 输出不是 raw prompt 拼接，而是结构化组合 prompt。
3. 生成记录能追溯使用了哪些案例信号。

### Phase 3：中枢大脑

1. 接入 `ChiefCreativeOrchestrator` process agent。
2. 复用 Veyra 的 Claude Code + Kimi settings injection。
3. 中枢输出 `caseRetrievalDirectives/styleLearningDirectives/promptAssemblyDirectives`。
4. 高价值任务可走 `claude_supervised`。

验收：

1. Claude/Kimi 能输出可执行 JSON。
2. JSON 能影响案例召回、prompt 合成、候选策略。
3. 中枢不可用时自动 fallback 到 runtime agents。

### Phase 4：质量闭环

1. Visual Critic。
2. 自动 prompt patch。
3. 多候选比较。
4. 用户反馈学习。
5. 模板质量排行榜。

验收：

1. 低分结果能给出修订建议。
2. 常用高满意模板能自动靠前。
3. 批量任务能按用途和风格分组展示。

## 14. 推荐最终主流程

```text
用户输入/选择模板
  -> RuntimeManager 判断模式
  -> 如需中枢，ChiefCreativeOrchestrator 出策略
  -> MaterialAnalyzer 分析用户素材
  -> CaseRetrieval 召回案例
  -> StyleSignalFusion 提炼风格
  -> PromptComposer 生成 PromptPlan
  -> SafetyLicenseGovernor 前后置检查
  -> ProviderSelector 选择图片模型
  -> ImageGeneration 生成候选
  -> VisualCritic 评分
  -> RevisionAgent 自动或用户驱动炼金
  -> Gallery + Trace + Template Feedback
```

## 15. 结论

这个方向有价值，而且和现有 AlchemyOS 文档、Veyra Agent 工程经验可以很好合并。

建议把 2.0 定义为：

```text
Agentic Creative Case Intelligence Platform
```

也就是“agent 化创意案例智能平台”。

第一性目标不是让 agent 记住案例，而是让系统持续同步案例、理解案例、检索案例、抽象案例、组合案例，并把这个能力放进可控、可审计、可扩展的中枢调度框架中。
