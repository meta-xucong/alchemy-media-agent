# 03. 技术路线与选型

## 1. 总体技术路线

采用“控制面 + 运行面”的双层架构：

- 控制面：Claude Code / Claude Agent SDK，负责复杂需求拆解、开发与运维自动化、跨 agent 调度、代码/配置生成、任务审查、Hook 与 MCP 集成。
- 运行面：OpenAI Agents SDK，负责用户请求中的专业 agents、tools、handoffs、sessions、guardrails、tracing。
- 媒体模型面：图片和视频模型通过 provider adapter 接入，默认图片模型为 OpenAI `gpt-image-2`，支持 Gemini 图片模型等备选；视频预留 Seedance/Veo 等异步任务式 provider。

## 2. 为什么用 OpenAI Agents SDK 做底层

OpenAI Agents SDK 的 Agent 是带 instructions、tools、handoffs、guardrails、structured outputs 的 LLM 单元，Runner 会循环处理模型输出、tool calls、handoff 和 final output。它适合把“需求分析、素材分析、提示词优化、生成调用、结果评价、继续修改”拆成多个专业 agent，同时保留统一 tracing 和 session 记忆。

## 3. 为什么用 Claude Code 做中枢

Claude Code/Claude Agent SDK 的优势在于代码库理解、文件读写、命令执行、subagents、hooks、MCP、权限控制。对本项目最有价值的不是让它替代所有业务请求，而是作为“中枢调度/控制面”：

- 管理专业 agents 的定义、prompt、工具权限。
- 在复杂任务中派发 subagents 并汇总结果。
- 自动生成/修复 provider adapter、测试、文档。
- 通过 hooks 做确定性审计、密钥拦截、安全检查。
- 通过 MCP 连接内部工具：对象存储、模型控制台、数据看板、提示词仓库。

## 4. 两种部署模式

### 模式 A：推荐生产模式

用户请求进入 Backend API，直接调用 OpenAI Agents SDK Manager Agent。Claude Code 作为后台控制面，用于复杂研发、运维、审核和低频复杂任务。

优点：低延迟、可控、成本低、稳定。缺点：不完全符合“所有请求都由 Claude Code 中枢先调度”的字面要求。

### 模式 B：严格中枢模式

用户请求进入 Claude Code Orchestrator，由它根据任务调用 OpenAI Agents Runtime、素材解析工具、图片/视频 provider、评测工具。

优点：中枢能力强，复杂任务拆解优秀。缺点：成本、权限、延迟、供应商稳定性和生产隔离复杂度更高。

建议实现为策略开关：`ORCHESTRATION_MODE=runtime_first|claude_first`。

## 5. 推荐技术栈

| 层 | 推荐 | 说明 |
|---|---|---|
| 前端 | Next.js + React + SSE/WebSocket | 类 ChatGPT UI、画廊、批量进度 |
| 后端 | Python 3.12 + FastAPI | 与 OpenAI Agents SDK、Claude Agent SDK 生态契合 |
| Agent 运行时 | OpenAI Agents SDK | manager + specialist agents |
| 中枢控制面 | Claude Code / Claude Agent SDK | subagents、hooks、MCP、代码/配置/任务调度 |
| 队列 | Redis + RQ/Celery/Arq | 图片/视频异步任务 |
| DB | PostgreSQL | 会话、消息、任务、成本、版本树 |
| 对象存储 | S3/R2/OSS/TOS | 原始素材、缩略图、输出图片/视频 |
| 向量库 | pgvector / Qdrant | 品牌资产库、素材记忆，MVP 可不做 |
| 观测 | OpenTelemetry + Sentry + Prometheus | trace、错误、性能、成本 |
| 部署 | Docker Compose → Kubernetes | MVP 简化，后续可扩展 |

## 6. 模型与 provider 选型

### LLM provider

- OpenAI：默认用于 Agents SDK 主流程、结构化输出、工具调用。
- Anthropic：Claude Code/Claude Agent SDK 中枢。
- Gemini：备用多模态理解、图片生成、视频生成生态。
- OpenAI-compatible provider：通过 base_url + api_key 接入；必要时使用 LiteLLM/Any-LLM，但要做能力探测。

### 图片 provider

- 默认：OpenAI GPT Image 2。
- 备选：Gemini Nano Banana / Gemini Image 模型。
- 扩展：其他支持 text-to-image/image-to-image/edit 的 API。

### 视频 provider

- 预留：Seedance / Veo / 其他 API。
- 统一接口：创建任务、查询任务、取消任务、下载结果、获取成本。
- 视频任务默认异步，不进入同步 HTTP 请求。

## 7. 模型切换机制

所有模型调用都不直接写死在 agent 中，而是走 `ProviderRegistry`：

```text
agent -> tool -> MediaService -> ProviderRegistry -> ProviderAdapter -> Vendor API
```

每个 provider 声明能力：

- `modalities`: text/image/video/file。
- `operations`: generate/edit/variation/analyze/upscale。
- `sync_mode`: sync/async/stream。
- `limits`: max_batch、max_resolution、max_duration、file types。
- `cost_profile`: 单价、估算方式。
- `safety_profile`: provider 自带审核能力与本地审核策略。

## 8. 文件输入路线

MVP 使用统一上传接口，所有文件进入对象存储。解析策略：

- 图片：保存原图 + 缩略图 + perceptual hash + 视觉摘要。
- PDF：保留原文件，必要时转页面图；多模态模型可提取文本和页面图。
- PPT/DOC：提取文本；如需保留视觉，转换 PDF 后进入 PDF 解析流程。
- 表格：解析前 1000 行摘要 + 字段类型 + 样例行；大表用后台任务和分页。
- 短视频：抽帧、转音频、ASR、镜头摘要；如果 provider 支持直接视频理解，可走 direct analyze。

## 9. 工程约束

- 所有 provider adapter 必须支持 mock。
- 所有生成任务必须有 idempotency key。
- 所有输出必须落对象存储，不依赖短期 URL。
- 所有用户素材和输出必须记录授权与保留策略。
- 所有模型调用必须记录 trace_id、model、provider、参数、输入摘要、输出摘要、成本。
