# 生成图片预览加载优化开发文档

## 背景

上一轮历史分页已经降低了 V1/V2 历史列表的首屏数据量，但用户点击生成图片进入放大预览时仍然偏慢。审计前端链路后确认：桌面端和移动端的 `openImageLightbox` 都优先把原图 `url` 传给图片元素，缩略图 `thumbnailUrl` 只是后备项。对于 VPS 带宽较小、原图体积较大的场景，弹窗会先等待原图下载，造成明显慢感。

## 第一阶段方案

先改前端预览加载顺序：

1. 打开 lightbox 时优先显示已有缩略图。
2. 同时在后台预加载原图。
3. 原图加载成功后，如果用户仍在看同一张图，再把缩略图替换为原图。
4. 下载、分享、继续修改、星标、历史数据不变。

## 第二阶段方案

为了避免 lightbox 最终仍然拉取多 MB 原图，本次同时落中尺寸预览图：

1. V1 在 `.media_storage/previews` 生成 1600px 内的 WebP 预览图，并提供 `/v1/outputs/{output_id}/preview`。
2. V2 在 V2 原生 `storage_dir/previews` 生成 1600px 内的 WebP 预览图，并提供 `/api/v2/image/history/{output_id}/preview`。
3. V1/V2 history schema 均新增可选 `preview_url`，旧记录没有该字段时会自动补 endpoint。
4. 前端 lightbox 先显示 thumbnail，再加载 preview；原图只用于下载和分享。

V2 preview 完全由 V2 原生存储和接口提供，不调用 V1 API，不读取 V1 存储。

## 验收标准

1. 桌面端和移动端 lightbox 均先使用缩略图候选。
2. 中尺寸 preview 会替代原图作为 lightbox 展示图，降低打开预览的传输量。
3. 分享和下载仍读取完整原图 URL。
4. 前端 JS 语法检查通过。
5. 前端静态服务烟测通过。
