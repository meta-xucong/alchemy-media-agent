# V2 视觉语法锁开发优化方案

## 背景

Custom Media Agent 2.0 已经具备 Template Lock、上传素材绑定、Claude Code 中枢和 V2-native 生图链路。但近期实测暴露出一个通用问题：当用户选定案例模板并上传成品图、菜单图或复杂参考图时，系统会部分继承模板装饰，却可能让上传素材的整体版式压过模板的主体框架、构图重心和设计语言。

本优化不把模板内容原样照搬，也不让 Claude 完全自由发挥。目标是建立一层稳定的视觉语法约束：

```text
模板控制视觉语法。
用户控制语义内容。
上传素材提供证据和硬身份约束。
信息密集海报必须保留业务关键信息。
Claude 自动适配，但不能让素材布局覆盖模板视觉语法。
```

## 核心定义

视觉语法锁锁定的是案例模板或自动锚点案例的视觉表达方式，而不是锁定其具体主题内容。

必须保留或强继承：

- 主体框架：主视觉、信息区、辅助图区、装饰区的大关系。
- 构图重心：画面主角、说明区、行动信息的空间优先级。
- 视觉层级：主视觉 > 标题/品牌 > 内容模块 > 辅助信息。
- 氛围和设计语言：光影、材质、色调、商业质感、版式节奏。
- 背景密度和空间感：不能因上传图拥挤而破坏模板留白和层级。
- 排版类型：海报、食谱卡、产品主图、杂志封面、信息图等。

允许替换或重写：

- 模板中的具体商品、食物、人物、道具。
- 模板原始文案、品牌名、标题内容。
- 模板原始 logo、icon、小图内容。
- 模板原主题。例如天妇罗模板可变为健康周餐菜单，但仍保留食谱海报式视觉语法。

## 两种运行模式

### 1. 显式模板：Template Visual Grammar Lock

当 `template_case_id` 存在时，用户手选模板是最高优先级视觉语法来源。

规则：

- `selected_case_ids[0]` 必须是手选模板。
- 其他案例不能改变主构图，只能提供兼容局部参考。
- 上传素材默认只能填充可替换语义内容，不能继承其整体版式。
- 如果上传素材缺少模板要求的关键主视觉，Claude 必须生成符合用户主题的虚拟内容补齐。

### 2. 无模板：Auto Visual Grammar Lock

当用户未指定模板时，也不能让 Claude 从零开始自由发挥。系统必须自动选择一个主视觉语法锚点。

规则：

- 从召回案例或 Claude 选择结果中确定 1 个 `primary_grammar_anchor`。
- 主锚点控制构图、美感、复杂度和设计语言。
- 最多允许 1-2 个辅助案例提供局部风格、材质、光影或行业细节。
- 不允许多个案例平均融合成无主构图。
- 强度低于手选模板，但仍必须让输出具备明确构图和设计复杂度。

## 上传素材策略

用户不应手动配置大量参数。系统需要自动判断上传素材在视觉语法锁中的用途：

- `subject_reference`：具体商品、人物、食物身份参考。
- `logo_reference`：logo 形状、比例、颜色和目标表面。
- `qr_source`：二维码像素级保留，优先走 deterministic crop overlay。
- `content_source`：菜单、文案、信息来源。
- `finished_poster_source`：成品海报或整页设计，只取语义内容，不取整体版式。
- `style_reference`：兼容风格证据，不能压过模板或主锚点。

当上传图表现为成品海报、菜单表、周卡、信息表、截图、整页设计时，不应默认绑定为 `subject_identity`。这类素材应作为内容证据和硬资产来源，不得成为主画面版式来源。

## 信息完整性锁

视觉语法锁不能只优化构图。对于海报、菜单、周卡、活动页、信息图、商品政策页等信息密集任务，V2 还必须自动启用 Information Integrity Lock。

触发条件：

- 上传素材被判定为 `composite_content_source`。
- 用户明确要求完整保留文案、价格、优惠、购买政策、二维码或业务信息。
- 请求和上传图显示为菜单、周卡、优惠页、购买说明、价格表、套餐说明等信息密集场景。

硬要求：

- 保留主商品/食物图片或等价主视觉。
- 保留条目名称、关键文案和品牌/业务含义。
- 保留价格、数量、日期、热量、卡数等数字信息。
- 保留套餐、优惠、赠品、加购、配送、购买规则、续卡政策等影响购买决策的信息。
- 保留二维码、CTA、联系方式、扫码说明。
- 允许压缩措辞，但不能丢失事实。

画幅策略：

- 空间不足时，优先扩大竖版/poster 画幅。
- 或增加有序信息模块、卡片、信息带、页脚政策区。
- 不得为了画面简洁而删除购买优惠、价格套餐、配送规则、二维码等关键内容。

优先级：

```text
模板控制视觉框架。
信息完整性控制内容槽填充。
上传整图不能控制旧布局，但其业务事实不能被随意丢弃。
```

## Prompt 编译顺序

最终给图像模型的 prompt 应由程序编译，Claude 提供适配内容，但不可打乱优先级。

固定顺序：

```text
[Visual Grammar Lock]
继承手选模板或自动主锚点的主体框架、构图重心、视觉层级、氛围、光影、背景密度、排版节奏和设计语言。

[User Semantic Content]
用户提示词、业务主题、具体主体、文案、品牌、二维码和上传素材语义成为画面内容。

[Asset Usage Rules]
上传图片是证据和 slot 变量。硬身份素材必须保持输入图引用；成品图/菜单图不得继承原布局。

[Information Integrity Lock]
信息密集海报必须完整保留业务关键信息；空间不足时扩展画幅或增加模块，而不是删除信息。

[Conflict Policy]
模板或主锚点的视觉语法胜过 Claude 草稿和上传图布局。缺失关键视觉元素时自动生成符合主题的虚拟内容补齐。
```

## Claude 中枢要求

Claude 仍是 V2 核心创意中枢，但它的自由度应被约束在正确边界内：

- Claude 可以决定如何把用户语义内容适配到视觉语法中。
- Claude 可以生成缺失的虚拟主视觉、道具、背景细节和信息组织方式。
- Claude 不得把上传整图的布局、背景、满屏网格或截图结构当作模板框架。
- Claude 不得把手选模板改成仅有颜色/边框相似的弱套皮。
- 无模板时，Claude 必须选择一个主视觉语法锚点，而不是平均融合多个案例。
- 信息完整性锁启用时，Claude 必须优先规划信息分区，不能为了简化输出而丢掉优惠政策、价格、购买规则、二维码或关键文案。

## 质量门禁

生成后应逐步增加自动质检风险项：

- `template_visual_grammar_weak`：只有颜色/边框像模板，核心构图不像。
- `template_core_structure_missing`：模板主视觉或核心层级缺失。
- `uploaded_layout_overcopied`：上传成品图/菜单图整体版式被照搬。
- `auto_anchor_missing`：无模板模式没有主视觉语法锚点。
- `semantic_subject_lost`：用户主体被模板原主题吞掉。
- `hard_identity_missing`：产品、logo、人脸、二维码等硬约束未保真。
- `information_content_missing`：信息密集海报丢失关键文案、价格、优惠、购买政策、二维码或主图。

首阶段先在 prompt 计划和 metadata 中显式记录视觉语法锁合同，为后续视觉审查和自动重试提供依据。

## 当前代码定向改造

首阶段落地范围：

1. 新增 `app/services/visual_grammar_lock.py`。
2. 显式模板和无模板模式都构建 `visual_grammar_contract`。
3. `compose_prompt_plan` 在最终 prompt 中优先注入视觉语法锁。
4. `user_variables` 持久化 `visual_grammar_contract`，便于历史诊断。
5. `output_review` 读取视觉语法锁信息并增加审查说明。
6. `asset_binding` 对成品图/菜单图/周卡/整页图提示词做降权，避免默认 `subject_identity` 压过模板框架。
7. `information_integrity_contract` 随视觉语法合同生成，并进入 Claude、最终 prompt、负面约束、provider 参数和 review。
8. 补充测试覆盖：
   - 手选模板时，Claude 的弱 prompt 不能覆盖模板视觉语法。
   - 无模板时，系统仍建立自动主锚点。
   - 成品菜单图上传时，资产绑定提示不得要求继承上传图整体布局。
   - 信息密集海报启用信息完整性锁，默认使用更大的竖版画幅，并把遗漏优惠政策等作为风险。

后续阶段：

- 接入 OCR/布局检测，把 `content_source`、`qr_source`、`food_evidence` 拆成更强结构化素材。
- 对 live 输出做视觉审查和自动重试。
- 增加前端诊断面板，展示当前视觉语法锁来源和强度。
