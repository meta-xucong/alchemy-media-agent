# 12. 开发任务拆解与里程碑

## 里程碑总览

建议按 6 个里程碑推进。

```text
M0 文档与脚手架
M1 独立后端基础
M2 ResourceProvider 同步和案例结构化
M3 Agents SDK Runtime 和核心 tools
M4 前端 2.0 工作台和模板库
M5 生图闭环、评测、安全与上线
M6 Veyra 账号桥与 sub2api 扣费闭环
```

## M0 文档与脚手架

目标：

1. 完成 2.0 文档包。
2. 确认独立目录和模块边界。
3. 确认 API、数据模型、Agent 契约。

交付：

1. 本文档包。
2. OpenAPI 初稿。
3. JSON contract 示例。
4. Mermaid 架构图。

## M1 独立后端基础

任务：

1. 新建 2.0 backend module。
2. 增加 `/api/v2/health`。
3. 增加独立配置加载。
4. 增加独立 DB schema/table migration。
5. 增加独立 Redis namespace。
6. 增加对象存储 prefix。
7. 增加 trace project 配置。

验收：

1. 旧版 API 不受影响。
2. `/api/v2/health` 可独立返回。
3. 关闭 v2 backend 不影响旧版。

## M2 ResourceProvider 同步和案例结构化

任务：

1. 实现 provider manifest。
2. 实现 GitHub adapter。
3. 实现 markdown parser。
4. 实现 case normalizer。
5. 实现 safety/license classifier。
6. 实现 embedding indexer。
7. 实现 sync scheduler。

验收：

1. 能同步 EvoLinkAI 仓库。
2. 能解析 `cases/*.md`。
3. 能生成 `PromptCase`。
4. 能按 category/use_case/style 检索。
5. 同步失败不破坏上一版 active index。

## M6 Veyra 账号桥与 sub2api 扣费闭环

目标：

1. sub2api 继续作为唯一账号、登录、充值和余额系统。
2. Veyra Agent 首页替代 sub2api 默认首页，但不影响 `/api/*`、`/v1/*`、`/v1beta/*`。
3. Alchemy 通过一次性 ticket 继承 sub2api 登录态。
4. Alchemy 生成前按配置扣 sub2api 余额。
5. Alchemy 普通用户只展示当前账号自己的生成记录、余额、扣费记录，以及所有账号可见的“旧版生图记录”；管理员可以看到所有用户生成记录。

sub2api 任务：

1. 保持 Veyra portal 静态资源独立在 `extensions/veyra/portal`。
2. Go embed 运行副本保持在 `backend/internal/veyra/portal_dist`。
3. `PortalMiddleware` 只接管 `/`、`/_veyra/*` 和 `/_veyra/return`。
4. Veyra API 只放在 `backend/internal/veyra`，并通过 `backend/internal/server/router.go` 注册。
5. 登录页仅增加 `/_veyra/return` 专用 reload 钩子，不改账号、密码、验证码、2FA 或 JWT 逻辑。
6. 更新 `docs/CUSTOM_PATCHES.md`，确保未来 upstream 更新后可恢复自定义补丁。

Alchemy 任务：

1. `POST /api/v2/veyra/login` 交换 sub2api ticket。
2. `GET /api/v2/veyra/me` 读取 sub2api 余额。
3. `GET /api/v2/veyra/history` 和 `GET /api/v2/veyra/usage` 按用户隔离。
4. 创意运行和直接 image job 从 Authorization 解析用户，不信任请求体用户 ID。
5. 生成前通过 sub2api debit API 扣费，余额不足 fail closed。
6. 桌面和 H5 前端都消费 `?ticket=...` 并自动附加本地 Alchemy Veyra session。

验收：

1. sub2api: `go test ./internal/veyra ./internal/config ./internal/server ./cmd/server`。
2. sub2api frontend: `npm run typecheck`。
3. Alchemy V2: `python -m pytest -q`。
4. 前端脚本: `node --check` 覆盖桌面、H5、Veyra portal。
5. V2 隔离扫描无 `/api/v1`、`/v1/`、V1 bridge 标记。
6. 全流程本地模拟：未登录点击入口、登录回跳、ticket 登录、余额展示、扣费成功、余额不足阻断。

## 开发顺序建议

先做数据和服务边界，再接 SDK runtime，最后做前端体验。

原因：

1. Agent 需要稳定 tool contract。
2. 案例库质量决定 prompt 质量。
3. 前端需要稳定 API。
4. 独立性必须从第一天建立。
