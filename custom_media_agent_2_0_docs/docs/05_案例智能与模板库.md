# 05. 案例智能与模板库

## 目标

把外部案例从“Markdown 和图片集合”变成可被 agent 检索、理解、解释和安全复用的结构化资产。

## 落地决策

案例库采用“在线同步源 + 本地结构化索引”的方式，不在用户每次检索时实时读取 GitHub。

原因：

1. GitHub 仓库适合作为 provider 原始来源，但不适合作为实时检索库。
2. 本地索引可以提前解析 prompt、图片、标题、分类、标签、风险和质量分。
3. 中枢 Agent 调度多个 provider 时，需要统一的 `PromptCase` 结构，而不是直接面对每个仓库的文件格式。
4. 本地索引可支持模糊搜索、标签过滤、排序、反馈权重和后续向量检索。
5. GitHub 不可用或同步失败时，系统可以继续使用上一版已发布索引。

当前 2.0 先使用 `.v2_data/case_index.json` 作为可审计的本地发布索引；后续案例数量扩大、provider 增多后，升级为 SQLite/Postgres + 向量索引，但 API 和 Agent 工具契约保持不变。

同步策略：

```text
GitHub provider -> archive adapter -> Markdown parser -> PromptCase normalizer
-> local published index -> search/template API -> CreativeManager Agent
```

默认用户浏览读取本地全量模板；用户输入关键词时，检索服务只返回存在语义重叠、案例画像匹配或模糊命中的案例，避免把无关案例混入“智能匹配”结果。

## 同步模式

2.0 后端保留两类同步入口，但前端默认不暴露工程按钮，由服务启动和后台 worker 自动检查：

1. `恢复内置种子`：读取程序包内的小型 seed 案例，不访问网络。用途是离线兜底、调试、或者远端同步失败后的最小可用状态。
2. `检查 GitHub 更新`：检查 EvoLinkAI 仓库 `main` 分支 commit。如果本地索引已经是同一 commit，直接跳过下载；如果 commit 变化，才下载 GitHub archive、重新解析 Markdown、更新本地索引。

程序启动时执行同样的 GitHub 检查流程：

```text
启动 -> 立即加载本地 case_index.json -> 后台检查 GitHub commit
  -> commit 未变：跳过，继续使用本地索引
  -> commit 已变：下载 archive，解析 704+ 案例，原子替换本地索引
  -> GitHub 不可用：保留上一版本地索引
```

因此，案例库不是每次都从 GitHub 现场读取。正常运行时都从本地索引检索，只有发现远端 commit 变化才更新本地索引。

## 特征标签检索

案例解析和检索层会把 prompt 归类为统一特征标签，例如：

1. `subject.perfume`
2. `subject.bottle`
3. `material.glass`
4. `color.black`
5. `color.gold`
6. `tone.luxury`
7. `composition.hero`
8. `use.ecommerce`

用户中文输入会先映射到同一组特征标签，再筛选案例。比如 `奢华香水 电商主图 黑金 玻璃瓶` 会优先匹配香水、瓶身、玻璃、黑金、高端质感、电商主图等特征。

这里不能做成“遇到一个词补一个词”的关键词表，而要按通用画像维度识别需求：

1. 主体/行业：食品饮料、香水美妆、宠物、服装时尚、家具家居、旅行城市、金融银行、教育课程、音乐演出、游戏科幻、婚礼情侣、儿童母婴、数码电子、花植自然等。
2. 用途：电商、广告、海报、社媒、包装、旅行宣传、时尚用途、金融用途、教育用途、信息图、故事板、房产空间等。
3. 风格：高级奢华、极简干净、电影感、赛博霓虹、复古、科技未来、医疗洁净、可爱童趣、优雅精致、暗黑氛围、动漫二次元、水彩、3D 渲染、国风东方等。
4. 材质/色彩/构图：玻璃、金属、织物、木质、纸张、陶瓷、皮革、水与液体；黑、金、红、白、银、紫、蓝、绿、粉；主图、俯拍、文案留白、近景特写、拼贴、信息图、故事板等。

检索时必须同时满足“画像命中或明确文本证据”。完全没有相关证据的中文查询返回空列表，不再因为案例质量分高就混入无关案例。规则无法覆盖的新词后续再交给 LLM 归类，并把新标签沉淀回本地索引。

## 案例画像标准

每个案例都会被抽象为 `CaseProfile`，供搜索、前端解释和 OpenAI Agents SDK tool 使用。当前先从已解析 prompt、标签和 visual features 本地派生，避免每次用户搜索都实时调用 LLM 扫描 700+ 案例。后续同步 worker 可以把 Claude Code 或 V1 思考模型生成的画像写回 `visual_features.case_profile`。

画像维度：

1. 主体：香水、护肤、腕表、食品饮料、UI、珠宝、鞋履、汽车、建筑空间等。
2. 风格：高级奢华、极简干净、电影感、赛博霓虹、复古、科技未来、医疗洁净等。
3. 用途：电商、海报、广告、社媒、包装等。
4. 材质：玻璃、金属、织物等。
5. 色彩：黑、金、绿、蓝、粉等关键色。
6. 光影：棚拍布光、霓虹、自然光等。
7. 构图：主图构图、俯拍平铺、文案留白等。
8. 可复用原则：构图、光影、色彩、材质、情绪、字体空间等。
9. 风险提醒：商标、肖像、受保护 IP、原图不可直用等。

画像来源可运行时切换：

1. `rules`：本地画像评分，速度最快，适合实时搜索。
2. `claude-code`：由 Claude Code 中枢作为最高审美判断来源，适合离线或后台沉淀高价值标签。

## PromptCase

`PromptCase` 是案例智能的核心实体。

必填字段：

1. `case_id`。
2. `provider_id`。
3. `source_url`。
4. `title`。
5. `category`。
6. `raw_prompt`。
7. `summary`。
8. `prompt_atoms`。
9. `visual_features`。
10. `use_cases`。
11. `style_tags`。
12. `risk_tags`。
13. `license_policy`。

## Prompt Atoms

Prompt 不直接作为一整段复用，而要拆为 atoms：

1. `subject`：主体。
2. `scene`：场景。
3. `composition`：构图。
4. `lighting`：光影。
5. `color_palette`：色彩。
6. `material_texture`：材质。
7. `camera_lens`：镜头。
8. `mood`：情绪。
9. `brand_tone`：品牌语气。
10. `typography`：字体和文字处理。
11. `post_processing`：后期效果。
12. `constraints`：限制和禁止项。

## Visual Features

用于检索和匹配的视觉特征：

1. 场景类型。
2. 主体类别。
3. 构图结构。
4. 背景复杂度。
5. 光源类型。
6. 色彩对比。
7. 景深。
8. 是否含文字。
9. 是否适合商品展示。
10. 是否适合广告转化。

## Visual Signals

V2 在 `visual_features` 和 `prompt_atoms` 之上增加一层轻量 `visual_signal_brief`。它不是新的中枢，也不替代 Claude Code，而是给中枢和本地兜底 PromptComposer 提供更高质量的案例证据。

提炼目标：

1. 背景氛围：米色、白底、黑底、冷蓝、粉色等主背景信号。
2. 关键点缀：小面积但重要的金色、墨绿、银色、红色、霓虹蓝等强调色。
3. 材质质感：玻璃、液体、金属、织物、纸张、陶瓷、木质、皮革等。
4. 光影结构：棚拍柔光、轮廓光、逆光、霓虹光、强对比戏剧光等。
5. 构图骨架：中心主图、近景细节、俯拍平铺、海报留白、多卡片/信息图等。
6. 审美方向：高级奢华、极简干净、商业编辑、电影感、科技未来、复古等。

使用原则：

1. Claude Code 中枢读取 `candidate_case_details.json` 时，必须把 `visual_signal_brief` 当作案例视觉 DNA 证据。
2. 有手选模板时，视觉信号服务于模板锁定，帮助中枢保留原型构图、强调色、材质边缘和版式层级。
3. 无手选模板时，视觉信号帮助 Claude 判断哪些案例真正相关，以及哪些可复用元素值得组合进最终 prompt。
4. Claude 不可把 `case_id`、`asset_id`、`provider_id`、`source_url`、API 或仓库工程标识写进最终生图 prompt。
5. Claude 不可只迁移背景主色；如果案例的质感来自小面积金属边、墨绿标签、玻璃高光或深色对比，也应判断是否抽象迁移。

## 模板定制模式

模板定制不是复制原 prompt，而是保留可复用结构：

```text
模板案例 -> 风格骨架 -> 用户变量 -> 安全过滤 -> 新 prompt
```

可复用：

1. 构图方式。
2. 光影策略。
3. 色彩节奏。
4. 画面层次。
5. 文案区域规划。
6. 材质与质感描述。

不可默认复用：

1. 原图中的具体人物。
2. 第三方品牌标识。
3. 未授权商标。
4. 可识别肖像。
5. 可能受版权保护的角色。
6. 原 prompt 的大段逐字表达。

## 智能增强模式

用户没有选择模板时：

1. 中枢判断需求。
2. CaseStrategistAgent 生成召回计划。
3. 检索服务取 top K。
4. PromptComposerAgent 提炼组合。
5. Safety Service 检查。
6. 生成 `ImagePromptPlan`。

召回策略要平衡：

1. 语义相关性。
2. 视觉风格相关性。
3. 用途相关性。
4. 多样性。
5. 授权安全。
6. 案例质量。

## 检索排序

推荐综合分：

```text
score =
  semantic_similarity +
  case_profile_match +
  fuzzy_text_match +
  style_match +
  use_case_match +
  category_match +
  profile_depth +
  quality_score
```

当前实现给 `case_profile_match` 更高权重，以提升中文短词、风格词和用途词的召回稳定性；完全无相关证据的搜索返回空列表，不再把全部案例原样展示给用户。

如果用户选择模板，模板案例固定为 primary case，其余案例只能作为补充灵感。

## 前端展示

模板卡片应展示：

1. 预览图。
2. 标题。
3. 用途标签。
4. 风格标签。
5. 可定制字段。
6. 授权状态。
7. 风险提示。

详情页展示：

1. 原案例摘要。
2. 可复用风格。
3. 不建议复用内容。
4. 推荐输入字段。
5. 相似案例。

## 反馈沉淀

用户反馈应反哺：

1. 案例质量分。
2. 召回排序。
3. prompt atom 权重。
4. 模板推荐。
5. 供应商选择。

反馈不能直接自动修改 provider 原始案例，只能修改本地索引和运营标签。
