# 18. 稳定化、可观测与质量闭环开发文档

> 当前状态：本文仍作为稳定化、可观测和质量闭环路线使用。旧文档中关于 `1.0 live image bridge` 的表述已经被 V2 Strict Isolation 原则替代；当前 V2 必须使用 V2 原生 provider、V2 原生存储、V2 原生历史和 V2 原生复检链路。

## 1. 目标

本阶段目标不是扩张新玩法，而是把 2.0 从“闭环可演示”推进到“稳定可部署、用户看得懂、结果可复盘”。

核心验收线：

1. Claude Code 中枢调用可观测。
2. Claude Code 中枢调用具备缓存、重试和明确 fallback 原因。
3. 前端能向用户解释本次生成到底用了哪个中枢、选了哪些案例、提炼了什么策略。
4. 生成结果能进入质量评估和后续自动返工链路。
5. 所有改动保持 2.0 与 1.0 功能隔离，只共用前端入口和视觉 shell，不再依赖 1.0 live image bridge。

## 2. 分章落地路线

### 第 1 章：Claude 中枢稳定性底座

后端能力：

1. `CreativeOrchestratorDecision` 增加调用元数据：
   - `invocation_status`
   - `latency_ms`
   - `attempts`
   - `cache_hit`
   - `cache_key`
   - `workspace_id`
2. `CreativeOrchestratorDecision` 增加 Claude 最终 prompt 包：
   - `final_prompt`
   - `negative_prompt`
   - `provider_parameters`
   - `prompt_rationale`
3. Claude 成功路径下，PromptComposer 不再重新扩写主 prompt，而是校验并承接 `final_prompt`；只有缺失、违规或 fallback 时才用本地合成逻辑。
4. Claude 同步规划路径增加缓存：
   - 相同用户需求、模板、输出参数和候选案例命中缓存时，不再重复调用 Claude CLI。
   - 缓存只保存脱敏后的 Claude 原始决策 JSON。
5. Claude 同步规划路径增加近重复语义缓存：
   - 仅当同模式、同模板、同输出参数、候选案例高度重合，并且文本相似度超过高阈值时复用。
   - 默认阈值 `0.92`，只覆盖标点、空格和轻微措辞差异，不跨产品或跨风格复用。
6. Claude inline schema 最小化：
   - Claude 只输出最终 prompt 包、短理由、选择案例和置信度。
   - 本地从 `final_prompt` 和 `prompt_rationale` 补齐解释层字段，避免 Claude 填写冗余 `prompt_directives`。
7. Claude 调用增加有限重试：
   - 空 stdout、非 JSON、CLI 异常、超时都进入可记录失败原因。
   - 超过最大次数后 deterministic fallback。
   - `claude_output_token_limit` 属于结构化输出长度失控，不重复重试，立即 deterministic fallback。
8. 增加中枢状态接口：
   - `GET /api/v2/orchestrator/status`
   - 返回启用状态、工具模式、缓存数量、最近调用、成功率、平均延迟。

Kimi / sub2api 兼容要求：

1. 如果 Claude Code 当前模型链路实际落到 Kimi，上游日志里的 `context canceled` 不应直接理解为“Kimi 返回了确定 502”。更合理的判断是：下游调用方或 sub2api 请求上下文先取消，上游请求被动中断。
2. 2.0 Python 层的 Claude CLI subprocess timeout 默认是 240 秒，不属于 2-6 秒短超时。因此如果日志显示 2-6 秒取消，重点应检查 Claude Code 所接入的模型网关、sub2api、反向代理或客户端 read timeout。
3. 程序层必须把以下失败单独归类并记录：
   - `kimi_context_canceled`
   - `upstream_context_canceled`
   - `kimi_upstream_error`
   - `kimi_sub2api_502`
   - `claude_output_token_limit`
4. `kimi_context_canceled` 属于瞬时链路抖动类型，优先走重试、缓存和 fallback model，而不是让用户生成失败。
5. 生产建议：
   - `V2_CLAUDE_ORCHESTRATOR_MAX_ATTEMPTS=2`
   - `V2_CLAUDE_ORCHESTRATOR_RETRY_DELAY_SECONDS=2`
   - 将 sub2api / Kimi 客户端 timeout 提到至少 20-30 秒。
   - 给 Kimi provider 增加备用 key 或同组账号，形成真正 failover。
   - 如果 VPS 到 Kimi 直连不稳定，为 Kimi 链路配置稳定出口或代理。
   - 将 `V2_CLAUDE_ORCHESTRATOR_MAX_OUTPUT_TOKENS` 保持在 8192 左右，避免 Claude Code/Kimi 在结构化输出前因 wrapper token cap 中断；最终 prompt 长度仍由 CLI 级 JSON schema 和本地清洗控制。
   - 不要求 Claude 输出完整推理链；只输出最终 prompt 包、短理由和必要参数，从源头规避 output token limit。
   - 同步快速模式保持 `V2_CLAUDE_ORCHESTRATOR_DISABLE_SLASH_COMMANDS=true`，减少 Claude Code 非必要命令系统开销。

自检：

1. 单元测试证明第一次调用执行 Claude，第二次相同请求命中缓存。
2. 单元测试证明 Claude 失败会记录 fallback 原因和 attempts。
3. 单元测试证明 `api.kimi.com ... context canceled` 会被归类为 `kimi_context_canceled` 并可重试恢复。
4. 单元测试证明 Claude `final_prompt` 会直接进入 `ImagePromptPlan.prompt`。
5. 单元测试证明近重复 prompt 可以命中 `semantic_cache_hit`，并且不会再次调用 Claude。
6. HTTP smoke 能读取 `/api/v2/orchestrator/status`，并能看到成功 run 的 `prompt_plan.user_variables.prompt_source=claude_final_prompt`。

### 第 2 章：前端中枢分析可解释化

前端能力：

1. 在 2.0 `生成结果` 区新增 `中枢分析` 面板。
2. 面板显示：
   - 中枢来源：Claude Code / deterministic fallback / cache
   - 置信度
   - 耗时
   - 重试次数
   - 是否缓存命中
   - 选案理由
   - 风格策略
   - 负向约束
3. 生成过程中的占位状态更明确：
   - `中枢规划中`
   - `案例融合中`
   - `真实出图中`
   - `已回退`

自检：

1. 前端静态语法检查通过。
2. mock run 后页面展示 Claude/fallback/cache 状态。
3. 文本不溢出，不影响现有案例区和历史区。

### 第 3 章：质量评估与复盘

后端能力：

1. 增加 `ImageReviewDecision` 数据模型。
2. 先实现规则型评估，后续再接 LLM/视觉 agent：
   - 是否 live 输出
   - 是否有 prompt
   - 是否存在 provider 错误
   - 是否符合安全门禁
   - 是否需要人工复核
3. 每个 output 附带 `review_score` 和 `review_notes`。

自检：

1. mock output 和 V2 native live output 都能产生 review。
2. blocked/failed job 不会被误判为高质量。

### 第 4 章：异步 worker 化

目标架构：

```text
v2-api
  接收请求、保存 run、返回 run_id

v2-claude-orchestrator-worker
  负责 Claude Code 中枢规划

v2-image-worker
  负责真实生图

v2-review-worker
  负责质量复检和返工建议
```

本章先落地可单进程运行的 worker 接口和状态模型，不强行引入 Redis/Celery。VPS 生产部署再替换为持久队列。

已落地 MVP：

1. 新增 `POST /api/v2/creative/runs/async`。
2. 接口先保存 `planning` 状态的 `CreativeRun`，返回 `run_id`、`trace_id` 和当前阶段。
3. 后台任务使用同一个 `run_id` 完成 Claude 中枢规划、案例融合、生图和复检。
4. `GET /api/v2/creative/runs/{run_id}` 可作为轮询接口。
5. 前端 2.0 生成按钮已切到异步入口，先拿 `run_id`，再轮询结果。

自检：

1. 同步 API 保持兼容。
2. 新增异步入口可返回 `run_id` 和阶段状态。
3. 前端可以轮询 run。

### 第 5 章：自动返工闭环

能力：

1. 当质量评估低于阈值，生成 `revision_directives`。
2. 用户可点击“一键优化重试”。
3. 自动重试必须保留原始 run、原始 prompt、原始案例和重试理由。

已落地 MVP：

1. 新增 `POST /api/v2/outputs/{output_id}/revisions`。
2. 新增 `POST /api/v2/outputs/{output_id}/revisions/async`，用于 VPS 和前端的一键优化重试。
3. 系统从原始 output、job 和 review 中提取 `source_output_id`、`source_job_id`、`source_run_id`、原始 prompt、review decision、score、revision directives 和用户反馈。
4. 新 run 使用 `mode_hint=revision`，并把完整 `revision_source` 写入 `prompt_plan.user_variables`。
5. 原始 run 和原始 output 不会被覆盖。

自检：

1. 低分 mock 可以触发返工建议。
2. 重试不会覆盖原始输出。

## 3. 当前优先级

本轮优先完成第 1 章、第 2 章和第 3 章规则型 MVP：

1. 这是稳定性和用户理解成本的核心。
2. 对现有生图链路侵入最小。
3. 可以立刻减少 Claude 慢调用、空输出和 fallback 难以解释的问题。

第 4、5 章已完成单进程 MVP。下一阶段不再扩同步 API，而是把后台任务替换为 VPS 生产队列，并把规则型 review 升级为视觉/LLM 复检 agent。

## 3.1 当前生产化增量

已落地：

1. `BackgroundTasks` 已替换为 SQLite 持久任务队列。
2. `v2_tasks` 保存 queued run、请求 payload、执行结果快照、错误和重试次数。
3. API 内联 worker 由 `V2_TASK_QUEUE_INLINE_WORKER_ENABLED` 控制，本地默认开启，VPS 建议关闭。
4. 独立 worker 入口为 `python -m app.workers.task_queue_worker`，PowerShell 启动脚本为 `scripts/run_worker.ps1`。
5. `/api/v2/task-queue/status` 返回队列 DB、inline worker 状态和各状态计数。
6. `GET /api/v2/creative/runs/{run_id}` 可在业务 repository 重置后从队列结果快照恢复已完成 run。
7. `CreativeManagerAgent` 已按 OpenAI Agents SDK 注册为可选工具边界：`case_strategist_search`、`case_detail`、`case_profile`、`prompt_safety_check`。
8. `VisualCriticAgent` 已成为复检 agent 边界，当前以 metadata fallback 运行，避免在未配置真实视觉模型时伪装成像素级审查。
9. `/api/v2/review-agent/status` 可查看复检 agent 当前模式。
10. `/api/v2/runtime/model-settings` 已提供 V2 运行时模型控制面，前端可展示并切换 V2 生图通道、OpenAI Agents SDK 模型、Claude Code 中枢模型、案例理解来源和复检模型。
11. `CaseIntelligenceService` 已加入 `CaseProfile` 画像，搜索结果带 `profile_tags`，并新增 `case_profile` SDK tool 供中枢 agent 读取案例画像。

仍需生产配置完成：

1. VPS systemd/docker 服务定义。
2. Kimi/sub2api 网关侧 20-30 秒以上 timeout。
3. Kimi 备用 key/account failover。
4. Kimi 稳定出口或代理。
5. 真实视觉模型接入 `VisualCriticAgent`。

## 4. 不做事项

本阶段暂不做：

1. Redis/Celery 强依赖。
2. 生产数据库迁移。
3. 新的外部 provider。
4. 自动扣费/配额系统。
5. 大规模 UI 重设计。

## 5. 回滚策略

1. `V2_CLAUDE_ORCHESTRATOR_ENABLED=false` 可关闭 Claude 中枢。
2. `V2_CLAUDE_ORCHESTRATOR_CACHE_ENABLED=false` 可关闭缓存。
3. Claude 失败时 deterministic fallback 保持可用。
4. 前端中枢分析面板只读展示，不影响出图主流程。
