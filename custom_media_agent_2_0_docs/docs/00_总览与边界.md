# 00. 总览与边界

## 版本定位

Custom Media Agent 2.0 是一个独立的智能生图创意系统。它的核心不是“提示词增强器”，而是一个可扩展的创意生产流水线：

```text
客户需求 -> CreativeManagerRuntime -> 案例召回/模板定制/创意编排/Prompt 合成/生图/评价/反馈沉淀
```

2.0 的首个资源来源是 GitHub 仓库 `EvoLinkAI/awesome-gpt-image-2-API-and-Prompts`。系统会把仓库案例结构化为可检索、可匹配、可解释、可复用的 PromptCase。

## 与原版的关系

2.0 仅复用：

1. 同一个域名。
2. 同一个前端 shell 或导航入口。
3. 必要的登录态透传能力，前提是授权边界清楚。

2.0 不复用：

1. 旧版后端业务代码。
2. 旧版数据库表。
3. 旧版任务队列。
4. 旧版对象存储目录。
5. 旧版 provider 配置。
6. 旧版 agent 或 prompt runtime。
7. 旧版日志和 trace project。

## 产品目标

2.0 要解决两个核心场景：

1. 模板定制：客户选择案例模板后，系统帮助客户保留风格骨架，同时替换主体、场景、品牌、尺寸、色彩和用途。
2. 智能增强：客户只输入简单需求时，系统自动匹配高相关案例，提炼 prompt 精华，组合出更高质量的新 prompt，再驱动生图。

## 架构目标

1. `CreativeManagerRuntime` 作为确定性业务主链路。
2. Claude Code Orchestrator 作为创意决策层，输出结构化 `CreativeOrchestratorDecision`。
3. Kimi / 火山 / DeepSeek / 豆包等作为 Claude Code 的模型源或备用源，不作为默认独立总管。
4. OpenAI Agents SDK 作为未来 planner/tool/tracing capsule，不接管当前端到端主链路。
5. 业务服务是可信执行层，agent 或 planner 不能直接访问密钥、数据库、扣费系统和外部供应商。
6. ResourceProvider、ModelProvider、ImageProvider、ToolProvider 都通过清晰 manifest 和 adapter 接入。
7. 后台同步、任务队列、存储、历史和复检独立于 Claude Code 中枢。

## 边界原则

2.0 的边界用四句话定义：

```text
Agent/Orchestrator 负责判断、规划、生成结构化意图。
Domain Service 负责执行、校验、持久化和审计。
Provider Adapter 负责接入外部资源和供应商。
Frontend 负责展示、选择、反馈和必要的人类确认。
```

## 成功标准

第一阶段成功标准：

1. 可以同步并结构化 GitHub 仓库案例。
2. 可以按用途、风格、主体、构图、色彩、行业和风险标签检索案例。
3. 可以展示模板库并支持模板定制。
4. 可以在无模板选择时自动召回案例并合成增强 prompt。
5. 可以记录每次生图的案例来源、prompt 组成、orchestrator trace、供应商、输出和反馈。
6. 可以在不影响旧版功能的前提下，从同域名前端进入 2.0。

## 关键非目标

1. 不做模型训练或微调。
2. 不宣称系统“学习了图片”，而是结构化案例并做检索增强。
3. 不绕过版权、肖像、商用授权和平台安全策略。
4. 不将某个 GitHub 仓库写死进核心逻辑。
5. 不让未来 provider 接入破坏中枢 agent 的统一治理。
