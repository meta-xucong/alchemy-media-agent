# 15. 迁移隔离策略：仅共用域名和前端

## 目标

2.0 是功能独立版本，不是旧版的模块扩展。它只复用同域名和前端入口，让用户体验连续，但工程上保持独立。

## 共享项

允许共享：

1. 域名。
2. 前端 shell。
3. 登录态桥接。
4. 通用 UI 组件。
5. 统一错误页。
6. 统一埋点网关，前提是 event namespace 独立。

## 禁止共享项

禁止共享：

1. 旧版后端 controller。
2. 旧版 service。
3. 旧版数据库表。
4. 旧版 Redis queue。
5. 旧版对象存储目录。
6. 旧版 provider 密钥和配置。
7. 旧版 prompt 模板。
8. 旧版任务状态机。

## 路由隔离

```text
/api/v1/* -> old backend
/api/v2/* -> v2 backend
/v2/*     -> frontend v2 route
```

前端通过 route-level boundary 加载 2.0 状态和 API client。

## 数据隔离

推荐：

1. DB schema：`alchemy_v2`。
2. 表前缀：`v2_`。
3. Redis：`alchemy:v2:*`。
4. Object storage：`v2/*`。
5. Trace project：`alchemy-media-agent-v2`。

任何 migration 都必须通过 dry-run 检查，确认不会改旧表。

## 配置隔离

2.0 所有环境变量使用 `V2_` 前缀。

旧版环境变量不能被 2.0 自动读取，除非显式做兼容映射，并在配置报告中标记。

## 权限隔离

2.0 应有独立 service account。

权限最小化：

1. 只能写 `v2/` object prefix。
2. 只能访问 v2 DB schema。
3. 只能访问 v2 queue namespace。
4. 只能读取必要登录态。

## 灰度策略

1. 先内部账号开放 `/v2`。
2. 再对指定客户开放。
3. 最后按比例开放入口。

灰度开关：

```text
V2_FRONTEND_ENABLED
V2_API_ENABLED
V2_PROVIDER_SYNC_ENABLED
V2_IMAGE_GENERATION_ENABLED
```

## 回滚策略

回滚 2.0：

1. 关闭 `/v2` 入口。
2. 停止 v2 workers。
3. 停止 v2 scheduler。
4. 保留 v2 数据用于排查。

不得执行：

1. 删除旧版数据。
2. 修改旧版路由。
3. 回滚共享前端 shell 到破坏旧版的版本。

## 验收清单

上线前必须证明：

1. v2 后端关闭时旧版可用。
2. 旧版后端关闭时 v2 health 可用。
3. v2 写入都在 v2 namespace。
4. v2 日志和 trace 不混入旧版。
5. v2 失败不会触发旧版降级。
