# 11. 部署配置与运行模式

## 部署目标

2.0 与旧版同域名共存，但后端和运行资源独立。

推荐结构：

```text
same-domain.example.com
  /           -> existing frontend shell
  /v2         -> frontend v2 route
  /api/v1     -> old backend
  /api/v2     -> v2 backend
```

## 服务组件

1. `v2-api`：HTTP API 和 V2 runtime 入口。
2. `v2-worker`：生图任务、评价任务、异步工具。
3. `v2-sync-worker`：ResourceProvider 同步。
4. `v2-scheduler`：定时检查更新。
5. `v2-db`：独立 schema 或独立库。
6. `v2-redis`：独立 namespace。
7. `v2-object-storage`：独立 prefix。
8. `v2-trace-exporter`：trace 和审计。

## 环境变量

### Agent Runtime

```text
V2_AGENT_RUNTIME=hybrid_pipeline
V2_AGENT_TRACE_PROJECT=alchemy-media-agent-v2
V2_AGENT_DEFAULT_MODEL=<model>
```

当前主链路是 `CreativeManagerRuntime` 的确定性业务流水线，OpenAI Agents SDK 只作为可选 planner/tool/tracing 边界。`POST /api/v2/runtime/model-settings` 会将可切换项持久化到 `.v2_data/runtime_model_settings.json`，重启后由 v2-api 加载。

### Image Provider

2.0 必须有独立 provider 配置，不依赖 1.0 后端运行态。推荐把密钥写入 `custom_media_agent_2_0/.env` 或生产环境变量：

```text
V2_IMAGE_GENERATION_PROVIDER=openai_gpt_image
V2_OPENAI_API_KEY=<secret>
V2_OPENAI_BASE_URL=https://aiself.vip/v1
V2_OPENAI_IMAGE_MODEL=gpt-image-2
V2_GEMINI_API_KEY=<optional-secret>
V2_GEMINI_BASE_URL=<optional-base-url>
V2_GEMINI_IMAGE_MODEL=<image-capable-gemini-model>
V2_ALLOW_MOCK_FALLBACK=true
```

本地开发允许从 `CODEX_AUTH_FILE` 或 `%USERPROFILE%\.codex\auth.json` 读取 `OPENAI_API_KEY` 作为兜底，优先级仍低于 `V2_OPENAI_API_KEY`。生产 VPS 不应依赖 Codex 本地 auth 文件，必须显式配置 `V2_OPENAI_API_KEY`。

`scripts/import_v1_provider_env.ps1` 只用于本地迁移：它会把 1.0 `.env` 中可安全映射的 provider 配置复制到 V2 `.env`，并在本地存在 Codex auth 时补齐 OpenAI key。`.env` 必须被 git 忽略。

前端不得只根据“是否存在 API key”判断可用性，必须读取 `GET /api/v2/provider-capabilities`。例如当前 Gemini key 可能可用，但模型只支持文本输出，这种 provider 必须显示为不可用于生图，不能让用户选择后再失败。

### Claude Code Orchestrator

当前实现使用 `V2_CLAUDE_ORCHESTRATOR_*` 配置 Claude Code 创意中枢。Kimi、DeepSeek、Doubao、GLM 等是 Claude Code 背后的模型源或备用源，不是绕开 Claude Code 的独立默认总管。

```text
V2_CLAUDE_ORCHESTRATOR_ENABLED=true
V2_CLAUDE_ORCHESTRATOR_CLI=claude
V2_CLAUDE_ORCHESTRATOR_MODEL=<primary-model-source>
V2_CLAUDE_ORCHESTRATOR_TIMEOUT_SECONDS=240
V2_CLAUDE_ORCHESTRATOR_MAX_OUTPUT_TOKENS=8192
V2_CLAUDE_ORCHESTRATOR_EFFORT=low
V2_CLAUDE_ORCHESTRATOR_DISABLE_SLASH_COMMANDS=true
V2_CLAUDE_ORCHESTRATOR_TOOLS=none
V2_CLAUDE_ORCHESTRATOR_PERMISSION_MODE=bypassPermissions
V2_CLAUDE_ORCHESTRATOR_FALLBACK_MODEL=<optional-fallback-model>
V2_CLAUDE_ORCHESTRATOR_WORKSPACE_DIR=/var/lib/alchemy/v2/claude_orchestrator_runs
V2_CLAUDE_ORCHESTRATOR_CACHE_ENABLED=true
V2_CLAUDE_ORCHESTRATOR_SEMANTIC_CACHE_ENABLED=true
V2_CLAUDE_ORCHESTRATOR_SEMANTIC_CACHE_THRESHOLD=0.92
V2_CLAUDE_ORCHESTRATOR_CACHE_PATH=/var/lib/alchemy/v2/claude_orchestrator_cache.json
V2_CLAUDE_ORCHESTRATOR_MAX_ATTEMPTS=2
V2_CLAUDE_ORCHESTRATOR_RETRY_DELAY_SECONDS=2
```

同步 API 推荐 `V2_CLAUDE_ORCHESTRATOR_TOOLS=none`，通过压缩上下文和结构化输出来取得 `CreativeOrchestratorDecision`。同时保持 `DISABLE_SLASH_COMMANDS=true` 和高阈值 semantic cache，减少 Claude Code 启动开销和近重复请求的重复规划。文件读写和外部工具型 Claude 中枢建议放入独立 `v2-claude-orchestrator-worker`，避免阻塞 HTTP 请求。

如果 Claude Code 的后端模型链路接入 Kimi，`context canceled` 更常见的原因是 sub2api 或下游客户端短超时/断开。2.0 Python 层应保持较长 Claude CLI timeout，并依靠 `MAX_ATTEMPTS`、缓存和 deterministic fallback 兜底；Kimi provider 侧仍建议配置 20-30 秒以上 timeout、备用账号 failover 和稳定出口。

VPS 部署原则：

1. Claude Code Orchestrator 只通过结构化 `CreativeOrchestratorDecision` 影响主系统。
2. `WORKSPACE_DIR` 必须是独立、可清理、不可访问敏感配置的目录。
3. API key、数据库密码、对象存储密钥不得写入 Claude 工作区。
4. 所有 Claude Code 调用必须有 timeout。
5. Claude Code 不可用时按配置 fail closed 或 deterministic fallback，不允许绕过 Template Lock / Asset Binding / Safety。
6. 生产环境建议将 Claude orchestrator 放入单独 worker 或受限容器，而不是直接在 API request thread 中长期阻塞。

### Kimi 兼容接入

保留 Veyra 的高价值思路，但不复制其 executor：

```text
V2_KIMI_API_KEY=<secret>
V2_KIMI_BASE_URL=<url>
V2_KIMI_MODEL=<model>
V2_KIMI_PROTOCOL=anthropic_compatible
```

是否能直接作为 SDK model provider 接入，以当前 SDK model/provider 能力和适配器实现为准。若不满足，作为受控 LLM tool 或 backend service 接入。

### Case Intelligence

```text
V2_CASE_INTELLIGENCE_PROVIDER=rules
V2_CASE_INTELLIGENCE_MODEL=
```

`rules` 适合实时搜索；`claude-code` 表示由 Claude Code 中枢沉淀高价值案例画像。生产环境建议实时搜索仍优先读取本地画像，不在用户每次搜索时同步调用 LLM 扫描全量案例。

### ResourceProvider

```text
V2_RESOURCE_SYNC_INTERVAL_MINUTES=360
V2_RESOURCE_GITHUB_TIMEOUT_SECONDS=30
V2_RESOURCE_SYNC_MAX_RETRIES=3
```

### Task Queue

```text
V2_TASK_QUEUE_DB_PATH=/var/lib/alchemy/v2/task_queue.sqlite3
V2_TASK_QUEUE_INLINE_WORKER_ENABLED=false
V2_TASK_QUEUE_POLL_INTERVAL_SECONDS=1
V2_TASK_QUEUE_CLAIM_TIMEOUT_SECONDS=900
V2_TASK_QUEUE_MAX_ATTEMPTS=3
```

低并发 VPS 初期可使用 SQLite 队列。推荐部署方式：

```text
v2-api:
  python -m uvicorn app.main:app --host 127.0.0.1 --port 8020

v2-worker:
  python -m app.workers.task_queue_worker --worker-id v2-worker-1
```

本地开发可保持 `V2_TASK_QUEUE_INLINE_WORKER_ENABLED=true`，API 进程会自行消费队列，便于调试。生产环境建议关闭内联 worker，并用独立进程消费队列，避免 HTTP 进程被长任务拖慢。

### Review Agent

```text
V2_OUTPUT_REVIEW_AGENT_ENABLED=true
V2_OUTPUT_REVIEW_AGENT_MODEL=<optional-vision-or-review-model>
```

当前 `VisualCriticAgent` 已接入 OpenAI Agents SDK 边界，但默认只使用 provider metadata 和 prompt evidence，不做像素级断言。接入真实视觉模型后，复检结果继续写入 `ImageReviewDecision`，前端和返工链路无需改 schema。

### Storage

```text
V2_OBJECT_STORAGE_PREFIX=v2/
V2_REDIS_PREFIX=alchemy:v2:
V2_DB_SCHEMA=alchemy_v2
```

### Veyra Account And Billing Bridge

Veyra 账号桥默认关闭。只有正式接入 sub2api 账户和余额时才启用：

```text
VEYRA_AUTH_ENABLED=false
VEYRA_SUB2API_BASE_URL=https://aiself.vip
VEYRA_INTERNAL_TOKEN=<same-value-as-sub2api-veyra.internal_token>
VEYRA_SESSION_SECRET=<long-random-secret>
VEYRA_SESSION_TTL_SECONDS=2592000
VEYRA_REQUIRE_UI_AUTH=false
VEYRA_LOGIN_BASE_URL=https://aiself.vip
VEYRA_SESSION_COOKIE_NAME=alchemy_veyra_session
VEYRA_GENERATION_CHARGE_AMOUNT=0
VEYRA_USAGE_PATH=/var/lib/alchemy/v2/veyra_usage.jsonl
```

生产启用时：

1. sub2api 必须设置 `veyra.enabled=true`。
2. sub2api 必须设置 `veyra.portal_enabled=true` 才会由 Veyra Agent 首页接管 `/`。
3. sub2api 必须设置 `veyra.alchemy_base_url=https://alchemy.aiself.vip`。
4. sub2api 的 `veyra.internal_token` 必须和 Alchemy 的 `VEYRA_INTERNAL_TOKEN` 完全一致。
5. `VEYRA_SESSION_SECRET` 必须独立生成，不能复用 sub2api JWT secret 或任何 provider key。
6. `VEYRA_GENERATION_CHARGE_AMOUNT` 大于 0 后，Alchemy 会在生图前扣 sub2api 余额；余额不足时不调用 image provider。
7. 若 `VEYRA_REQUIRE_UI_AUTH=true`，Alchemy 页面壳也必须要求登录；仅靠 localStorage token 不足以服务端拦截首个 `GET /`，因此生产推荐使用同域 HttpOnly Cookie 会话。
8. 如果 V1 运行在 Docker 容器、V2 运行在宿主机 systemd，V1 的 `VEYRA_BILLING_SETTINGS_URL` 不能写成 `http://127.0.0.1:8020/...`，因为容器内的 `127.0.0.1` 指向容器自身。生产建议使用反代后的可达地址：

```text
VEYRA_BILLING_SETTINGS_URL=https://alchemy.aiself.vip/api/v2/veyra/billing/settings/public
VEYRA_BILLING_RULE_KEY_V1=alchemy:v1
```

发布后必须从 V1 容器内部验证该地址可访问，否则 V1 会在生成前因读取计费规则失败而停止，不会扣费也不会调用生图 provider。

生产登录门禁和来源回跳细节见：

```text
custom_media_agent_docs/docs/30_生产登录门禁与来源回跳开发文档.md
```

本地验证建议：

1. 先保持 `VEYRA_AUTH_ENABLED=false`，确认 V2 原生生成链路不受影响。
2. 再用本地 sub2api 地址和测试 token 启用账号桥。
3. 验证 `?ticket=...` 可以换取 Alchemy 本地 session。
4. 验证 `/api/v2/veyra/me`、`/api/v2/veyra/history`、`/api/v2/veyra/usage`。
5. 验证余额不足时 job 失败且 provider 未被调用。

## 运行模式

### Local Dev

1. 使用本地数据库 schema。
2. 只启用 GitHub provider。
3. 生图供应商可使用 mock。
4. trace 写本地或开发项目。
5. Veyra 账号桥默认关闭，除非正在验证 sub2api 联动。

### Staging

1. 启用真实同步。
2. 启用至少一个真实图像供应商。
3. 运行评测集。
4. 前端只对内部用户开放。

### Production

1. provider 同步走 scheduler。
2. 生图任务走 worker。
3. 所有安全决策持久化。
4. trace 脱敏。
5. 2.0 可独立回滚。
6. Veyra 账号桥和扣费桥必须先在 staging 完成登录、余额、扣费、余额不足和历史隔离回归。

## 可观测性

必须监控：

1. API 错误率。
2. Agent run 延迟。
3. Tool 调用失败率。
4. Provider 同步失败率。
5. 生图 job 失败率。
6. 安全阻断数量。
7. 用户反馈分布。

## 回滚

2.0 回滚只影响：

1. `/v2` 页面入口。
2. `/api/v2`。
3. v2 workers。
4. v2 scheduler。

不允许影响旧版 `/api/v1`。
