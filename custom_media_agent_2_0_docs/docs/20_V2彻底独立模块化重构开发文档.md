# 20. V2 彻底独立模块化重构开发文档

> 当前状态：本文的 V1/V2 严格隔离原则仍然有效；其中关于 `Claude Code / OpenAI Agents SDK Runtime` 的旧表述按 `00_CURRENT_ARCHITECTURE.md` 修正为 `CreativeManagerRuntime deterministic pipeline + Claude Code Orchestrator + pluggable model sources + SDK-compatible boundary`。

## 1. 背景

V2 当前已经在目录、API 前缀和数据命名上做了初步隔离，但生图执行链路仍存在一个关键架构债：

```text
V2 Creative Run
  -> CreativeManagerRuntime deterministic pipeline
  -> Claude Code Orchestrator with pluggable model sources
  -> V2 PromptPlan / AssetBindingPlan
  -> legacy_image_bridge
  -> V1 /v1/image/jobs
  -> V1 provider / V1 asset planning / V1 history
```

这条桥接链路会把 V2 的语义翻译成 V1 的语义。最近的 Logo 案例已经证明，语义翻译会造成真实用户意图丢失：

1. V2 判断上传图是 `logo_reference`，要求作为 input image 传给生图 provider。
2. 桥接层把 `logo_reference` 映射为 V1 的 `logo_overlay`。
3. V1 把 `logo_overlay` 规划成后处理贴图，而不是 provider reference image。
4. 生图模型没有拿到上传 Logo 作为真实参考，只能按文字自行重绘。
5. 后处理又把 Logo 默认贴到右下角，偏离用户要求的“贴在衣服上”。

结论：继续补桥接层会让 V2 被 V1 语义污染。V2 必须拥有完整独立的后端、生图 provider、素材系统、存储、历史和复检链路。

## 2. 重构目标

V2 必须成为彻底独立的模块化程序。允许共享的只有：

1. 同一个网页前端入口。
2. 同一个域名。
3. 同一套视觉设计语言。

除此之外，V1 与 V2 后台不得有任何运行时依赖。

最终目标：

```text
Browser Frontend Shell
  +-- V1 UI -> /api/v1 -> custom_media_agent_docs/src_skeleton
  +-- V2 UI -> /api/v2 -> custom_media_agent_2_0

V1 backend and V2 backend are separate applications.
V2 never calls V1 APIs, imports V1 modules, writes V1 storage, reads V1 history, or reuses V1 queues.
```

## 3. 硬隔离红线

以下规则是硬约束，后续开发不得绕过。

### 3.1 禁止调用

V2 禁止：

1. 调用任何 `/api/v1/*` 或 `/v1/*` 接口。
2. 使用 `legacy_image_bridge` 作为默认或 fallback 生图路径。
3. 在 V2 生图失败时降级到 V1 生图服务。
4. 从 V1 runtime provider settings 读取模型、key、base URL 或限流配置。
5. 通过 V1 session/job/output API 创建、查询或迁移 V2 任务。

### 3.2 禁止导入

`custom_media_agent_2_0/app/**` 禁止 import：

1. `custom_media_agent_docs.src_skeleton.app.*`
2. V1 providers
3. V1 services
4. V1 repositories
5. V1 storage helpers
6. V1 schemas

如需复用思路，只能重新实现为 V2 原生模块，不允许运行时依赖 V1 代码。

### 3.3 禁止共享存储

V2 禁止读写：

1. `custom_media_agent_docs/src_skeleton/.media_storage/**`
2. V1 history JSONL
3. V1 generated image 目录
4. V1 assets 目录
5. V1 thumbnails 目录
6. V1 SQLite/Redis/队列 namespace

V2 只能读写：

```text
custom_media_agent_2_0/.v2_data/**
custom_media_agent_2_0/.v2_storage/**
```

生产环境对应：

```text
DB schema/table prefix: v2_ or alchemy_v2
Redis namespace: alchemy:v2:*
Object storage prefix: v2/*
Trace project: alchemy-media-agent-v2
```

### 3.4 禁止共享配置

V2 配置必须使用 `V2_` 前缀或 V2 独立 runtime settings。

推荐环境变量：

```text
V2_OPENAI_API_KEY
V2_OPENAI_BASE_URL
V2_OPENAI_IMAGE_MODEL
V2_GEMINI_API_KEY
V2_GEMINI_BASE_URL
V2_GEMINI_IMAGE_MODEL
V2_CLAUDE_ORCHESTRATOR_ENABLED
V2_CLAUDE_ORCHESTRATOR_CLI
V2_IMAGE_GENERATION_PROVIDER
```

V2 不得自动读取 V1 的 `OPENAI_API_KEY`、`GEMINI_API_KEY`、provider settings 或 `.media_storage` 里的配置。若需要把当前页面上的 V1 key 迁移给 V2，必须是一次性显式导入动作，并写入 V2 自己的配置存储。

## 5. V2 原生生图架构

新增 V2 Native Image Engine，替代 `legacy_image_bridge`。

```text
CreativeManagerRuntime
  -> Claude Code Orchestrator with pluggable model sources
  -> optional OpenAI Agents SDK tool/tracing boundary
  -> V2 PromptPlan
  -> V2 AssetBindingPlan
  -> V2 NativeImageGenerationService
  -> V2 ImageProviderRegistry
      +-- V2OpenAIGPTImage2Provider
      +-- V2GeminiImageProvider
      +-- V2MockImageProvider
  -> V2 OutputStorage
  -> V2 ImageHistoryStore
  -> V2 VisualReviewAgent
```

### 5.1 NativeImageGenerationService

职责：

1. 接收 V2 `CreateImageJobRequest`。
2. 读取 `ImagePromptPlan`。
3. 读取 `ProviderInputImage[]`。
4. 选择 V2 原生 provider。
5. 创建 V2 output。
6. 写入 V2 history。
7. 触发 V2 复检。

禁止：

1. 转发到 V1 `/v1/image/jobs`。
2. 创建 V1 session。
3. 写入 V1 generated_images。

### 5.2 V2 ImageProviderRegistry

职责：

1. 只注册 V2 provider。
2. 从 V2 runtime settings 读取配置。
3. 暴露 provider 能力。
4. 执行 provider fallback，但 fallback 只能在 V2 provider 之间发生。

示例：

```python
class V2ImageProviderRegistry:
    def get(provider_hint: str | None) -> V2ImageProvider:
        ...
```

### 5.3 V2OpenAIGPTImage2Provider

必须支持：

1. text-to-image。
2. prompt + input images。
3. 多张 reference image。
4. 本地限流。
5. 429/backoff。
6. 输出缩略图。
7. metadata 保存 requested/actual provider/model。

Logo、商品、人脸、强背景这类硬约束必须走 input image 能力。不能静默降级成纯 prompt。

### 5.4 V2GeminiImageProvider

必须支持：

1. text-to-image。
2. inline image parts/reference image。
3. V2 独立 key/base URL/model。
4. provider 能力声明。
5. 输出统一转存到 V2 storage。

如果当前 Gemini 上游不支持某类 input image，必须返回 V2 可解释错误，不能调用 V1。

## 6. 素材语义保持

V2 的素材角色不得被翻译成 V1 角色。

标准角色：

```text
style_reference
subject_reference
logo_reference
face_reference
background_reference
composition_reference
color_reference
negative_reference
```

V2 provider 接收这些角色时必须保持原义。

示例：

```text
logo_reference
  意义：用户上传的 Logo/品牌视觉资产
  可选处理：
    1. provider input image 融合
    2. V2 deterministic postprocess overlay
    3. 二段式：先生成可贴标主体，再确定性贴 Logo
  禁止处理：
    1. 翻译成 V1 logo_overlay
    2. 默认贴到右下角
    3. 让模型只靠文字重绘 Logo
```

### 6.1 Logo 场景特殊规则

Logo 场景必须区分用户意图：

1. `product_surface_logo`：贴在衣服、瓶身、包装、商品表面。
2. `poster_brand_mark`：海报角标、页脚、页眉、品牌区。
3. `typographic_brand_text`：只要求文字品牌名，不要求上传 Logo 图形完全一致。
4. `exact_logo_overlay`：必须保持上传 Logo 形态，走确定性后处理。

判断优先级：

```text
用户备注 > 用户 prompt > 前端显式 placement > 自动识别
```

例如：

```text
用户备注：“logo照片要在衣服上”
=> role=logo_reference
=> placement_intent=product_surface_logo
=> target_surface=shirt_chest
=> 不允许默认 bottom_right
```

实现要求：

1. V2 `AssetBinding` 必须保存 `fusion_mode`、`placement_intent`、`target_surface` 和 `review_expectations`。
2. V2 `ProviderInputImage` 必须携带同一套融合意图，便于 provider 调用、历史和复检追踪。
3. `provider_input_plan` 必须暴露 `placement_targets` 和 `review_expectations`。
4. Claude Code 必须把这些字段视为硬约束；它可以优化审美表达，但不能把 `logo_product_surface` 改写成 `poster_brand_mark`。
5. 前端历史/提示词计划必须展示融合意图，方便用户和开发者判断系统是否理解上传素材。

### 6.2 二段式 Logo 策略

为了兼顾审美与保真，V2 支持二段式：

1. 第一阶段：provider 生成无 Logo 或预留 Logo 区域的商品/海报。
2. 第二阶段：V2 deterministic overlay 把原始 Logo 贴到指定商品表面或海报位置。
3. 复检确认 Logo 出现在目标区域。

二段式由 V2 中枢或确定性规则触发，不能由 V1 逻辑接管。

## 7. V2 存储与历史

V2 输出必须进入 V2 自己的 storage。

建议结构：

```text
custom_media_agent_2_0/.v2_storage/
  uploads/
    asset_<id>/
      original.png
      brief.json
  outputs/
    job_<id>/
      out_<id>.png
      out_<id>.json
  thumbnails/
    out_<id>.webp

custom_media_agent_2_0/.v2_data/
  image_history.jsonl
  task_queue.sqlite3
  runtime_model_settings.json
```

历史记录必须保存：

1. 用户原始 prompt。
2. Claude final prompt。
3. 最终 provider prompt。
4. selected case。
5. template lock contract。
6. uploaded assets。
7. provider input images。
8. provider input plan。
9. postprocess plan。
10. visual review result。
11. requested/actual provider/model。

V2 history API 只能读取 V2 history。

## 9. 配置与 Provider 独立

V2 runtime model settings 独立保存：

```json
{
  "image_generation_provider": "openai_gpt_image_2",
  "openai_image_model": "gpt-image-2",
  "gemini_image_model": "gemini-2.5-flash-image",
  "claude_orchestrator_enabled": true,
  "claude_orchestrator_model": null
}
```

前端可展示与 V1 类似的模型卡片，但保存时必须写入 V2：

```text
POST /api/v2/runtime/model-settings
```

禁止读取：

```text
GET /v1/runtime/provider-settings
```

## 10. 任务队列与 Worker

V2 worker 只消费 V2 queue。

```text
alchemy-v2-api.service
alchemy-v2-worker.service
alchemy-v2-sync-worker.service
```

禁止：

1. V2 worker 触发 V1 worker。
2. V2 task_queue 写入 V1 queue。
3. V2 任务状态由 V1 job status 决定。

## 11. 复检升级

V2 VisualReviewAgent 必须基于 V2 metadata 和 V2 output 运行。

对于上传素材场景，复检至少检查：

1. `provider_input_required=true` 的素材是否真的进入 provider input images。
2. 如果采用 deterministic postprocess，是否存在 V2 postprocess step。
3. Logo 是否出现在目标区域。
4. 商品主体是否使用上传图身份。
5. 手选案例 template lock 是否保留。
6. 输出中是否出现模型自行编造的品牌标识。

Logo 场景验收示例：

```text
用户要求：Logo 在衣服上
输出：Logo 出现在海报右下角
=> review_status=failed
=> issue=logo_wrong_placement
=> retry_recommendation=use product_surface_logo plan or deterministic shirt_chest overlay
```

## 14. 验收标准

本重构完成后必须满足：

1. 停止 V1 后端，V2 health、上传、案例、生成、历史仍可用。
2. 停止 V2 后端，V1 生图/历史仍可用。
3. V2 生成结果不出现在 V1 history。
4. V1 生成结果不出现在 V2 history。
5. V2 上传素材不出现在 V1 assets。
6. V2 输出不写入 V1 generated_images。
7. V2 代码静态扫描无 V1 import。
8. V2 运行日志无 `/v1/image/jobs`。
9. V2 的 `provider_input_required=true` 不会静默降级为纯 prompt。
10. ALCOEN POLO Logo 回归案例通过。

## 15. 回滚策略

V2 native provider 上线后，回滚只允许在 V2 内部发生：

1. 切换 V2 provider：OpenAI -> Gemini -> Mock。
2. 暂停 V2 image generation。
3. 保留 V2 run 和 queue 供排查。

禁止回滚到 V1 bridge。因为这会重新引入语义污染。

## 16. 非目标

本次重构不做：

1. 重写 V1。
2. 把 V1 历史迁移进 V2。
3. 把 V2 历史迁移进 V1。
4. 共享 V1/V2 provider 配置。
5. 共用 V1/V2 asset planning。
6. 训练 Logo 识别模型。
7. 完整 PSD/图层编辑器。

## 17. 开发优先级

优先级从高到低：

1. 断开 V2 -> V1 生图桥接。
2. V2 OpenAI gpt-image-2 native provider。
3. V2 output storage/history/thumbnail。
4. V2 input images 原生传递。
5. Logo placement/postprocess。
6. Gemini native provider。
7. 视觉复检升级。
8. E2E 隔离测试。

## 18. 一句话原则
