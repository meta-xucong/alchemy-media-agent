# Photography 前端 UX 修复合同（P14）

状态：`implementation_required`

基线：`origin/main@56fbac6`

共享状态、缓存、错误和响应式验收遵循 Doc172。本文件只处理 Photography 的入口、
模式、角色和档案交互，不新增摄影私有 Provider、Review、Retry 或选择器。

## 1. P0 修复范围

### 1.1 模板可用状态必须来自目录

桌面和移动端都必须读取同一个模板目录。禁止移动端把 `photographer_template` 写死
成“稍后开放”，也禁止目录失败时静默显示 General。

```text
目录返回可用 → 摄影师模板显示“可创建项目”
目录返回关闭 → 显示“暂未开放”及原因
目录加载失败 → 显示“暂时无法读取”及重新加载
```

新建项目、最近项目、项目详情、刷新和重开必须保持
`photographer_template`，不能退化为 General。

### 1.2 参考图角色必须由用户意图和模板合同决定

上传界面不能只用宽泛关键词把动物参考误判为 `face_reference` 或 unknown。界面
必须让用户明确选择：

- 人物参考；
- 动物/宠物身份参考；
- 风景/场景参考；
- 静物/物品参考。

动物身份参考缺失或能力不支持时，必须阻断并说明“需要一张可识别的动物参考图”，
不能退化为纯文字生成。

### 1.3 professional_set 必须可见地展示三个角色

选择专业套图后，用户必须看到：

```text
人物主图
环境场景
细节/瞬间
```

每个角色独立显示准备中、生成中、检查中、完成、需要确认、失败或未交付。三角色
不能只显示一张总图，也不能在一个角色失败时显示整组成功。

## 2. 单图与专业套图

用用户语言解释区别：

```text
单张作品：生成一张最终照片
专业套图：生成三张相互关联、用途不同的照片
```

提交后显示实际角色进度和预计下一步；不显示 `session_hero` 等内部 lineage 名称，
除非在高级诊断面板中。

角色续作必须使用已存在的角色绑定和共享 continuation 合同；当前接口未准备好时，
不显示可点击的“重做这张”。

## 3. 四类场景与具名档案

入口必须让用户明确选择：

- 人像摄影；
- 风景摄影；
- 静物摄影；
- 动物摄影。

具名摄影师档案必须显示名称、版本和“确认使用此档案”动作。未确认、版本不一致或
目录异常时阻断，不能伪装成通用摄影或自动回退。

## 4. 失败与状态

摄影页面必须把以下结果用人话呈现：

- 需要远程创意规划：正在整理拍摄方向；
- 缺少动物身份参考：请上传参考图；
- 角色尚未完成：暂时不会显示整组完成；
- 图片已生成但需要人工确认：请查看后决定；
- 没有生成图片：本次未完成，可返回修改；
- 服务暂时不可用：项目已保存，稍后刷新；
- 已完成：显示每个角色的最终图片。

不得把 Brain、Provider、Vision、metadata-only、job ID 或内部 failure code 直接给普通用户。

## 5. 必须补的回归

- 桌面/移动端模板目录状态一致；
- 直达、刷新、最近项目、详情、重开均保持摄影模板；
- 四个场景入口可理解，General/E-Commerce 不出现摄影字段；
- single_hero=1 张，professional_set=3 个角色，数量和状态一致；
- 三角色中任一失败都显示诊断状态，不伪装成单张成功；
- 角色级进度和最终交付只显示当前 winner；
- 参考图类型、缺图、高保真动物身份阻断正确；
- 具名档案需要二次确认且不可静默换档案；
- 首轮失败、Provider/Vision 阻断均能返回项目、刷新、重试或修改输入；
- 1440/1024/390/430 浏览器路径通过；
- 页面不显示 asset/candidate ID，不依赖 hover。

## 6. 完成标准

摄影前端只有在普通用户能清楚选择场景、单图/套图和参考方式，并且能看到每个角色
的真实终态、失败原因和下一步时，才可标记 `photography_frontend_ux_passed`。
这不等于 P10、Gate C/D 或生产门禁通过。
