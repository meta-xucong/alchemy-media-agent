# Doc140 - V3 Remote Brain Semantic Task Profile Completeness

Status: **active shared-foundation implementation authority.**

## 1. Problem corrected

An enforced real-image request previously asked the remote Brain only for a
small rendering-medium decision. The adapter then merged that partial response
with a deterministic fallback task profile. This could silently omit a visible
real person from the frozen capability plan even though the remote Brain was
otherwise available. An adult-person request is sufficient to demonstrate the
fault; it is not a children, apparel, E-Commerce, Photography, or prompt
quality branch.

## 2. Authority and boundary

```text
frozen user/reference facts
-> remote Brain complete semantic task profile
-> shared activation planner
-> frozen execution envelope / constraint ledger
-> remote Brain canonical prompt finalization
-> optional Human Realism independent Brain re-signing
-> Provider or Local MCP exact relay
```

The remote Brain owns the semantic judgment of visible subjects, rendering
intent, evidence, uncertainty, confidence, and the capability activation
intent that follows from them. Local code only validates that the existing
contract shape is complete and binds non-creative job identifiers. It must
never infer a person, age, garment, culture, style, composition, capability
request or renderer wording from text.

## 3. Real-image contract

For a new enforced real-image job, `visual_task_profile` must include:

- a remote-owned rendering intent;
- an explicit `subject_entities` list, including an empty list when no target
  subject is visible;
- concise semantic tags, explicit unknowns and calibrated profile confidence;
- concise evidence entries with their source and confidence.
- a complete `capability_activation_intent`, using only the supplied shared
  catalog and allowing explicit empty requested/rejected lists.

For a visible real person, the Brain records the existing generic
`visible_person` and/or `real_human_output` evidence purpose. This is typed
capability evidence, not a prompt fragment, content recipe, keyword detector
or demographic rule.

If either response is absent, malformed or partial, the job is blocked before
template planning, Provider/MCP materialization, review or delivery. The
deterministic compatibility profile and capability proposal remain readable
for legacy/non-real draft planning only; they cannot fill semantic gaps or
activate a new real-image run.

For a Local MCP planning block, the relay may project only the existing V3
safe remote-outcome receipt: state, reason class, remote availability, safe
HTTP/error classification and cardinality facts. It never returns a raw
provider exception, endpoint, credential, prompt, reference content or a
second lifecycle record.

## 4. Universal consequences

- A real visible person activates shared `human_realism` through the normal
  evidence-driven planner, then receives the existing Brain finalizer and
  independent Brain re-signing steps.
- A full product-only/no-person profile remains valid and does not invoke the
  human-only re-signer.
- General, E-Commerce and Photography all use the same contract. No template
  obtains a local semantic fallback, special prompt phrase, slot or role.
- The final renderer-facing natural-language prompt remains exclusively Brain
  authored. The contract profile is never appended to that prompt.

## 5. Verification

1. Compact real-image payloads require complete semantic profile fields.
2. A rendering-intent-only or local-intent-only remote answer blocks before
   final prompt signing.
3. A remote-described adult visible person activates Human Realism without
   local text classification and gets the two Brain signing stages; a
   remote-described no-person decision overrides any older fallback proposal.
4. A remote-described product-only task stays human-inactive.
5. Existing E-Commerce and Photography fixtures return a full contract;
   cross-template and Local MCP parity regressions pass.

## 6. Non-goals

This document does not create a child or kidswear capability, modify a
specialized deliverable map, add an OCR/review recipe, make Local MCP pixel
certified, or replace shared visual review. It closes only the semantic
ownership gap before capability activation.
