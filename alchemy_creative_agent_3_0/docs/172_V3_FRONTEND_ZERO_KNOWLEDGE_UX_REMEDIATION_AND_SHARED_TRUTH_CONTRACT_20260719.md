# Doc172 — V3 前端零基础用户体验与共享状态真相修复合同

状态：`implementation_required`

基线：`origin/main@56fbac6`

本文件是 V3 主线前端收口合同。它把 General、E-Commerce、Photography 和
Professional Mode 的前端修复统一到同一个用户体验与状态真相原则下，但不把任何
专业交付逻辑搬进 General，也不改变 Brain、Provider、Review、Retry 或最终交付的
所有权。

## 1. 目标

没有计算机或设计基础的用户应能完成：

```text
进入 V3
→ 看懂不同模板的用途和当前是否可用
→ 选择一个明确模板
→ 创建项目
→ 上传或跳过参考图
→ 提交一次需求
→ 看懂准备中/需要确认/完成/失败
→ 看到最终图片或明确下一步
→ 刷新、返回、重新打开后仍看到同一项目真相
```

界面可以隐藏复杂实现，但不能隐藏影响用户决策的事实。特别是：

- 模板不可用时必须明确说明原因，不能静默改成 General；
- 图片未生成时不能显示“已完成”或空白成功态；
- 旧记录可以读取，但旧 slot、recipe、overlay 不能重新变成新任务的界面语义；
- 内部 ID、hash、Provider、Brain、Job、route、manifest 不出现在普通用户路径。

## 2. 审计结论

当前前端设计方向基本正确：项目优先、显式模板、自然语言输入、共享工作台、结果
优先和项目续作都符合既有 Doc27/31/35 的目标。但实现仍有以下共享缺口：

### P0：会导致用户走错路径或无法恢复

1. 模板目录失败时，前端使用硬编码模板卡，并可能把不可确认的状态显示成可创建。
2. `photographer_template` 在部分映射中会被当作 General；模板刷新/重开真相不稳定。
3. 项目 API 失败时直接回退本地项目缓存，旧缓存可能覆盖当前模板、状态或图片事实。
4. `renderV3ProjectNextActions()` 当前存在提前返回，阻断、失败、完成后的恢复动作不显示。
5. 普通错误可能直接展示 HTTP/HTML/内部错误文本。
6. 结果面仍可从历史 `image_recipes`、slot 等字段拼出普通用户可见内容。

### P1：能完成任务但会造成误解或不必要的失败

1. 处理中、需要人工确认、未认证、被阻断、失败可重试之间缺少统一的人话状态。
2. 本地缓存没有区分“服务端权威数据”和“最后一次已知数据”。
3. 结果图片加载失败时缺少统一的“重新加载/打开项目/稍后刷新”恢复路径。
4. 专业模块、摄影模块和电商模块各自对模板、角色、数量和交付状态的显示不一致。
5. 关键操作的键盘焦点、非颜色状态提示、移动端按钮可达性和弹窗返回路径未完成实测。

## 3. 不可变的共享前端原则

### 3.1 模板真相

模板目录是可创建状态的唯一来源。前端可以提供加载骨架，但不得用硬编码卡片
代替失败的模板目录。

```text
目录成功
→ 渲染真实 template_id / scenario_id / can_create / reason

目录加载中
→ 显示“正在读取可用项目类型”，禁止创建

目录失败
→ 显示“项目类型暂时无法读取” + “重新加载”
→ 禁止创建、禁止回退 General
```

`template_id`、`scenario_id` 和 `project_can_create_jobs` 必须作为一个不可拆分的
绑定保留。不能用“场景字符串”二次猜测模板。

### 3.2 项目缓存

localStorage 只能作为短暂的离线展示缓存，不能作为运行真相，也不能与服务端
对象做无条件字段合并。

- 服务端成功返回：以服务端完整字段为准，替换同 ID 缓存；
- 服务端失败：可以展示“上次读取的项目”，但必须显式标记为只读/可能过期；
- 服务端失败时禁止创建、继续生成、归档、删除、选择结果等写操作；
- 缓存缺少 `primary_template_id`、`status`、最终交付信息时，不能自行补成 General
  或成功；
- 刷新/重开必须重新读取项目、项目输出、时间线和当前任务状态。

### 3.3 用户可见状态

所有模块使用同一组状态语义，模块只提供模块名称和下一步动作：

| 内部状态 | 用户显示 | 必须提供的动作 |
|---|---|---|
| loading | 正在读取项目 | 等待/返回 |
| planning | 正在整理你的需求 | 等待、取消或返回项目 |
| generating | 图片正在生成 | 等待、刷新 |
| reviewing | 正在检查图片 | 等待、刷新 |
| manual_confirmation_required | 图片已生成，等待你确认 | 查看结果、确认/返回 |
| ready/completed | 已完成 | 查看、下载、继续 |
| held/blocked | 暂时不能交付 | 人话原因、下一步 |
| failed_retryable | 这次没有完成 | 有界重试/返回项目 |
| failed_non_retryable | 这次无法完成 | 修改输入/重新开始 |

“已生成像素”不等于“已认证交付”。普通界面必须区分图片已返回、需要确认、已
认证交付和未生成图片。

### 3.4 错误投射

前端适配器必须把后端错误映射成以下可理解类别：

- 缺少参考图：请上传一张符合要求的参考图；
- 参考图尚未准备好：正在保存或检查，请稍后刷新；
- 参考图不支持：请换一张清晰图片；
- 模板暂未开放：返回项目类型选择；
- 需要人工确认：查看结果后决定是否继续；
- 服务暂时不可用：项目已保存，稍后刷新重试；
- 本次未生成图片：不会伪装成成功，可返回修改需求；
- 数量/能力不匹配：明确显示实际可选数量或让用户选择。

原始 HTML、HTTP 状态页、Provider 错误、base URL、Job ID、hash 和完整诊断只进入
受控日志或高级诊断区，不能进入普通 notice、toast 或项目卡。

### 3.5 结果真相

普通结果区只读取当前最终交付投影：

```text
current final delivery
→ 预览、下载、继续使用

held / manual confirmation
→ 显示状态，不当作最终交付

retry-superseded / process-only / historical recipe/slot/overlay
→ 折叠到历史或诊断，不进入普通结果标题、数量和角色说明
```

兼容读取旧字段可以保留，但不能由旧字段生成新的用户可见专业语义。

## 4. 主线共享实施项

主线负责以下共享修复，模块分支不得各自复制：

1. 模板目录加载、加载失败和模板绑定适配器；
2. Project Summary、Project Detail、Recent Project 的统一字段投射；
3. 服务端权威缓存与只读过期缓存的区分；
4. 统一状态和错误人话映射；
5. 当前最终交付投影和历史兼容字段隔离；
6. 图片加载失败、刷新、重开和重复提交的通用行为；
7. 键盘焦点、可访问名称、非颜色状态和移动端基础布局。

主线不得因此新增第二套 Provider、Brain、Review、Retry 或图片存储。

## 5. 模块分支实施边界

- Professional：只增加 Professional Mode / People Asset / Anchor Pack 的用户入口、
  生命周期呈现和确认动作；复用主线共享状态和既有 people-assets API。
- Photography：只增加摄影场景、单图/专业套图、角色状态和档案确认；复用共享生成、
  审查、重试、结果投射和模板目录。
- E-Commerce：只增加电商事实输入、平台证据、数量选择和 LLM 方向结果呈现；不得
  恢复固定套图、slot、recipe、默认文案或本地规则循环。

## 6. 统一浏览器验收矩阵

每个模块都必须在当前构建上验证：

### 桌面

- 1440px：首次进入、创建、结果、错误和恢复；
- 1024px：长文案、按钮、卡片和侧栏不重叠。

### 移动端

- 390px、430px：不依赖 hover；主要按钮可见；上传、返回、刷新、预览和确认可达；
- 弹层打开后焦点进入弹层，关闭后回到触发按钮；
- 状态不只依赖颜色，有文本或图标辅助。

### 共享新手路径

```text
新会话
→ 选择模板
→ 创建项目
→ 上传/跳过参考图
→ 提交一次
→ 等待终态
→ 查看结果或失败下一步
→ 刷新
→ 最近项目重新打开
→ 继续生成
```

每一步都必须能回答：现在发生了什么、用户是否需要操作、下一步点哪里。

## 7. 完成标准

本合同只有在以下条件全部满足后才可标记 `frontend_ux_closure_passed`：

1. 静态测试不再接受旧固定套图同义词、旧 slot/recipe 普通投射和隐式 General 回退；
2. 模板目录失败不会显示可创建模板；
3. 刷新/重开不改变模板、场景、数量、角色或交付状态；
4. 所有阻断和失败都有下一步，且不会泄漏内部错误；
5. General、E-Commerce、Photography、Professional 互不污染；
6. 四个尺寸的浏览器路径均通过；
7. 各模块专项审计回传，无未解决 P0；
8. 共享代码、模块代码和文档均完成独立测试、审计和提交。
