# Doc124 V3 Safety-Sensitive Person Provider Materialization and Policy Closure

Status: corrective shared-foundation authority following the controlled General
reference diagnosis. It applies to all V3 templates through Human Realism,
Provider materialization, Gateway provenance, and final-delivery state. It
does not add a children, apparel, E-Commerce, Photography, or General
Template-specific prompt, route, role, deliverable map, or policy bypass.

## 0. Observed production fact

The controlled reference request reached the Gateway successfully. Gateway
local review allowed it, the first route failed transiently, and Gateway
failover then received an explicit upstream `content_policy_violation` before
any image pixel was returned. This establishes the following distinction:

1. V3 did create the image-edit request and did not fail local reference
   parsing.
2. Gateway admission is not an upstream generation-policy approval.
3. A no-pixel result with an explicit policy signal is a Provider-policy
   terminal state, not a visual-quality result and not a reason to rewrite the
   request repeatedly.

## 1. Corrected shared invariant

1. Human Realism may recognise an explicitly safety-sensitive young-person
   direction only as an auxiliary shared signal. It exposes
   `provider_safety_profile: safety_sensitive_person_v1`; it must not create a
   template branch or infer age from visual appearance.
2. That profile changes only framework-owned Provider wording. The lossless
   user direction, remote Brain creative decision, reference-channel policy,
   and frozen garment facts retain their existing owners.
3. Its Provider contract is concise and family-friendly: age-appropriate,
   fully dressed, ordinary everyday activity, and ordinary full-frame or
   medium-distance framing. It avoids framework-added anatomy, beauty,
   micro-detail, or internal review-code wording.
4. A product/garment reference remains product/garment truth. It must not be
   upgraded to same-person identity evidence merely because Human Realism is
   active.
5. A visual retry may retain one bounded, shared retry. For this profile it
   uses only a generic image-level repair direction: photographic naturalness,
   clothing/material clarity, scene coherence, reference-channel fidelity,
   and the same safe context. It never serializes `bad_hands_or_body`,
   `ai_face_render`, raw vision messages, or anatomy/beauty repair text.

## 2. Provider policy closure

1. Gateway exclusively owns line switching, transport retries, and any
   allowed independent-route attempt. V3 must not wrap a Gateway policy
   response in another SDK/HTTP retry.
2. An explicit `content_policy_violation` is recorded internally as
   `provider_policy_blocked`, with safe operation, pixel, and lifecycle
   provenance. It is not collapsed into timeout, generic quality failure, or
   reference rejection.
3. User-visible status may state that the Provider blocked the request before
   pixels. It must not expose account identifiers, route details, raw prompts,
   raw errors, credentials, or hidden candidate metadata.
4. Policy-blocked/no-pixel operations create no post-generation review,
   visual retry candidate, final delivery, or manual-confirmation path.
5. If a first attempt did create pixels but later shared review rejects them,
   those pixels remain append-only history and the public final-delivery gate
   stays withheld. A later policy-blocked retry never makes the rejected first
   candidate deliverable.

## 3. Prompt ownership and non-bypass boundary

- This is ambiguity reduction, not policy evasion. V3 never substitutes an
  adult subject, changes the requested age, removes required factual garment
  truth, obfuscates terms, or repeatedly rotates routes to defeat an upstream
  decision.
- V3 does not rewrite explicit user content. The profile suppresses only its
  own reusable Human Realism and review-repair fragments where they would add
  unnecessary anatomy, appearance, or internal-quality detail.
- A same-person portrait reference keeps Doc93/Doc95 authority only when the
  resolved reference policy has assigned portrait identity channels. This
  document does not weaken that explicit user contract.
- Adult and ordinary-person Human Realism remain unchanged; the safety profile
  must not reduce their existing camera, skin, face, or bounded-repair quality
  guidance.

## 4. Required regressions

1. Explicit school-age apparel-on-person direction activates the shared
   safety profile and retains Human Realism activation and garment truth.
2. A real General/remote-Brain materialization with that profile contains the
   concise safety contract, but contains no internal issue codes or
   framework-added skin/hand/face micro-detail repair text. Its generic retry
   retains the safe context and does not expose raw retry patches.
3. An ordinary adult portrait continues to receive the normal Human Realism
   guidance, proving no global quality regression.
4. A nested Gateway response that explicitly contains
   `content_policy_violation` becomes `provider_policy_blocked`, performs no
   V3 outer retry, and projects no pixels as available.
5. Provider admission, source MIME/decode validation, image-generation versus
   image-edit selection, append-only review history, and final-delivery
   withholding retain their existing contracts.

## 5. Acceptance boundary

This change can reduce unnecessary ambiguity in framework-created prompts and
make a Provider-policy outcome honest. It cannot guarantee that an upstream
Provider will allow a particular request, and a policy-blocked/no-pixel run is
neither a visual failure nor a visual-quality pass. A later controlled pixel
run must still clear the existing shared certified review and garment-truth
checks before any real child/apparel Gate is considered passed.
