# 89 V3 Portrait Photography Stability Test Protocol

## 1. 文档目的

本文档把 V3 通用模板中的“人像摄影稳定性测试流程”固化为标准验收协议。

它不是新的功能模块，不改变 V3 中枢框架，也不替代 Doc85-Doc88 的人像一致性规则。它的作用是：

```text
用一套固定、可复测、可对比的测试流程，判断 V3 人像图生图能力是否稳定。
```

本协议重点回答四个问题：

```text
1. 上传参考脸之后，生成图是否还能看出是同一个人？
2. 场景、妆容、发型、服装、光影、风格变化时，是否只改变造型表现，不破坏骨相？
3. 多轮继续生成后，人物身份是否会逐渐漂移？
4. 当前质量距离 Lovart 级商业人像成片还有多远？
```

## 2. 适用范围

适用于所有 V3 中涉及人像、人物写真、模特、角色、真人摄影风格的生成路径，尤其是：

```text
通用模板中的人像图生图
用户上传人物参考图后的继续生成
用户选中已有生成图作为参考后的继续生成
未来摄影师模板中的人物摄影能力
未来电商模板中的真人模特图
未来品牌 IP / 私域 / 新媒体模板中的人物视觉资产
```

不适用于：

```text
纯产品图
无人物场景图
插画 / 动漫 / 非真人角色图
用户明确要求换脸、重塑人物、改变身份的任务
```

## 3. 与历史文档的关系

本文档依赖并继承：

```text
Doc76  Foundation vs specialized-template governance
Doc78  Long-term identity and beautiful realism
Doc85  Image-to-image identity transfer and reference truth closure
Doc86  Portrait bone-structure identity lock
Doc87  Portrait reference identity and style separation
Doc88  Portrait reference balance and prompt mood preservation
```

Doc89 是验收与测试协议，不是实现标准的替代品。

如测试结论发现 Doc85-Doc88 的实现没有达标，应回到对应功能文档补实现，而不是在 Doc89 中新增业务规则。

Doc89 的核心判断原则与 Doc88 保持一致：

```text
保留人物身份，保留用户想要的画面氛围，不为了锁脸而破坏 prompt 的场景、光影和审美。
```

## 4. 核心验收标准

V3 人像摄影能力的稳定性，不应只看某一张图是否好看，而要看同一张参考脸在多场景、多风格、多机位下是否稳定。

合格结果必须同时满足：

```text
人物骨相可识别为同一人
妆容、发型、服装、光影、场景可以按 prompt 合理变化
不因追求真人感而变丑
不因追求美感而变成硅胶脸、网红脸、AI 精修脸
连续多轮生成后身份不明显漂移
一组图中至少有商业可用结果
```

禁止把以下情况误判为合格：

```text
只是同一种美女类型，但不是同一个人
只保留发型或发色，脸型和五官关系已明显变化
只复制参考图光影，却没有执行当前 prompt 的场景
脸很像，但画面氛围、服装、光线完全跑偏
图很好看，但身份继承失败
图很真实，但五官比例、脸型美感被破坏
```

## 5. 标准测试矩阵

每次进行人像稳定性验收时，必须固定同一张参考人物图，然后覆盖以下维度。

### 5.1 场景变化测试

至少测试 6 类场景：

```text
1. 海边 / 夏日自然光
2. 城市夜景 / 街拍
3. 室内咖啡馆 / 生活方式
4. 古典 / 国风 / 传统服饰氛围
5. 时尚棚拍 / 杂志封面
6. 公园 / 花园 / 自然柔焦
```

观察目标：

```text
场景可以明显变化，但人物骨相、脸部比例、整体气质不能换人。
```

### 5.2 风格变化测试

至少测试 5 类风格：

```text
1. 真实纪实摄影
2. 商业写真摄影
3. 柔焦胶片摄影
4. 高级杂志大片
5. 自然生活方式摄影
```

观察目标：

```text
风格可以变化，但不能把同一个人变成不同审美模板下的不同模特。
```

### 5.3 机位与动作变化测试

至少测试 6 类机位 / 动作：

```text
1. 正脸近景
2. 侧前方半身
3. 低机位
4. 全身站姿
5. 回头抓拍
6. 坐姿或走动抓拍
```

观察目标：

```text
姿势、表情、角度可以自然变化，但眼距、鼻口关系、脸型轮廓、下颌方向不能大幅漂移。
```

### 5.4 复杂造型测试

至少测试 5 类造型：

```text
1. 简单白裙 / 日常服装
2. 复杂礼服
3. 传统服饰 / 层叠服装
4. 职业装
5. 运动或户外造型
```

观察目标：

```text
服装复杂度不应反向污染人物脸部。
同一批次内，如果 prompt 要求同一套造型，服装结构、颜色、主要纹理也应保持一致。
```

### 5.5 连续项目漂移测试

必须测试至少 3 轮连续生成：

```text
第 1 轮：上传参考人物图，生成 4 张
第 2 轮：从第 1 轮中选中 1 张满意图作为参考，继续生成 4 张
第 3 轮：从第 2 轮中选中 1 张满意图作为参考，再继续生成 4 张
```

观察目标：

```text
人物身份不能一轮比一轮更像新模特。
如果选中图本身已偏离参考人物，系统应提示或降低其作为强身份参考的权重。
```

## 6. 单张评分标准

每张生成图按 5 个维度评分，每项 0-100 分。

### 6.1 人物骨相继承

检查：

```text
脸型长宽比例
额头、太阳穴、颧骨、脸颊、下颌线方向
眼距和眼型关系
鼻梁、鼻翼、嘴唇、下巴之间的比例
整体气质是否像同一个人
```

评分参考：

```text
90-100  明显是同一个人，妆造变化不影响身份判断
80-89   大体是同一个人，有轻微美化或风格化漂移
70-79   像同一类人，但身份继承不够稳
60-69   只能靠发型、气质或服装猜测有关联
0-59    明显换人
```

### 6.2 场景与造型适配

检查：

```text
妆容是否符合 prompt
发型是否按 prompt 合理变化
服装是否符合 prompt
光影、色调、环境是否跟随当前任务
是否错误继承参考图原本的光线、背景、色温
```

评分参考：

```text
90-100  完整执行 prompt，参考图只提供身份
80-89   大体执行 prompt，少量元素受参考图干扰
70-79   场景成立，但部分关键设定缺失
60-69   画面好看但偏离用户需求
0-59    与 prompt 明显不符
```

### 6.3 商业美感

检查：

```text
脸是否好看
五官比例是否协调
构图是否像商业摄影
皮肤、光影、服装、背景是否统一
是否有明显廉价感、网红模板感或过度修图感
```

评分参考：

```text
90-100  可作为商业成片直接使用
80-89   商业可用，但仍有轻微 AI 或审美瑕疵
70-79   可作为候选图，需要人工筛选
60-69   普通好看，但商业质感不足
0-59    不建议使用
```

### 6.4 真人感与去 AI 感

检查：

```text
皮肤是否真实但仍美观
是否有硅胶脸、塑料脸、过磨皮、水光油亮脸
头身比例、手部、牙齿、眼神是否自然
脸部是否像同一套 AI 模板
光影是否符合真实摄影逻辑
```

评分参考：

```text
90-100  接近真实摄影，细看也不明显 AI
80-89   大体真实，局部仍有 AI 精修感
70-79   初看可用，细看能发现 AI 痕迹
60-69   AI 感明显但未崩坏
0-59    明显假、崩、塑料或比例错误
```

### 6.5 同组稳定性

检查：

```text
同一组多张图是否像同一个人
是否只有动作、表情、角度变化
是否出现一张明显换脸
是否出现一张明显更丑或比例异常
是否出现提示词外的奇怪物体、文字、水印
```

评分参考：

```text
90-100  一组图高度稳定，像同一次商业拍摄
80-89   大体稳定，有 1 张轻微漂移
70-79   可用图存在，但组内差异较大
60-69   多张图之间不像同一个人
0-59    组内身份失控
```

## 7. 组级通过标准

每组建议生成 4 张图。验收时按组判断。

### 7.1 最低通过线

一组图通过必须满足：

```text
至少 2 张商业可用
不得出现明显换人的图片
不得出现明显崩脸、崩手、比例失真、水印、乱码文字
人物骨相继承平均分 >= 80
商业美感平均分 >= 80
真人感平均分 >= 78
同组稳定性 >= 80
```

### 7.2 优秀通过线

一组图优秀必须满足：

```text
至少 3 张商业可用
人物骨相继承平均分 >= 85
商业美感平均分 >= 85
真人感平均分 >= 82
同组稳定性 >= 85
不同场景下仍能一眼看出是同一人
```

### 7.3 Lovart 对标线

接近 Lovart 级别应满足：

```text
单张美感接近专业商业成片
人物身份长期稳定，不只是一组内稳定
图生图参考能稳定继承脸部骨相
当前 prompt 的场景、光影、服装、妆造执行准确
多轮继续生成后不明显身份漂移
失败图能被识别、降权或重试
```

当前 V3 通用模板达到 Lovart 的百分比，应按以下方式估算：

```text
基础单张审美：单张商业美感 / Lovart 预期
身份继承：人物骨相继承 / Lovart 预期
长期稳定：连续项目漂移测试结果 / Lovart 预期
真实摄影感：真人感与去 AI 感 / Lovart 预期
综合分：以上四项加权平均
```

建议权重：

```text
人物骨相继承 35%
商业美感     25%
真人感       20%
长期稳定     20%
```

## 8. 推荐测试流程

### 8.1 准备阶段

测试前固定：

```text
1 张清晰参考人物图
1 个测试项目
同一个模型与 provider 配置
同一批测试 prompt
每组生成数量建议为 4 张
保留所有原图、预览图、最终 prompt、重试记录
```

参考图要求：

```text
脸部清晰
无遮挡
不过度美颜
不使用严重滤镜
至少能看清脸型、眼鼻口比例、发型方向
```

### 8.2 首轮测试

流程：

```text
上传参考图
输入第一个场景 prompt
生成 4 张
记录每张图评分
选择最满意且最像本人的一张作为正向参考
```

注意：

```text
不得只选最好看的图。
必须优先选择“好看且像本人”的图。
```

### 8.3 多场景测试

流程：

```text
依次使用场景变化测试中的 6 类场景
每个场景生成 4 张
每组记录通过 / 不通过
标记失败原因
```

失败原因建议使用：

```text
identity_drift        人物不像参考图
beauty_loss           为了真实导致变丑
ai_face               AI 精修感明显
prompt_mood_loss      prompt 氛围丢失
reference_overcopy    过度复制参考图光影/色调/构图
wardrobe_drift        同组服装结构漂移
composition_failure   构图或机位失败
artifact_or_watermark 水印、乱码、异常物体
```

### 8.4 连续漂移测试

流程：

```text
第 1 轮选择最优图
第 2 轮以选中图作为参考继续生成
第 3 轮再以新选中图继续生成
将第 3 轮结果与原始上传参考图对比
```

判断：

```text
如果第 3 轮仍能看出是原始参考人物，说明长期一致性基本可用。
如果第 3 轮只像第 2 轮生成图，但不像原始参考人物，说明项目参考链路发生身份漂移。
```

## 10. 稳定性结论分级

测试完成后，用以下等级描述当前能力。

```text
A  稳定商用
   多场景、多风格、多轮继续生成均稳定，接近 Lovart 级别。

B  可商用但需筛片
   大多数场景可用，但仍需要用户或系统筛掉少量身份漂移 / AI 感图片。

C  局部可用
   简单场景表现好，复杂造型、强风格迁移、连续生成仍不稳定。

D  不稳定
   人物身份经常漂移，上传参考图不能可靠继承。
```

V3 通用模板进入下一阶段前，至少应达到：

```text
B  可商用但需筛片
```

摄影师模板或电商真人模特模板进入正式商业化验收前，应达到：

```text
A  稳定商用
```

## 11. 测试员注意事项

测试时必须避免以下误区：

```text
不要只看图好不好看，要先判断是不是同一个人。
不要把妆容变化误判为身份漂移。
不要把骨相变化误判为正常妆造变化。
不要只测最容易成功的美女写真。
不要只测单张，要测一组和连续多轮。
不要只看前端缩略图，必须打开大图检查脸、手、皮肤和光影。
不要把上游偶发失败计入画质失败，但要记录为链路稳定性问题。
```

合理的变化包括：

```text
妆容浓淡变化
发型在 prompt 要求下变化
服装变化
光影、色温、背景变化
表情、动作、机位变化
符合场景的肤色和质感变化
```

不合理的变化包括：

```text
脸型明显变宽或变窄
眼距明显变化
鼻子、嘴唇、下巴比例改变
从清冷气质变成完全不同的甜美 / 网红 / 欧美化脸
为了真人感导致五官变钝、皮肤发黑、比例变丑
为了美感导致硅胶脸、过度磨皮、AI 水光肌
```

## 12. 后续开发约束

如果后续开发要提升人像质量，应优先在既有 V3 视觉能力模块内优化：

```text
portrait_identity
visual review
retry patch builder
prompt recipe / negative prompt recipe
project selected-reference context
post-generation scoring
```

不得为了通过本测试而新增分散在中枢框架之外的临时规则。

不得把某个单独场景的修复词硬编码成所有人像任务的通用规则。

不得为了提高身份分而牺牲 prompt 的场景、色调、光影和审美方向。

最终目标不是“复制参考照片”，而是：

```text
同一个人，在不同摄影场景、不同妆造、不同光影下，仍然自然、好看、真实、可商用。
```
