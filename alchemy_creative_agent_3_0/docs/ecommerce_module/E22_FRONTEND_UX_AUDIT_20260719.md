# E22：E-Commerce 前端 UX 专项审计

日期：2026-07-19
审计基线：`origin/main@56fbac6b7a0ac1b1cad66fa293a469cd4347492c`
范围：V3 E-Commerce 模板的普通用户路径与前端投射；不修改 Provider、
Central Brain、共享 review/retry 或生产门禁。

## 结论

运行时的 LLM-first 合同已经存在：用户明确选择模板后，商品事实、用户
约束和平台证据进入远端 Brain；数量或 Brain 合同不成立时会 fail-closed。
但当前前端仍把电商说成一套预先拆好的固定图片包，且存在数量、下一步恢复
和一项输入语义映射的 P0 缺陷。它会让没有技术背景的用户误解系统能力，
也可能把用户的“店铺感觉”错误送成“默认卖点”。

以下 P0 应在全版面收口中优先解决；不得用更多规则、preset 或 role 来补救。

## 已确认的正向合同

1. 首页模板选择是显式的。`v3ProjectTemplateId` 只读项目的
   `primary_template_id` / 历史公开标签，不从用户文字或商品关键词推断模板。
   `test_general_product_keywords_do_not_activate_ecommerce_or_send_its_context`
   在该基线通过。
2. 电商前向路径的数量或远端 Brain 计划错误会阻断，不会生成本地静态套图；
   `v3EcommerceFailureMessage` 已有人话说明。
3. 长请求有后台恢复轮询，项目结果会刷新；生成按钮在 `v3State.loading`
   时禁用，重复点击已有基本保护。
4. CSS 在 `760px` 以下把主要网格收为单列，具备响应式基础；但尚未完成
   真机/窄窗口行为验证。

## P0：必须先修复

### P0-1 固定套图承诺仍在用户主路径

**复现**：首页选择“电商模板”并进入项目；查看模板说明、工作区提示与
项目“产物与记录”。

**实际表现 / 证据**：

- `src_skeleton/app/static/app.js:1861-1868` 将其定义为“电商套图项目”，
  承诺“商品主图、卖点图、详情页配图和上架套图”。
- `app.js:1971-1974`、`2011-2035`、`4178`、`4404-4426` 反复把本次任务
  称为“生成套图”。
- 最严重的是 `app.js:3636-3661`：项目流程硬编码“规划套图”，并写明
  “自动拆成主图、卖点图、场景图、细节图和信任图”。

**用户风险**：用户会以为数量、角色和镜头由系统固定决定；当 Brain 输出
不同的整图方向、数量为 1 或被阻断时，会误判为失败或功能不完整。

**建议修复**：将所有文案改为“本次商品图片任务 / 本次 N 张图片”。流程只
保留：补充商品事实 → 选择本次数量与平台 → 生成整图方向 → 查看可交付图。
在计划区显示“第 1 张 / 第 2 张：本次方向摘要”，而不是预定义角色。不得
新增主图、卖点、场景、信任等 UI 选项。

**涉及**：电商前端文案与 `renderV3ProjectWorkflow`；不需共享接口。

### P0-2 用户无法请求 N=7，前端与冻结数量合同不一致

**复现**：进入电商工作区，查看“生成数量”滑杆；尝试选择 7 张。

**实际表现 / 证据**：`index.html:1538-1543` 的范围是 `min=1 max=4`，
`app.js:5124-5128` 的 `v3BoundedGenerationCount` 也强制 `1..4`。但
E-Commerce 的当前合同和 Doc132 fixture 已覆盖 `N=1/2/4/7`。

**用户风险**：UI 宣称数量可选，实际却无法表达一项已支持的请求；用户也
不知道当前平台/线路不支持某个数量时应在提交前如何调整。

**建议修复**：替换连续滑杆为明确的 1、2、4、7 张单选卡（显示“本次生成
N 张完整图片”）；提交前以当前运行时返回的能力信息标记不可用项，并解释
“不可用时不会自动少生成”。不要把平台上限硬编码到浏览器。

**涉及**：前端为主。若现有模板能力接口不能返回“当前平台 + 当前通道可接受
的数量集合”，需要主线提供只读能力投射；不得建立新的静态平台配方表。

### P0-3 “店铺希望的感觉”被错误写入 core_selling_points

**复现**：电商工作区填写“店铺希望的感觉”（例如“克制、清爽、年轻”）并
提交。

**实际表现 / 证据**：UI 将该输入标为 `v3BrandToneInput`（
`app.js:2024-2026`），但 `v3EcommerceProfilePatch` 在
`app.js:5103-5115` 把它命名为 `sellingPoint` 并写入
`core_selling_points`。

**用户风险**：审美偏好会被 Brain 当作产品卖点，污染商品事实和后续方向；
这与“只传产品事实、用户明确约束、平台证据”的边界不符。

**建议修复**：前端把它作为明确的用户视觉偏好传递（现有通用
`metadata.visual_tone` 已在 `buildV3JobPayload` 存在），而不是
`core_selling_points`。真正的“商品卖点”需用独立、明确命名的可选字段，
或让用户自然语言写在需求中。

**涉及**：电商前端 payload 修复；先核对现有 Brain 合同是否已消费
`visual_tone`。若没有，需要主线确认一个场景中立的“用户明确视觉偏好”
字段，不得把它塞回卖点 schema。

### P0-4 “下一步”恢复区被代码永久隐藏

**复现**：创建、blocked、held 或完成任一电商项目后，打开项目续作子页，
寻找可理解的下一步（补图、修改事实、调整数量、刷新项目）。

**实际表现 / 证据**：`renderV3ProjectNextActions` 在
`app.js:4160-4164` 清空内容、设置 `hidden=true` 并立即 `return`；后方
完整的“继续 / 加入参考图 / 选择结果”渲染代码不可达。

**用户风险**：项目虽被保留，用户却没有恢复路径。blocked 状态会变成“任务
停了”，而非“现在可做什么”；这直接违背新手路径的失败恢复要求。

**建议修复**：按当前状态只展示一个主操作和一个辅助操作：

- 缺商品事实/参考：补充商品图或关键事实；
- 数量能力不支持：改为可用数量；
- blocked/上游不可用：保留项目，显示“稍后刷新项目”，不自动重提；
- held/manual confirmation：查看图片并确认是否可用；
- 已交付：选择一张作为后续参考或创建新的明确任务。

不显示 slot redo，也不要调用内部 retry/select/delete 伪装成“重做”。

**涉及**：前端状态映射；需要复用现有 job/project 状态，无新增共享接口。

### P0-5 历史 recipe/slot 仍可能进入普通结果面

**复现**：打开一个只有历史 `job.ecommerce.image_recipes` 或旧
`ecommerce_slot` 字段、没有当前普通候选的项目。

**实际表现 / 证据**：`renderV3ResultBoard`（`app.js:6254-6260`）把
`image_recipes` 作为 candidates 为空时的结果来源；`app.js:6316-6320`
继续读取 `item.slot` / `metadata.ecommerce_slot`。虽然后续会把
`ecommerce_output_N` 映射为“中央大脑输出 N”，这仍让历史执行字段影响
普通结果面。

**用户风险**：历史兼容字段会被误解为可操作、固定的图片槽位；用户会期待
“重做这张角色图”。

**建议修复**：历史 recipe/slot 只在折叠的兼容记录中可读，绝不作为当前结果
卡来源或标题依据。普通结果板仅消费 shared final delivery / current project
outputs；无当前交付时显示“此历史任务没有可交付图片”。

**涉及**：前端投射和回归测试；无需改变 continuation 运行时。

## P1：应在本轮 UX 收口完成

### P1-1 必填、强烈建议和可选信息没有层级

**证据**：电商工作区写“可上传商品图，不传也能生成”（`app.js:2020-2022`），
却没有明确“本次需求”为必填、“商品图会显著提高实物一致性”、“平台会影响
约束”；类目、市场、文案语言、批准原文和产品事实又分散在折叠的“更多设置”
中（`index.html:1604-1710`）。

**建议**：将首屏按三层组织：

1. 必填：本次想做什么；
2. 强烈建议：商品参考图与不能改错的产品事实；
3. 可选：平台、市场、语言、批准原文、受众、禁用宣称。

每一项用一句结果影响说明，不要求用户理解模型、提示词或参考通道。

### P1-2 阻断状态语言与可恢复路径不一致

**证据**：`v3StatusLabel` 将 `blocked` 显示为“未开放”（`app.js:6512-6526`），
但真实含义可能是数量合同、参考准入、Brain fail-closed 或上游暂不可用；
`v3EcommerceFailureMessage` 已能给更具体的人话原因（`5588-5616`），却没有
配套下一步按钮（P0-4）。

**建议**：统一显示“本次未提交生成 / 本次已安全停止”，再用一句原因与一项
下一步替代“未开放”。不要向普通用户显示 Brain、Provider、retry、hash、
plan 或内部错误代码。

### P1-3 平台、语言和批准原文的影响说明不足

**证据**：平台选择有默认“通用电商”，但没有“会影响什么”的说明；语言和
“仅填写需要原样出现在最终图片中的文字”藏在高级区。当前 summary 还会对
普通用户展示“真实 Provider Gate C/D”“最终像素层审查”（`6030-6038`），
这些是内部发布门禁，不是完成电商任务所需概念。

**建议**：平台旁说明“用于让本次图片更贴近上架约束”；语言仅在确实需要图内
文字时显示；批准原文提示“请只填已经确认可使用的文字，模型会把它作为画面
内容尝试呈现”。将生产门禁改为产品状态文案或仅在内部验收面显示。

### P1-4 结果标签和高级提示词面向内部，而非普通用户

**证据**：`v3EcommerceOutputLabel` 显示“中央大脑输出 N”（`6489-6492`）；
结果 lightbox 与项目记录可展示并复制“完整提示词”（`index.html:1955-1964`、
`app.js:6459-6467`、`13313-13319`）。

**建议**：普通卡显示“第 N 张 · 本次方向”与一句用户可读方向摘要；内部
prompt/provenance 只在受控开发诊断面可见，普通项目不展示或复制。

### P1-5 现有静态回归覆盖遗漏了同义变体

**证据**：`test_doc132_browser_surface_has_no_executable_ecommerce_preset_or_fixed_suite_promise`
在本基线通过，但它只拦截两个精确句子；当前 `app.js:3650-3656` 的“主图、
卖点图、场景图、细节图和信任图”变体未被捕获。

**建议**：用语义更明确的 UI contract 测试：E-Commerce 普通页面不得同时
出现固定角色列表、preset/recipe/slot 入口，且不得把 `image_recipes` 作为当前
交付来源；不需要也不应测试固定图片角色本身。

## P2：移动与视觉收口验证

源码显示 `styles.css:9575-9605` 已把模板卡、输入、生成设置、输出板等在
`760px` 以下收为单列，因此没有静态证据证明会横向溢出。但本机无受控 V3
服务：`127.0.0.1:8017/creative-agent-v3` 返回连接拒绝；浏览器截图验收未能
进行。

主线应在当前受控实例补做 360px、390px、768px 的以下清单：模板选择、上传
失败、折叠高级项、1/2/4/7 数量、长 blocked 文案、held 卡、项目刷新后结果、
重复点生成按钮。验收只记录 UI 行为，不触发真实 Provider/Gate C/D。

## 推荐修复顺序与所有权

1. **E-Commerce 前端**：P0-1、P0-3、P0-4、P0-5、P1 文案与结果投射。
2. **主线共享能力投射（如缺失）**：仅提供当前平台/通道可接受的数量集合，
   供 P0-2 的 UI 禁用项使用；不把平台规则复制到浏览器。
3. **主线前端收口**：P2 受控窄屏验收，以及受限内部诊断与普通用户面的
   prompt/provenance 隔离。

任何修复都必须保持：显式 `ecommerce_template`、远端 Brain 必需、exact-N、
整图方向、共享 generation/review/retry/final delivery；不得恢复静态套图、
slot/recipe、默认卖点、CopyRenderPlan、本地字体/OCR/叠字或私有重试。
